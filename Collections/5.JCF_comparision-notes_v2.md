## 1. Collection vs Collections

| | `Collection` | `Collections` |
|---|---|---|
| Type | Interface | Utility class |
| Purpose | Root of the collection hierarchy | Static helper methods for collections |
| Instantiable | No — implemented by classes | No — all methods are static |

```java
// Collection — interface, implemented by ArrayList, HashSet, etc.
Collection<String> names = new ArrayList<>();
names.add("Alice");
names.contains("Alice"); // true

// Collections — utility class, static methods only
List<String> list = new ArrayList<>(Arrays.asList("Charlie", "Alice", "Bob"));
Collections.sort(list);                          // sorts in place
Collections.binarySearch(list, "Alice");         // search in sorted list
Collections.shuffle(list);                       // randomize
Collections.synchronizedList(list);              // thread-safe wrapper
```

## 2. Iterator vs ListIterator

| Feature | `Iterator` | `ListIterator` |
|---|---|---|
| Works with | Any Collection | List only |
| Direction | Forward only | Forward and backward |
| `add()` | No | Yes |
| `set()` | No | Yes |
| Index access | No | Yes (`nextIndex()`, `previousIndex()`) |

```java
List<String> list = new ArrayList<>(Arrays.asList("Apple", "Banana", "Cherry"));

// Iterator — forward, remove only
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("Banana")) it.remove(); // safe removal
}

// ListIterator — bidirectional, add/set supported
ListIterator<String> lit = list.listIterator();
while (lit.hasNext()) {
    String s = lit.next();
    if (s.equals("Cherry")) lit.set("Cranberry"); // replace in place
}
while (lit.hasPrevious()) {
    System.out.print(lit.previous() + " "); // backward traversal
}
```

## 3. Enumeration vs Iterator

| Feature | `Enumeration` | `Iterator` |
|---|---|---|
| Introduced | JDK 1.0 (legacy) | JDK 1.2 (modern) |
| Methods | `hasMoreElements()`, `nextElement()` | `hasNext()`, `next()`, `remove()` |
| Remove during traversal | No | Yes |
| Fail-fast | No | Yes |
| Used with | `Vector`, `Hashtable` | All modern collections |

```java
// Enumeration — legacy, read-only
Vector<String> v = new Vector<>(Arrays.asList("One", "Two", "Three"));
Enumeration<String> e = v.elements();
while (e.hasMoreElements()) { System.out.println(e.nextElement()); }

// Iterator — modern, supports removal
List<String> list = new ArrayList<>(Arrays.asList("One", "Two", "Three"));
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("Two")) it.remove(); // not possible with Enumeration
}
```

**Gotcha**: Prefer `Iterator` in all new code. `Enumeration` exists only for legacy class compatibility.

## 4. Comparable vs Comparator

| Feature | `Comparable` | `Comparator` |
|---|---|---|
| Package | `java.lang` | `java.util` |
| Method | `compareTo(T o)` | `compare(T o1, T o2)` |
| Defined in | The class itself | Separate class or lambda |
| Sort sequences | One (natural order) | Many (multiple comparators) |
| Modify class needed | Yes | No |

```java
// Comparable — natural order defined inside the class
public class Student implements Comparable<Student> {
    String name; int age;
    public int compareTo(Student o) { return Integer.compare(this.age, o.age); }
}
Collections.sort(students); // uses compareTo

// Comparator — external, multiple orderings possible
students.sort(Comparator.comparing(s -> s.name));                        // by name
students.sort(Comparator.comparingInt((Student s) -> s.age).reversed()); // by age desc
students.sort(Comparator.comparing((Student s) -> s.name)
              .thenComparingInt(s -> s.age));                             // multi-field
```

**Rule of thumb**: Use `Comparable` for the natural/default order. Use `Comparator` when you need multiple sort strategies or can't modify the class.

## 5. Array vs ArrayList vs LinkedList

| Feature | `Array` | `ArrayList` | `LinkedList` |
|---|---|---|---|
| Size | Fixed | Dynamic | Dynamic |
| Random access | O(1) | O(1) | O(n) |
| Insert at start | O(n) | O(n) | O(1) |
| Insert at end | O(1) | Amortized O(1) | O(1) |
| Insert in middle | O(n) | O(n) | O(n)* |
| Memory | Minimal | Moderate | High (node pointers) |
| Primitives | Yes | No (autoboxing) | No |
| Implements | — | `List` | `List`, `Deque` |

\* O(1) after finding position, but finding takes O(n)

```java
// Array — fixed size, primitives allowed
int[] arr = new int[5];
arr[0] = 10; // O(1)

// ArrayList — fast random access
List<Integer> al = new ArrayList<>();
al.add(10); al.get(0); // O(1)
al.add(0, 5);          // O(n) — shifts elements

// LinkedList — fast insert/delete at ends
LinkedList<Integer> ll = new LinkedList<>();
ll.addFirst(1);  // O(1)
ll.addLast(2);   // O(1)
ll.get(1);       // O(n) — must traverse
```

## 6. ArrayList vs Vector

| Feature | `ArrayList` | `Vector` |
|---|---|---|
| Thread-safe | No | Yes (all methods synchronized) |
| Performance | Faster | Slower |
| Growth | 50% of current size | 100% of current size (doubles) |
| Introduced | JDK 1.2 | JDK 1.0 (legacy) |
| Iterator | Fail-fast | Fail-fast + Enumeration |

```java
// ArrayList — not thread-safe, use in single-threaded or with external sync
List<String> list = new ArrayList<>();

// Vector — thread-safe but slow, avoid in new code
Vector<String> v = new Vector<>();

// Preferred thread-safe alternative
List<String> safe = Collections.synchronizedList(new ArrayList<>());
// or
List<String> concurrent = new CopyOnWriteArrayList<>();
```

**Gotcha**: `Vector` doubles its size on resize — wastes more memory than `ArrayList`.

## 7. ArrayList vs CopyOnWriteArrayList

| Feature | `ArrayList` | `CopyOnWriteArrayList` |
|---|---|---|
| Thread-safe | No | Yes |
| Write operation | Modifies in place | Creates a new array copy |
| Iterator | Fail-fast | Fail-safe (snapshot) |
| Read performance | Fast | Fast |
| Write performance | Fast | Slow (O(n) per write) |
| Use case | Single-threaded | Read-heavy concurrent |

```java
// ArrayList — ConcurrentModificationException on modify during iteration
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c"));
for (String s : list) {
    list.add("d"); // throws ConcurrentModificationException
}

// CopyOnWriteArrayList — safe, iterator works on snapshot
List<String> safe = new CopyOnWriteArrayList<>(Arrays.asList("a", "b", "c"));
for (String s : safe) {
    safe.add("d"); // no exception — iterator sees original snapshot only
}
System.out.println(safe); // [a, b, c, d, d, d]
```

## 8. Fail-Fast vs Fail-Safe Iterator

| | Fail-Fast | Fail-Safe |
|---|---|---|
| On modification during iteration | Throws `ConcurrentModificationException` | No exception |
| Works on | Original collection | Snapshot / copy |
| Memory | Efficient | Higher (copy needed) |
| Sees latest changes | Yes | No (snapshot) |
| Examples | `ArrayList`, `HashMap`, `HashSet` | `CopyOnWriteArrayList`, `ConcurrentHashMap` |

```java
// Fail-fast — throws exception
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c"));
Iterator<String> it = list.iterator();
list.add("d");         // structural modification
it.next();             // ConcurrentModificationException

// Fail-safe — no exception, iterator sees snapshot
List<String> safe = new CopyOnWriteArrayList<>(Arrays.asList("a", "b", "c"));
Iterator<String> sit = safe.iterator();
safe.add("d");         // creates new internal array
sit.next();            // still works — sees "a", "b", "c" only
```

## 9. containsKey() vs keySet() vs values() in HashMap

| Method | Returns | Purpose | Complexity |
|---|---|---|---|
| `containsKey(key)` | `boolean` | Check if key exists | O(1) |
| `keySet()` | `Set<K>` | All keys as a Set | O(1) |
| `values()` | `Collection<V>` | All values | O(1) |

```java
Map<String, Integer> map = new HashMap<>();
map.put("alice", 95); map.put("bob", 80); map.put("charlie", 90);

map.containsKey("bob");    // true — O(1) check
map.containsKey("dave");   // false

map.keySet();              // [alice, bob, charlie] — backed by map
map.values();              // [95, 80, 90] — backed by map, can have duplicates

// All three views are backed by the map — changes reflect both ways
map.keySet().remove("bob");        // also removes from map
map.values().removeIf(v -> v < 85); // removes alice entry
```

## 10. peek() vs poll() vs remove() in Queue

| Method | Removes element | Empty queue behavior |
|---|---|---|
| `peek()` | No | Returns `null` |
| `poll()` | Yes | Returns `null` |
| `remove()` | Yes | Throws `NoSuchElementException` |

```java
Queue<String> queue = new LinkedList<>(Arrays.asList("First", "Second", "Third"));

queue.peek();    // "First" — queue still has 3 elements
queue.poll();    // "First" — queue now has 2 elements
queue.remove();  // "Second" — queue now has 1 element

Queue<String> empty = new LinkedList<>();
empty.peek();    // null — no exception
empty.poll();    // null — no exception
empty.remove();  // NoSuchElementException
```

**Rule**: Use `poll()` and `peek()` in production — they handle empty queues gracefully.

## 11. ArrayBlockingQueue vs LinkedBlockingQueue

| Feature | `ArrayBlockingQueue` | `LinkedBlockingQueue` |
|---|---|---|
| Backed by | Array | Linked nodes |
| Capacity | Always bounded (must specify) | Optional (unbounded by default) |
| Locking | Single lock for put + take | Separate locks for put and take |
| Throughput | Lower under high concurrency | Higher under high concurrency |
| Memory | Pre-allocated, efficient | Grows dynamically, more overhead |
| Use when | Fixed capacity known upfront | Dynamic sizing or high throughput |

```java
// ArrayBlockingQueue — fixed capacity, single lock
BlockingQueue<String> abq = new ArrayBlockingQueue<>(10);
abq.put("task");              // blocks if full
abq.offer("task", 500, TimeUnit.MILLISECONDS); // waits up to 500ms
String t = abq.take();        // blocks if empty

// LinkedBlockingQueue — optional capacity, separate put/take locks
BlockingQueue<String> lbq = new LinkedBlockingQueue<>();       // unbounded
BlockingQueue<String> lbqBounded = new LinkedBlockingQueue<>(10); // bounded
lbq.put("task");
String t2 = lbq.take();
```

## 12. Set vs Map

| Feature | `Set` | `Map` |
|---|---|---|
| Stores | Unique elements | Key-value pairs |
| Duplicate keys | Not allowed | Not allowed |
| Duplicate values | N/A | Allowed |
| Extends | `Collection` | Does not extend `Collection` |
| Access | Direct element lookup | Access value via key |

```java
Set<String> set = new HashSet<>();
set.add("Java"); set.add("Python"); set.add("Java"); // duplicate ignored
System.out.println(set.size()); // 2

Map<String, Integer> map = new HashMap<>();
map.put("Java", 1995); map.put("Python", 1991);
map.put("Java", 2000); // overwrites — no duplicate keys
System.out.println(map.size()); // 2
System.out.println(map.get("Java")); // 2000
```

## 13. HashSet vs LinkedHashSet

| Feature | `HashSet` | `LinkedHashSet` |
|---|---|---|
| Order | No guaranteed order | Insertion order maintained |
| Backed by | `HashMap` | `LinkedHashMap` |
| Performance | Slightly faster | Slightly slower (linked list overhead) |
| Memory | Less | More |
| Use when | Order doesn't matter | Need insertion order |

```java
Set<String> hs = new HashSet<>(Arrays.asList("Banana", "Apple", "Orange"));
System.out.println(hs); // [Apple, Orange, Banana] — unpredictable order

Set<String> lhs = new LinkedHashSet<>(Arrays.asList("Banana", "Apple", "Orange"));
System.out.println(lhs); // [Banana, Apple, Orange] — insertion order
```

## 14. HashSet vs TreeSet

| Feature | `HashSet` | `TreeSet` |
|---|---|---|
| Order | No order | Sorted (natural or Comparator) |
| Backed by | `HashMap` | `TreeMap` (Red-Black Tree) |
| `add/remove/contains` | O(1) | O(log n) |
| Null elements | One allowed | Not allowed |
| Extra methods | None | `first()`, `last()`, `subSet()`, `headSet()`, `tailSet()` |

```java
Set<Integer> hs = new HashSet<>(Arrays.asList(5, 3, 9, 1));
System.out.println(hs); // [1, 3, 5, 9] or any order

TreeSet<Integer> ts = new TreeSet<>(Arrays.asList(5, 3, 9, 1));
System.out.println(ts);           // [1, 3, 5, 9] — always sorted
System.out.println(ts.first());   // 1
System.out.println(ts.last());    // 9
System.out.println(ts.subSet(3, true, 7, true)); // [3, 5]
System.out.println(ts.higher(5)); // 9
```

## 15. HashSet vs HashMap

| Feature | `HashSet` | `HashMap` |
|---|---|---|
| Interface | `Set` | `Map` |
| Stores | Single elements | Key-value pairs |
| Internally | Uses `HashMap` with dummy values | Direct hash table |
| Key methods | `add()`, `contains()`, `remove()` | `put()`, `get()`, `containsKey()` |
| Iteration | Over elements | Over keys, values, or entries |

```java
Set<String> set = new HashSet<>();
set.add("Java"); set.contains("Java"); // true

Map<String, Integer> map = new HashMap<>();
map.put("Java", 1995); map.get("Java"); // 1995

// HashSet internally: map.put("Java", PRESENT) where PRESENT is a dummy Object
```

## 16. HashMap vs IdentityHashMap

| Feature | `HashMap` | `IdentityHashMap` |
|---|---|---|
| Key equality | `.equals()` | `==` (reference) |
| Hashing | `.hashCode()` | `System.identityHashCode()` |
| Same content, different objects | Same key | Different keys |
| Use case | General purpose | Object graph traversal, deep clone tracking |

```java
String a = new String("key");
String b = new String("key");

Map<String, Integer> hm = new HashMap<>();
hm.put(a, 1); hm.put(b, 2);
System.out.println(hm.size()); // 1 — a.equals(b), b overwrites a

Map<String, Integer> im = new IdentityHashMap<>();
im.put(a, 1); im.put(b, 2);
System.out.println(im.size()); // 2 — a != b (different references)
```

## 17. HashMap vs Hashtable

| Feature | `HashMap` | `Hashtable` |
|---|---|---|
| Thread-safe | No | Yes (all methods synchronized) |
| Performance | Faster | Slower |
| Null key | One allowed | Not allowed |
| Null values | Allowed | Not allowed |
| Introduced | JDK 1.2 | JDK 1.0 (legacy) |
| Extends | `AbstractMap` | `Dictionary` |
| Modern alternative | `HashMap` + `ConcurrentHashMap` | `ConcurrentHashMap` |

```java
HashMap<String, Integer> hm = new HashMap<>();
hm.put(null, 0);    // allowed
hm.put("a", null);  // allowed

Hashtable<String, Integer> ht = new Hashtable<>();
// ht.put(null, 0);   // NullPointerException
// ht.put("a", null); // NullPointerException

// Preferred thread-safe alternative
Map<String, Integer> safe = new ConcurrentHashMap<>();
```

**Gotcha**: Never use `Hashtable` in new code — use `ConcurrentHashMap` for thread safety.

## 18. HashMap vs TreeMap

| Feature | `HashMap` | `TreeMap` |
|---|---|---|
| Order | No order | Sorted by key |
| Backed by | Hash table | Red-Black Tree |
| `get/put/remove` | O(1) average | O(log n) |
| Null keys | One allowed | Not allowed (natural ordering) |
| Extra methods | None | `firstKey()`, `lastKey()`, `subMap()`, `headMap()`, `tailMap()` |

```java
Map<String, Integer> hm = new HashMap<>();
hm.put("Banana", 2); hm.put("Apple", 5); hm.put("Orange", 3);
System.out.println(hm.keySet()); // unpredictable order

TreeMap<String, Integer> tm = new TreeMap<>();
tm.put("Banana", 2); tm.put("Apple", 5); tm.put("Orange", 3);
System.out.println(tm.keySet());       // [Apple, Banana, Orange] — sorted
System.out.println(tm.firstKey());     // Apple
System.out.println(tm.lowerKey("Banana")); // Apple
System.out.println(tm.subMap("Apple", "Orange")); // {Apple=5, Banana=2}
```

## 19. HashMap vs ConcurrentHashMap

| Feature | `HashMap` | `ConcurrentHashMap` |
|---|---|---|
| Thread-safe | No | Yes |
| Null key | One allowed | Not allowed |
| Null values | Allowed | Not allowed |
| Lock granularity | None | Node-level (Java 8+) |
| Iterator | Fail-fast | Weakly consistent (no exception) |
| Atomic operations | None | `putIfAbsent()`, `replace()`, `remove(k,v)` |
| Performance (single thread) | Faster | Slightly slower |
| Performance (multi-thread) | Poor | Excellent |

```java
// HashMap — not safe for concurrent use
Map<String, Integer> hm = new HashMap<>();
hm.put(null, 0);    // allowed

// ConcurrentHashMap — thread-safe, no nulls
Map<String, Integer> chm = new ConcurrentHashMap<>();
chm.putIfAbsent("key", 1);    // atomic — only puts if absent
chm.replace("key", 1, 100);   // atomic — only replaces if current value matches
chm.remove("key", 100);       // atomic — only removes if value matches

// No need to synchronize iteration
for (Map.Entry<String, Integer> e : chm.entrySet()) {
    System.out.println(e.getKey() + "=" + e.getValue()); // safe
}
```

**Gotcha**: `ConcurrentHashMap` does not allow null keys or values — `NullPointerException` at runtime.

## 20. Queue vs Stack

| Feature | `Queue` | `Stack` |
|---|---|---|
| Access pattern | FIFO (First In, First Out) | LIFO (Last In, First Out) |
| Type | Interface | Legacy class (extends `Vector`) |
| Add | `offer(e)` | `push(e)` |
| Remove | `poll()` | `pop()` |
| Peek | `peek()` | `peek()` |
| Modern implementation | `ArrayDeque`, `LinkedList` | `ArrayDeque` (preferred over `Stack`) |

```java
// Queue — FIFO
Queue<String> queue = new ArrayDeque<>();
queue.offer("First"); queue.offer("Second"); queue.offer("Third");
System.out.println(queue.poll()); // First — FIFO

// Stack (legacy) — LIFO
Stack<String> stack = new Stack<>();
stack.push("First"); stack.push("Second"); stack.push("Third");
System.out.println(stack.pop()); // Third — LIFO

// Modern Stack — use ArrayDeque
Deque<String> modernStack = new ArrayDeque<>();
modernStack.push("First"); modernStack.push("Second"); modernStack.push("Third");
System.out.println(modernStack.pop()); // Third — LIFO
```

**Gotcha**: `Stack` extends `Vector` — it is synchronized and slow. Always use `ArrayDeque` as a stack in new code.
