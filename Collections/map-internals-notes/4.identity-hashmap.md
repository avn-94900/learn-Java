## Internal Structure and Working

`IdentityHashMap` stores key-value pairs using **reference equality (`==`)** for key comparison instead of `equals()`. It also uses `System.identityHashCode()` instead of the key's `hashCode()` method.

### How it differs from HashMap internally

- Does **not** use a linked list or tree per bucket.
- Uses a single **flat array** where keys and values are stored in alternating positions: `[key0, value0, key1, value1, ...]`.
- Collision resolution is done via **linear probing** (open addressing), not chaining.
- Capacity is always a **power of 2**; default is 32 slots (16 key-value pairs).

### Internal array layout

```
index:  0       1       2       3       4       5  ...
array: [key0] [val0] [key1] [val1] [key2] [val2] ...
```

### How a put() works

1. Compute `System.identityHashCode(key)` — this is the key's memory-based hash.
2. Find the slot index: `index = (hash & (table.length - 1)) & ~1` (even index for keys).
3. Probe linearly until an empty slot or the **exact same object** (`==`) is found.
4. Store the key and value at the found slot.

### How a get() works

1. Compute `System.identityHashCode(key)`.
2. Probe the array linearly from that index.
3. Return the value only if `table[i] == key` (reference equality, not `equals()`).

```java
IdentityHashMap<String, Integer> map = new IdentityHashMap<>();

String a = new String("hello");
String b = new String("hello");

map.put(a, 1);
map.put(b, 2);

System.out.println(map.size()); // 2 — a and b are different objects
System.out.println(map.get(a)); // 1
System.out.println(map.get(b)); // 2
```

## Reference Equality vs Object Equality

This is the defining characteristic of `IdentityHashMap`. Two keys are considered equal only if they are the **exact same object in memory**.

| Comparison | HashMap | IdentityHashMap |
|-----------|---------|-----------------|
| Key equality | `key1.equals(key2)` | `key1 == key2` |
| Hash source | `key.hashCode()` | `System.identityHashCode(key)` |

### Side-by-side example

```java
String x = new String("test");
String y = new String("test");

// HashMap — treats x and y as the same key
HashMap<String, Integer> hashMap = new HashMap<>();
hashMap.put(x, 1);
hashMap.put(y, 2);
System.out.println(hashMap.size()); // 1 — y overwrote x

// IdentityHashMap — treats x and y as different keys
IdentityHashMap<String, Integer> identityMap = new IdentityHashMap<>();
identityMap.put(x, 1);
identityMap.put(y, 2);
System.out.println(identityMap.size()); // 2 — x and y are separate entries
```

### System.identityHashCode()

`System.identityHashCode(obj)` returns the hash code that `Object.hashCode()` would return — based on the object's memory address — even if `hashCode()` has been overridden.

```java
String a = new String("hello");
String b = new String("hello");

System.out.println(a.hashCode());                    // same for both (31-based algorithm)
System.out.println(System.identityHashCode(a));      // different
System.out.println(System.identityHashCode(b));      // different
```

## Null Support

- Allows **one null key** and **multiple null values**.
- Null is treated as a valid object reference and compared with `==`.

```java
IdentityHashMap<String, Integer> map = new IdentityHashMap<>();
map.put(null, 99);
map.put("key", null);

System.out.println(map.get(null));  // 99
System.out.println(map.get("key")); // null
```

## IdentityHashMap vs HashMap

| Feature | HashMap | IdentityHashMap |
|---------|---------|-----------------|
| Key equality | `equals()` | `==` (reference) |
| Hash source | `hashCode()` | `System.identityHashCode()` |
| Internal structure | Bucket array + linked list / tree | Flat array with linear probing |
| Null keys | 1 allowed | 1 allowed |
| Ordering | None | None |
| Thread-safe | No | No |
| Respects `equals()` override | Yes | No |
| Use case | General key-value lookup | Object graph traversal, serialization |

## When to Use IdentityHashMap

Use `IdentityHashMap` when the **identity** of an object matters — not its logical equality — and you need to track or annotate specific object instances.

### Good use cases

- **Serialization / deep copy** — track which objects have already been visited to handle cycles.
- **Object graph traversal** — mark visited nodes without modifying them.
- **Proxy and instrumentation frameworks** — map original objects to their proxies by reference.
- **Interning / canonicalization** — distinguish between two logically equal but physically distinct objects.

```java
// Deep copy with cycle detection
IdentityHashMap<Object, Object> visited = new IdentityHashMap<>();

public Object deepCopy(Object original) {
    if (visited.containsKey(original)) {
        return visited.get(original); // return already-copied instance
    }
    Object copy = createShallowCopy(original);
    visited.put(original, copy);     // register by reference
    // recursively copy fields...
    return copy;
}
```

### Bad use cases

- **String or value-based keys** — two `String` objects with the same content are treated as different keys, which is almost always wrong.
- **General-purpose maps** — the reference-equality semantics are surprising and error-prone for everyday use.

## Pitfalls and Gotchas

### Pitfall 1: String literals may behave unexpectedly

String literals are interned, so two literals with the same content are the **same object**. But `new String(...)` objects are always distinct.

```java
IdentityHashMap<String, Integer> map = new IdentityHashMap<>();

map.put("hello", 1); // interned literal
map.put("hello", 2); // same interned object → overwrites

System.out.println(map.size()); // 1 — surprising if you expected 2

map.put(new String("hello"), 3); // new object
System.out.println(map.size()); // 2 — now distinct
```

### Pitfall 2: autoboxed primitives may be cached

The JVM caches `Integer` values in the range -128 to 127. Autoboxed values in this range return the **same object**, which can cause unexpected key collisions.

```java
IdentityHashMap<Integer, String> map = new IdentityHashMap<>();
map.put(42, "A");
map.put(42, "B"); // same cached Integer object

System.out.println(map.size()); // 1 — 42 is cached, same reference

map.put(200, "C");
map.put(200, "D"); // different Integer objects — outside cache range
System.out.println(map.size()); // 3
```

### Pitfall 3: equals() and hashCode() are completely ignored

Even if a key class overrides `equals()` and `hashCode()`, `IdentityHashMap` ignores them entirely.

```java
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override public boolean equals(Object o) { ... }
    @Override public int hashCode() { return Objects.hash(x, y); }
}

IdentityHashMap<Point, String> map = new IdentityHashMap<>();
Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);

map.put(p1, "first");
map.put(p2, "second");

System.out.println(map.size()); // 2 — equals() was never consulted
```

### Pitfall 4: Not thread-safe

`IdentityHashMap` is not thread-safe. Wrap with `Collections.synchronizedMap` when sharing across threads.

```java
Map<Object, String> syncMap =
        Collections.synchronizedMap(new IdentityHashMap<>());
```

## Time Complexity

| Operation | Average | Worst Case |
|-----------|---------|------------|
| `get(key)` | O(1) | O(n) |
| `put(key, value)` | O(1) | O(n) |
| `remove(key)` | O(1) | O(n) |
| `containsKey(key)` | O(1) | O(n) |
| `containsValue(value)` | O(n) | O(n) |
| Iteration | O(n) | O(n) |

> Worst case occurs when many keys hash to the same probe sequence. `IdentityHashMap` does not use trees, so worst case is O(n) unlike `HashMap`'s O(log n) in Java 8+.

## Common Coding Problems

### 1. Cycle detection during deep copy

```java
public Object deepCopy(Object original,
                        IdentityHashMap<Object, Object> visited) {
    if (original == null) return null;
    if (visited.containsKey(original)) return visited.get(original);

    Object copy = createShallowCopy(original);
    visited.put(original, copy);

    for (Field field : original.getClass().getDeclaredFields()) {
        field.setAccessible(true);
        Object fieldValue = field.get(original);
        field.set(copy, deepCopy(fieldValue, visited));
    }
    return copy;
}
```

### 2. Object graph serialization (visited tracking)

```java
public void serialize(Object obj,
                       IdentityHashMap<Object, Integer> visited,
                       DataOutput out) throws IOException {

    if (visited.containsKey(obj)) {
        out.writeInt(visited.get(obj)); // write back-reference ID
        return;
    }

    int id = visited.size();
    visited.put(obj, id);
    out.writeInt(id);
    // serialize fields...
}
```

### 3. Proxy registry

Map original objects to their proxies by reference.

```java
IdentityHashMap<Object, Object> proxyRegistry = new IdentityHashMap<>();

public <T> T getOrCreateProxy(T original) {
    return (T) proxyRegistry.computeIfAbsent(
            original,
            obj -> Proxy.newProxyInstance(
                    obj.getClass().getClassLoader(),
                    obj.getClass().getInterfaces(),
                    new LoggingHandler(obj)
            )
    );
}
```

### 4. Detect shared references in an object graph

Check whether two variables point to the exact same object instance.

```java
public Set<Object> findSharedObjects(List<Object> graph) {
    IdentityHashMap<Object, Integer> seen = new IdentityHashMap<>();
    Set<Object> shared = Collections.newSetFromMap(new IdentityHashMap<>());

    for (Object obj : graph) {
        seen.merge(obj, 1, Integer::sum);
        if (seen.get(obj) > 1) shared.add(obj);
    }
    return shared;
}
```
