## Java Atomic Concepts – Final Clean Notes



## 1. Atomic Operation (Basic Idea)

* An operation that **completes in one step**
* Cannot be seen in a half-done state by other threads

### Key points

* Either fully happens or does not happen
* No partial update is visible

### Example

* `count++` ❌ not atomic (read → add → write)
* `AtomicInteger.incrementAndGet()` ✅ atomic

---

## 2. Problem in Multi-threading

* Shared variable: `count = 10`
* Two threads do `count++`
* Both read 10 → both write 11 ❌ (lost update)

---

## 3. CAS (Compare-And-Swap)

### Core idea

> Update value only if it was not changed after reading.

---

### Steps

For `value = 10`:

1. Read current value → 10
2. Compare → is it still 10?
3. Swap:

   * yes → update to new value ✅
   * no → fail ❌ and retry

---

### Version / Timestamp concept (optional enhancement)

```
value = 10
version = 1
```

CAS checks both value + version.

If updated:

```
value = 11
version = 2
```

→ CAS fails if mismatch → retry

---

### Key idea

* No locking
* Instead of blocking → **retry**

---

### One-line definition

> CAS updates value only if expected value matches current memory; otherwise it retries.

---

## 4. AtomicInteger Internal Working

### What it uses

* CAS mechanism
* Unsafe class (low-level memory access)
* Retry loop

---

### How `incrementAndGet()` works

### Step 1: Read value

```
current = 10
```

### Step 2: Compute new value

```
next = 11
```

### Step 3: CAS attempt

```
if value == 10 → update to 11
else → fail
```

---

### Step 4: Retry loop

If another thread changes value:

* Thread A reads 10
* Thread B updates to 11
* Thread A CAS fails
* Thread A retries again

---

### Simplified internal code

```java
while (true) {
    int current = value;
    int next = current + 1;

    if (CAS(value, current, next)) {
        return next;
    }
}
```

---

### What is Unsafe?

* Low-level Java class
* Direct memory access
* Used to perform CAS at CPU level

---

### Why retry loop exists

* No locks
* No thread blocking
* Keeps trying until success

---

### Key interview points

* AtomicInteger = CAS + Unsafe + retry loop
* It is lock-free
* Uses CPU-level atomic instructions
* Better than synchronized in many cases

---

### Final one-line summary

> AtomicInteger uses CAS with a retry loop via Unsafe to update values atomically without locking.
