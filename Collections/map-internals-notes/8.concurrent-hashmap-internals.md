# ConcurrentHashMap — Internals, Concurrency Mechanics & Interview Guide

> Companion to `6.concurrent-hashmap.md`. Focuses on internal machinery, low-level locking, Java 7 vs 8 differences, and interview readiness.

---

## 1. Core Terminology Quick Notes


**CAS (Compare-And-Swap)**
- Definition: A CPU-level atomic instruction. It says: "set this memory location to new value ONLY IF it currently holds the expected value."
- Why Used: Avoids acquiring a lock for simple updates. Much faster than `synchronized` under low contention.
- When Used: Inserting into an empty bucket, updating the size counter, initializing the table.
- Simple Example:
```java
// Conceptual model — actual JDK uses Unsafe/VarHandle internally
AtomicReference<Node> bucket = new AtomicReference<>(null);
Node newNode = new Node("key", "value");

// succeeds only if bucket is still null — no lock needed
boolean inserted = bucket.compareAndSet(null, newNode);
```

---

**Atomic Operation**
- Definition: An operation that completes fully or not at all — no thread can observe it in a half-done state.
- Why Used: Prevents race conditions without using locks.
- When Used: `putIfAbsent`, `compute`, `merge`, CAS-based bucket insert.
- Simple Example:
```java
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // read + increment + write as one atomic step
```

---

**Synchronized Locking**
- Definition: Java's built-in monitor lock. Only one thread can hold it at a time.
- Why Used: Needed when CAS alone is not enough — e.g., traversing or modifying a non-empty bucket chain.
- When Used: Inserting into a non-empty bucket, treeification, bucket transfer during resize.
- Simple Example:
```java
synchronized (bucketHeadNode) {
    // only one thread modifies this bucket at a time
    // other buckets are completely unaffected
}
```

---

**Segment-Level Locking (Java 7 only)**
- Definition: The map was divided into fixed segments (default 16). Each segment had its own `ReentrantLock`. A put() locked the entire segment.
- Why Used: Better than locking the whole map (like Hashtable), but still coarse-grained.
- When Used: Every write operation in Java 7 ConcurrentHashMap.
- Simple Example:
```
Java 7 — 16 segments, each covering ~N/16 buckets
put("key") → hash → segment 3 → lock segment 3 → insert
// all other keys in segment 3 are blocked during this insert
```

---

**Bucket-Level Locking (Java 8+)**
- Definition: Lock is taken only on the head node of a single bucket, not a segment.
- Why Used: Far more granular than segment locking. Only one bucket is blocked, not 1/16th of the map.
- When Used: Every write to a non-empty bucket in Java 8+.
- Simple Example:
```
Java 8 — 1 million buckets possible
put("key") → hash → bucket 42 → lock bucket 42 head node
// all other 999,999 buckets are completely free
```

---

**Unsafe / VarHandle**
- Definition: `sun.misc.Unsafe` (Java 8 and earlier) and `java.lang.invoke.VarHandle` (Java 9+) provide direct memory access and atomic operations that bypass normal Java safety checks.
- Why Used: Standard Java has no way to do CAS on an array element. Unsafe/VarHandle makes it possible.
- When Used: Reading/writing the table array atomically, CAS on bucket slots, volatile reads of node fields.
- Simple Example:
```java
// What JDK does internally (simplified)
// Unsafe.compareAndSwapObject(table, offset, null, newNode)
// → atomically sets table[i] = newNode only if table[i] == null
```

---

**Concurrent Hazards**
- Definition: Problems that arise when multiple threads access shared data without proper coordination.
- Why Used (to know): Understanding hazards explains why ConcurrentHashMap was designed the way it is.
- Types:
  - Race Condition: Two threads read-modify-write the same value, one update is lost.
  - Visibility Problem: One thread's write is not seen by another thread (CPU cache issue).
  - Deadlock: Two threads each hold a lock the other needs.
  - ABA Problem: CAS sees expected value, but it was changed and changed back — CAS succeeds incorrectly.
- Simple Example:
```java
// Race condition with plain HashMap
map.put("count", map.get("count") + 1); // NOT thread-safe
// Thread A reads 5, Thread B reads 5, both write 6 — one increment is lost

// Safe with ConcurrentHashMap
map.merge("count", 1, Integer::sum); // atomic
```

---

**Forwarding Node**
- Definition: A special sentinel node placed in a bucket when that bucket has been fully transferred to the new (larger) table during resize.
- Why Used: Tells other threads "this bucket has moved — go look in the new table."
- When Used: During resize/rehash in Java 8+.
- Simple Example:
```
Resize in progress:
  bucket[5] in old table → transfer complete → place ForwardingNode at bucket[5]
  Thread B calls get("key") → lands on bucket[5] → sees ForwardingNode → reads from new table
  Thread C calls put("key") → lands on bucket[5] → sees ForwardingNode → helps resize, then retries
```

---

**Resize Mechanism**
- Definition: When the map exceeds its load factor threshold, the table doubles in size and all entries are rehashed into the new table.
- Why Used: Keeps bucket chains short for O(1) average performance.
- When Used: When `size > capacity * loadFactor` (default 0.75).
- Java 8 key detail: Resize is **cooperative** — multiple threads help transfer buckets in parallel. Each thread claims a stripe of buckets to transfer.
- Simple Example:
```
Table size = 16, load factor = 0.75 → resize triggers at 12 entries
New table size = 32
Thread 1 transfers buckets 0–7
Thread 2 transfers buckets 8–15  (simultaneously)
```

---

**Negative Hash / Special Markers**
- Definition: ConcurrentHashMap reserves negative hash values as internal signals. Normal keys always have non-negative hashes (the spread function forces the sign bit to 0).
- Why Used: Lets threads detect special bucket states without a separate field.
- Values:

| Hash Value | Constant | Meaning |
|---|---|---|
| `-1` | `MOVED` | Bucket holds a ForwardingNode — resize in progress |
| `-2` | `TREEBIN` | Bucket holds a TreeBin (red-black tree root) |
| `-3` | `RESERVED` | Bucket is reserved by a `compute` call in progress |

- Simple Example:
```java
// Inside ConcurrentHashMap.put() — actual JDK logic pattern
if (fh == MOVED)          // hash == -1
    tab = helpTransfer(tab, f);   // join the resize
else if (fh == TREEBIN)   // hash == -2
    // use tree operations
```

## 2. Java 7 vs Java 8+ Comparison

| Aspect | Java 7 | Java 8+ |
|---|---|---|
| Internal structure | Array of Segments, each containing an array of buckets | Single flat array of Node buckets (same as HashMap) |
| Locking strategy | `ReentrantLock` per Segment | `synchronized` on bucket head node |
| Lock granularity | 1/16th of the map (default 16 segments) | Single bucket |
| CAS usage | Limited — mainly for size counter | Used for empty-bucket insert, table init, size counter |
| Segment class | Yes — `Segment<K,V> extends ReentrantLock` | Removed entirely |
| Concurrency level | Fixed at construction time (default 16) | Dynamic — scales with table size |
| Resize | Per-segment resize, independent | Whole-table resize, cooperative multi-thread transfer |
| ForwardingNode | Not present | Yes — placed in transferred buckets |
| Tree buckets | No — only linked lists | Yes — converts to Red-Black Tree at 8 entries |
| get() locking | Lock-free (volatile reads) | Lock-free (volatile reads) |
| Null keys/values | Not allowed | Not allowed |
| `Unsafe` usage | Yes | Yes (VarHandle in Java 9+) |
| Special hash markers | Not used | `-1` MOVED, `-2` TREEBIN, `-3` RESERVED |

## 3. Internal Flow Explanation

### put() — Step by Step

```
1. Compute hash:
   hash = spread(key.hashCode())
   → spread() forces sign bit to 0, ensuring hash is always >= 0
   → this is why negative hashes are safe to use as internal markers

2. Find bucket index:
   index = hash & (table.length - 1)

3. Check if table is initialized:
   → if table is null, initialize it using CAS (only one thread wins, others spin)

4. Read bucket head node (volatile read — no lock):
   f = table[index]

5. If bucket is EMPTY (f == null):
   → CAS insert: compareAndSet(table[index], null, newNode)
   → if CAS succeeds → done, no lock taken
   → if CAS fails (another thread inserted) → loop and retry from step 4

6. If bucket head hash == MOVED (-1):
   → a resize is in progress
   → this thread helps transfer buckets to the new table (helpTransfer)
   → after helping, retry the put() in the new table

7. If bucket is NON-EMPTY (normal node or tree):
   → synchronized(f) { ... }   ← lock only this bucket's head node
   → traverse the chain:
       a. If key exists → update value
       b. If key not found → append new node at end of chain
   → after insert, check if chain length >= TREEIFY_THRESHOLD (8)
       → if yes, convert bucket to Red-Black Tree

8. Increment size counter (distributed CounterCell — CAS-based)
9. Check if resize is needed → trigger if size > capacity * 0.75
```

---

### get() — Step by Step

```
1. Compute hash, find bucket index (same as put)
2. Volatile read of table[index] — no lock
3. If bucket is empty → return null
4. Check head node:
   → if head.hash == hash && head.key == key → return head.value
5. If head.hash < 0 (negative):
   → ForwardingNode (-1): delegate find() to the new table
   → TreeBin (-2): use tree search
6. Otherwise: traverse linked list until key found or end
7. Return value or null — entirely lock-free
```

---

### resize() — Step by Step

```
1. A thread detects size > threshold → calls transfer()
2. New table allocated at double the size
3. Table is divided into stripes (minimum 16 buckets per stripe)
4. Each thread atomically claims a stripe using CAS on transferIndex
5. For each bucket in the stripe:
   a. Lock the bucket head (synchronized)
   b. Split entries into "low" and "high" bins (same trick as HashMap)
   c. Place low bin at same index in new table
   d. Place high bin at index + oldCapacity in new table
   e. Place ForwardingNode in old table bucket
6. When all buckets are transferred, swap table reference to new table
7. Any thread that encounters a ForwardingNode during put/get
   → automatically redirected to new table
```

## 4. Special Internal Signals and Behaviors

**What negative hash values mean**

Normal keys: `spread(hashCode())` always produces a value with the top bit cleared → always `>= 0`.
This frees up negative values as internal signals that no real key can ever produce.

```
hash >= 0  → normal node (linked list entry)
hash == -1 → MOVED       → ForwardingNode, bucket was transferred
hash == -2 → TREEBIN     → TreeBin, bucket is a red-black tree
hash == -3 → RESERVED    → ReservationNode, compute() is in progress on this key
```

---

**What "forwarding" means**

When a bucket is fully transferred to the new table during resize, the old bucket slot is replaced with a `ForwardingNode`. This node holds a reference to the new table.

- A `get()` hitting a ForwardingNode calls `ForwardingNode.find()` → searches the new table.
- A `put()` hitting a ForwardingNode calls `helpTransfer()` → joins the resize work, then retries.
- This means no thread ever blocks waiting for resize to finish — it helps instead.

---

**How threads detect moved buckets**

```java
// Inside put() — simplified
Node f = tabAt(tab, i);          // volatile read
if (f == null) {
    // CAS insert
} else if (f.hash == MOVED) {    // -1 detected
    tab = helpTransfer(tab, f);  // join resize, get reference to new table
    // loop restarts with new table
}
```

---

**What "helping resize" means**

Instead of one thread doing all the rehashing while others wait, every thread that tries to write during a resize automatically picks up a stripe of buckets to transfer. This distributes the O(n) resize cost across all active writer threads.

```
Thread 1: triggered resize, transfers buckets 48–63
Thread 2: tried put(), saw MOVED, now transfers buckets 32–47
Thread 3: tried put(), saw MOVED, now transfers buckets 16–31
→ resize finishes 3x faster
```

---

**RESERVED hash (-3) and compute()**

When `compute()` or `computeIfAbsent()` is called on a key that maps to an empty bucket, a `ReservationNode` (hash = -3) is placed in that bucket first. This blocks other threads from inserting into the same bucket while the compute function runs. Once the function returns, the ReservationNode is replaced with the real node.

## 5. Mini Code Examples

**CAS-style update idea**

```java
AtomicReference<String> ref = new AtomicReference<>("initial");

// only updates if current value is still "initial"
boolean success = ref.compareAndSet("initial", "updated");
System.out.println(success);  // true
System.out.println(ref.get()); // "updated"

// second attempt fails — value is no longer "initial"
boolean retry = ref.compareAndSet("initial", "again");
System.out.println(retry); // false
```

---

**Synchronized bucket update (how ConcurrentHashMap does it internally)**

```java
class BucketNode {
    String key;
    int value;
    BucketNode next;
}

BucketNode head = getBucketHead(key);

synchronized (head) {
    BucketNode curr = head;
    while (curr != null) {
        if (curr.key.equals(key)) {
            curr.value = newValue; // update in place
            return;
        }
        curr = curr.next;
    }
    // key not found — append new node
}
// only this bucket is locked; all other buckets run freely
```

---

**Simple atomic counter example**

```java
AtomicLong counter = new AtomicLong(0);

// safe to call from multiple threads simultaneously
void increment() {
    counter.incrementAndGet();
}

// CAS-based accumulation — same pattern ConcurrentHashMap uses for size()
void add(long delta) {
    counter.getAndAdd(delta);
}
```

---

**Safe concurrent insertion idea**

```java
ConcurrentHashMap<String, List<String>> map = new ConcurrentHashMap<>();

// WRONG — two separate operations, not atomic
if (!map.containsKey("group")) {
    map.put("group", new ArrayList<>());  // race condition here
}
map.get("group").add("item");

// CORRECT — computeIfAbsent is a single atomic operation
map.computeIfAbsent("group", k -> new CopyOnWriteArrayList<>()).add("item");
```

## 6. Interview-Oriented Summary

### 10–15 Key Points to Remember

1. Java 8+ uses **bucket-level locking** (`synchronized` on head node), not segment locking.
2. **Empty bucket insert uses CAS** — no lock is ever taken for the first entry in a bucket.
3. **get() is always lock-free** — it only reads `volatile` fields.
4. Negative hash values are **internal signals**: `-1` = MOVED, `-2` = TREEBIN, `-3` = RESERVED.
5. `ForwardingNode` is placed in a bucket after it is transferred during resize — it redirects reads and writes to the new table.
6. Resize is **cooperative** — threads that encounter a ForwardingNode during a write help transfer other buckets instead of waiting.
7. Java 7 used `Segment extends ReentrantLock` — removed entirely in Java 8.
8. `spread()` forces the top bit of the hash to 0, ensuring all real keys have non-negative hashes.
9. `size()` uses distributed `CounterCell` objects to avoid contention — the result is approximate.
10. Null keys and null values are **never allowed** — ambiguity between "absent" and "mapped to null" in concurrent reads.
11. `compute()` on an empty bucket places a `ReservationNode` (hash = -3) to block concurrent inserts on the same key.
12. Buckets convert to **Red-Black Trees** at 8 entries (TREEIFY_THRESHOLD) — worst-case get/put becomes O(log n).
13. `Unsafe` (Java 8) / `VarHandle` (Java 9+) is used for CAS on array slots — standard Java cannot do this.
14. The ABA problem is not a concern here because ConcurrentHashMap only CAS-inserts into `null` slots — it never CAS-swaps one non-null node for another.
15. `computeIfAbsent` on the **same key** inside its own callback will deadlock — the outer call holds the bucket lock.

---

### Common Interview Traps

| Trap | Reality |
|---|---|
| "ConcurrentHashMap is fully lock-free" | Wrong. get() is lock-free. put() uses CAS or synchronized. |
| "It uses segment locking" | Only Java 7. Java 8+ uses bucket-level synchronized. |
| "size() is accurate" | No. It is approximate. Use `mappingCount()` for large maps. |
| "It is safe to do get() then put() atomically" | No. Use `computeIfAbsent` or `putIfAbsent` instead. |
| "Iteration sees a consistent snapshot" | No. Iterators are weakly consistent — they may miss concurrent updates. |
| "Null values are allowed like HashMap" | No. Both null keys and null values throw NullPointerException. |
| "Resize blocks all threads" | No. Resize is cooperative — writers help transfer buckets in parallel. |
| "synchronized on the whole map during put()" | No. Only the single bucket head is locked. |

---

### Quick Mental Models

```
Empty bucket  → CAS (no lock, fast path)
Non-empty bucket → synchronized(head) (only this bucket blocked)
Bucket hash == -1 → resize in progress → help and retry
Bucket hash == -2 → tree bucket → use tree operations
get() → always lock-free, just volatile reads

Java 7: map → [seg0][seg1]...[seg15] → each seg has its own ReentrantLock
Java 8: map → [b0][b1][b2]...[bN]   → each bucket head is the lock object

Resize: old table → ForwardingNodes → new table (double size)
        multiple threads transfer stripes in parallel
```
