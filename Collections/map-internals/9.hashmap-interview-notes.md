## 1. Why is ConcurrentHashMap faster than Hashtable?

**Hashtable** uses a **single lock** on the entire map — only one thread can do anything at a time, including reads.

**ConcurrentHashMap** uses **node-level locking** (Java 8+) and **CAS (Compare-And-Swap)** — reads need no lock at all, writes only lock the affected bucket.

| | Hashtable | ConcurrentHashMap |
|---|---|---|
| Lock scope | Entire map | Single node/bucket |
| Read locking | Yes — blocks all reads | No — reads are lock-free |
| Write concurrency | One thread at a time | Multiple threads, different buckets |
| Null keys/values | No | No |
| Legacy | Yes (Java 1.0) | No (Java 1.5+) |

- **Pre Java 8** — segment-level locking (16 segments by default)
- **Java 8+** — node-level locking + CAS for empty bins

**Gotcha**: Never manually `synchronized` on a `ConcurrentHashMap` — it defeats the purpose.

## 2. How do you synchronize a HashMap?

Three options — know the trade-offs:

```java
// Option 1 — synchronizedMap (simple but slow)
Map<String, Integer> map = Collections.synchronizedMap(new HashMap<>());

// iteration MUST be manually synchronized
synchronized (map) {
    for (Map.Entry<String, Integer> e : map.entrySet()) { }
}

// Option 2 — ConcurrentHashMap (preferred)
Map<String, Integer> map = new ConcurrentHashMap<>();
map.putIfAbsent("key", 1);      // atomic
map.replace("key", 1, 100);     // atomic compare-and-replace
// no manual sync needed for iteration

// Option 3 — synchronized block (manual, fine-grained)
private final Object lock = new Object();
synchronized (lock) {
    if (!map.containsKey("key")) map.put("key", 1);
}
```

| Method | Performance | Iteration Safe | Use When |
|---|---|---|---|
| `synchronizedMap` | Low | Manual sync needed | Legacy, simple use |
| `ConcurrentHashMap` | High | Yes | Modern concurrent apps |
| `synchronized` block | Medium | Manual | Custom multi-op atomicity |

## 3. How does LinkedHashMap maintain insertion order?

`LinkedHashMap` = `HashMap` + **doubly linked list**.

On every `put`, the node is inserted into the hash table **and** appended to the linked list. Iteration walks the linked list — so order is preserved.

```java
Map<String, Integer> map = new LinkedHashMap<>();
map.put("c", 3);
map.put("a", 1);
map.put("b", 2);
map.forEach((k, v) -> System.out.print(k + " ")); // c a b — insertion order
```

**Access-order mode** (LRU cache pattern):

```java
// true = access-order, false = insertion-order (default)
Map<String, Integer> lru = new LinkedHashMap<>(16, 0.75f, true) {
    protected boolean removeEldestEntry(Map.Entry<String, Integer> eldest) {
        return size() > 3; // evict when size exceeds 3
    }
};
lru.put("a", 1); lru.put("b", 2); lru.put("c", 3);
lru.get("a");    // "a" is now most recently used
lru.put("d", 4); // evicts "b" (least recently used)
System.out.println(lru.keySet()); // [c, a, d]
```

- Extra memory — each node has `before` / `after` pointers
- Slightly slower than `HashMap` due to linked list maintenance
- Best use case: **LRU cache**

## 4. What is the difference between HashMap and IdentityHashMap?

`HashMap` compares keys using `.equals()` and `.hashCode()`.
`IdentityHashMap` compares keys using `==` (reference equality) and `System.identityHashCode()`.

```java
String a = new String("hello");
String b = new String("hello");

Map<String, Integer> hashMap = new HashMap<>();
hashMap.put(a, 1);
hashMap.put(b, 2);
System.out.println(hashMap.size()); // 1 — a.equals(b) is true, b overwrites a

Map<String, Integer> identityMap = new IdentityHashMap<>();
identityMap.put(a, 1);
identityMap.put(b, 2);
System.out.println(identityMap.size()); // 2 — a != b (different references)
```

| | HashMap | IdentityHashMap |
|---|---|---|
| Key equality | `.equals()` | `==` |
| Hashing | `.hashCode()` | `System.identityHashCode()` |
| Same content, different objects | Treated as same key | Treated as different keys |
| Use case | General purpose | Object identity tracking, serialization internals |

## 5. What is the difference between HashMap and WeakHashMap?

`HashMap` holds **strong references** to keys — entries stay until you remove them.
`WeakHashMap` holds **weak references** to keys — entries are removed automatically when the key is garbage collected.

```java
Map<Object, String> weak = new WeakHashMap<>();
Object key = new Object();
weak.put(key, "value");

System.out.println(weak.size()); // 1

key = null;      // remove the only strong reference
System.gc();
Thread.sleep(100);

System.out.println(weak.size()); // 0 — entry auto-removed after GC
```

| | HashMap | WeakHashMap |
|---|---|---|
| Key reference | Strong | Weak |
| Value reference | Strong | Strong |
| Entry removal | Manual | Automatic on GC |
| Use case | General purpose | Caches, metadata tied to object lifecycle |

**Gotcha**: Values are still strongly referenced. If a value holds a reference back to its key, the key will never be GC'd — the entry will never be removed.

## 6. Quick Reference — All Map Types

| Map | Ordered | Thread-safe | Null key | Key equality | Use case |
|---|---|---|---|---|---|
| `HashMap` | No | No | Yes (one) | `.equals()` | General purpose |
| `LinkedHashMap` | Insertion / Access | No | Yes | `.equals()` | LRU cache, ordered iteration |
| `ConcurrentHashMap` | No | Yes | No | `.equals()` | Concurrent access |
| `Hashtable` | No | Yes (slow) | No | `.equals()` | Legacy only |
| `IdentityHashMap` | No | No | Yes | `==` | Reference identity tracking |
| `WeakHashMap` | No | No | Yes | `.equals()` | Auto-evicting caches |
