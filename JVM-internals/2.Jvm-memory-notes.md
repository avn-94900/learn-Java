# JVM Memory and Security Notes


## 1. JVM Memory Architecture Overview

JVM memory is split into two main categories: **heap memory** and **non-heap memory**.

```
JVM Memory
├── Heap Memory
│   ├── Young Generation
│   │   ├── Eden Space
│   │   ├── Survivor 0 (S0)
│   │   └── Survivor 1 (S1)
│   └── Old Generation (Tenured)
└── Non-Heap Memory
    ├── Stack Memory (per thread)
    ├── Method Area / Metaspace
    ├── PC Registers
    └── Native Method Stacks
```

### Memory Areas

| Area | Responsibility |
|---|---|
| Heap | Stores all objects and instance variables. Shared across threads. |
| Stack | Stores method calls, local variables, references. One per thread. |
| Metaspace | Stores class metadata, static variables, method bytecode. |
| PC Registers | Tracks current instruction per thread. |
| Native Method Stacks | Used for JNI calls to native code. |

### Process Memory Layout

```
┌─────────────────┐
│   Heap Memory   │  Objects, arrays
├─────────────────┤
│   Metaspace     │  Class metadata (native memory)
├─────────────────┤
│ Thread Stacks   │  One per thread
├─────────────────┤
│ Direct Memory   │  NIO, off-heap allocations
├─────────────────┤
│ Code Cache      │  Compiled native code (JIT)
└─────────────────┘
```

### JVM Memory Flags

| Flag | Purpose |
|---|---|
| `-Xms` / `-Xmx` | Minimum and maximum heap size |
| `-Xss` | Stack size per thread (default ~1 MB) |
| `-XX:MaxMetaspaceSize` | Cap for Metaspace growth |
| `-XX:MaxDirectMemorySize` | Limit for direct (off-heap) memory |
| `-XX:ReservedCodeCacheSize` | Limit for JIT code cache |


### Standard Common Values

| Application Type            | Typical Heap Config     |
| --------------------------- | ----------------------- |
| Small utility app           | `-Xms256m -Xmx512m`     |
| Normal Spring Boot app      | `-Xms512m -Xmx2g`       |
| Medium microservice         | `-Xms1g -Xmx4g`         |
| Large enterprise app        | `-Xms4g -Xmx8g` or more |
| Heavy batch/data processing | `8g+`                   |

## 2. Heap Memory

### Young Generation

All new objects are allocated in **Eden Space**. This area is collected frequently via **Minor GC**.

- **Eden Space** - All new objects start here. Minor GC triggers when Eden fills.
- **Survivor Spaces (S0 / S1)** - Hold objects that survived at least one Minor GC. Only one is active at a time.
- Objects alternate between S0 and S1 on each Minor GC cycle.
- After surviving the configured number of GC cycles, objects move to the Old Generation.

### Old Generation (Tenured)

- Holds long-lived objects promoted from the Young Generation.
- Larger than Young Generation (default ratio ~1:2).
- Collected during **Major GC** or **Full GC**, which are less frequent but more expensive.

### Object Lifecycle

```
New Object
  → Eden Space
  → Minor GC
  → Survivor Space (S0/S1)
  → Multiple GC Cycles
  → Old Generation
  → Major GC
  → Collected
```

## 3. Object Promotion

**Object Promotion** means moving an object from **Young Generation** to **Old Generation**.

### Promotion Triggers

* Objects surviving multiple Minor GCs get promoted after reaching age limit (`MaxTenuringThreshold`).
* If Survivor space becomes full, JVM promotes objects early.
* Large objects may directly move to Old Generation to avoid repeated copying.
* JVM may promote objects based on runtime memory and GC behavior (ergonomics).

| Trigger | Reason |
|---|---|
| Object survived many Minor GCs | Reached tenuring threshold (`MaxTenuringThreshold`) |
| Survivor space is full | No space to keep surviving objects |
| Object is too large | Avoid expensive copying |
| JVM ergonomics | JVM runtime optimization decisions |

## 4. TLAB (Thread Local Allocation Buffer)

A **TLAB** is a small private memory area inside Eden assigned to each thread.

- Each thread gets its own TLAB inside Eden space.
- Object allocation inside TLAB needs no synchronization, making it very fast.
- When TLAB is full, the thread requests a new TLAB from Eden.
- When Eden is exhausted, a Minor GC is triggered.
- Large objects bypass TLAB and allocate directly in the heap.

### TLAB Flow

| Step | Description |
|---|---|
| Thread gets TLAB | Private memory assigned from Eden |
| Object allocation | Happens without synchronization |
| TLAB full | Thread requests a new TLAB |
| Eden full | Minor GC triggered |
| Large object | Bypasses TLAB directly |

## 5. Humongous Objects (G1GC)

A **Humongous Object** is any object whose size is greater than 50% of a G1 heap region size.

- These objects skip the normal Young Generation lifecycle.
- They are allocated directly in the Old Generation.
- Too many humongous objects reduce GC performance.
- Increase `-XX:G1HeapRegionSize` to reduce humongous object creation.

| Concept | Description |
|---|---|
| Humongous Object | Object larger than 50% of G1 region size |
| Allocation Area | Directly in Old Generation |
| Young Gen Usage | Skipped |
| Optimization | Increase `-XX:G1HeapRegionSize` |

## 6. Stack Memory

### Characteristics

| Aspect | Details |
|---|---|
| Scope | Thread-specific, each thread has its own stack |
| Structure | LIFO (Last In, First Out) |
| Contents | Method calls, local variables, object references |
| Management | Automatic, cleared when method returns |
| Speed | Very fast allocation and deallocation |

### What Stack Stores

- Method call frames
- Local primitive variables (`int`, `double`, `boolean`, etc.)
- Object references (the pointer only, not the object itself)
- Method parameters and return addresses

> **Note:** Objects themselves are never stored on the stack. Only references are stored.

### Stack Frame

Each method call pushes a new **Stack Frame** containing:

- **Local Variable Array** - Parameters and local variables
- **Operand Stack** - Intermediate calculation values
- **Frame Data** - Method metadata, return address, exception table

When a method returns, its frame is popped and local variables are cleared.

### Stack and Heap Interaction

```java
public void example() {
    int primitive = 42;                    // Stack: stores value directly
    String reference = "Hello";            // Stack: reference → Heap: String Pool
    List<String> list = new ArrayList<>(); // Stack: reference → Heap: ArrayList object
}
```

### Stack Size Flags

```
-Xss512k   # Lightweight apps with shallow call stacks
-Xss2m     # Apps with deep recursion or many nested calls
```

### StackOverflowError

- Thrown when a thread's stack exceeds its configured size limit.
- Most common cause is **infinite or unintended deep recursion**.
- Fix: add correct base cases, convert recursion to iteration, or increase `-Xss`.

## 7. Method Area and Metaspace

### What Method Area Stores

- Class metadata: structure, method signatures, field definitions
- Method bytecode
- Runtime Constant Pool: symbolic references and class-level constants
- Static variables and static method references

> **Note:** String literals are stored in the **String Pool**, which lives in the **Heap**, not in Metaspace.

### PermGen vs Metaspace

| Aspect | PermGen (Java 7 and earlier) | Metaspace (Java 8+) |
|---|---|---|
| Location | Inside JVM heap | Native (OS) memory |
| Size | Fixed (`-XX:MaxPermSize`) | Dynamic growth |
| GC | Part of heap GC | Separate cleanup |
| OOM Risk | Common | Rare unless capped |
| String Pool | Stored in PermGen | Moved to Heap |

### Class Unloading

A class can be unloaded from Metaspace only when all three conditions are met:

1. No instances of the class exist on the heap.
2. The ClassLoader that loaded the class is eligible for GC.
3. The Class object has no active references.

### Causes of OutOfMemoryError: Metaspace

- Excessive dynamic class generation (frameworks, proxies, bytecode libraries)
- ClassLoader leaks during hot redeployment in app servers
- Heavy use of reflection or runtime code generation

**Fix:** Increase `-XX:MaxMetaspaceSize`, fix ClassLoader leaks, monitor classloading.

## 8. Static vs Instance Memory

```java
public class MemoryExample {

    // Stored in Metaspace (class-level data)
    private static int staticVar = 100;

    // Reference in Heap; "Hello" stored in String Pool (Heap)
    private static final String CONSTANT = "Hello";

    // Stored in Heap inside the object instance
    private int instanceVar = 50;

    public static void staticMethod() {
        // Method bytecode stored in Metaspace
        // Execution uses a Stack frame
        int localVar = 10;  // Stored in Stack during execution
    }
}
```

### Static Variable Behavior

- A single copy is shared across all instances and all threads.
- Lifecycle is tied to the ClassLoader.
- Static variables mutated by multiple threads require **synchronization**.

```java
public class Counter {
    private static volatile int count = 0; // volatile for thread visibility

    public static synchronized void increment() {
        count++; // synchronized for atomicity
    }
}
```

## 9. Memory Layout Quick Reference

```
Metaspace
├── Class Metadata
├── Method Metadata
├── Field Metadata
└── Runtime Constant Pool metadata

Heap
├── Objects
│   ├── instance variables
│   └── Static field values
└── String Pool
    └── String literals ("Hello")

Stack (per thread)
└── Method Calls → Stack Frames
    ├── local variables
    ├── parameters
    └── references

Class Area (Logical JVM Concept)
├── method bytecode
├── symbolic references
└── runtime constant pool
    (Implemented using Metaspace in modern JVM)
```

## 10. Memory Optimization Techniques

### Avoid Unnecessary Object Creation

```java
// Bad - creates a new String on every iteration
String result = "";
for (String item : items) {
    result += item;
}

// Good - reuses a single buffer
StringBuilder sb = new StringBuilder(items.size() * 16);
for (String item : items) {
    sb.append(item);
}
return sb.toString();
```

### Pre-Size Collections

```java
// Avoids rehashing by specifying initial capacity
Map<String, String> map = new HashMap<>(expectedSize, 0.75f);
List<String> list = new ArrayList<>(expectedSize);
```

### String Pool Behavior

```java
String s1 = "Hello";              // Stored in String Pool
String s2 = "Hello";              // Reuses same pool object
String s3 = new String("Hello");  // New object in Heap, outside pool
String s4 = s3.intern();          // Returns pool reference

System.out.println(s1 == s2);     // true  - same pool object
System.out.println(s1 == s3);     // false - different objects
System.out.println(s1 == s4);     // true  - both refer to pool object
System.out.println(s1.equals(s3));// true  - same content
```

### Batch Processing for Large Datasets

```java
int batchSize = 1000;
for (int i = 0; i < data.size(); i += batchSize) {
    processBatch(data.subList(i, Math.min(i + batchSize, data.size())));
    // Limits live objects at any one time, reducing Old Generation pressure
}
```

### General Optimization Practices

| Practice | Benefit |
|---|---|
| Use primitive types | Eliminates autoboxing overhead |
| Lazy initialization | Defers object creation until needed |
| Object pooling | Reuses expensive resources (connections, parsers) |
| `StringBuilder` in loops | Avoids repeated String allocation |
| Explicit listener removal | Prevents memory leaks |
| `threadLocal.remove()` in finally | Prevents leaks in thread pool environments |

## 11. Measuring Memory Usage

```java
Runtime runtime = Runtime.getRuntime();
long before = runtime.totalMemory() - runtime.freeMemory();

// Code under measurement
List<String> data = createLargeDataSet();

long after = runtime.totalMemory() - runtime.freeMemory();
System.out.println("Memory used: " + (after - before) + " bytes");
```
