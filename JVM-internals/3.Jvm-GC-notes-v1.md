## 1. JVM Memory Spaces

### PermGen (Pre-Java 8)

- Stored class metadata, methods, and interned Strings.
- Fixed size, allocated as part of the JVM heap.
- Could cause `OutOfMemoryError: PermGen space` if too many classes were loaded.
- Difficult to tune in dynamic class-loading environments such as application servers and frameworks.

### Metaspace (Java 8 onwards)

- Replaced PermGen entirely.
- Allocated in native (OS) memory, not heap.
- Grows dynamically and is limited by system memory unless explicitly capped.
- If too many classes are loaded (e.g., frameworks using proxies or dynamic code generation), native memory exhaustion can still cause server crashes.

**Key JVM flags:**

```bash
-XX:MetaspaceSize=256m        # Initial metaspace size
-XX:MaxMetaspaceSize=512m     # Maximum metaspace size (cap to prevent OOM)
```

**Monitoring:**

```bash
jcmd <pid> VM.classloaders    # Inspect classloader activity
```

### Heap Generations

The Java heap is divided into generations based on object lifespan:

- **Eden Space** - New objects are allocated here.
- **Survivor Spaces (S0, S1)** - Objects that survive a Minor GC are moved here.
- **Old Generation (Tenured)** - Objects that survive multiple GC cycles are promoted here.

## 2. Garbage Collection Fundamentals

### What Is GC?

Garbage Collection is the JVM's automatic memory management mechanism. It:

- Identifies objects that are no longer reachable from any live thread.
- Reclaims the memory occupied by those objects.
- Prevents memory leaks without requiring explicit deallocation by the developer.

Java has no `free()` or `delete` operator. The GC is the sole mechanism for reclaiming heap memory.

### Object Eligibility for GC

An object becomes eligible for GC when no reachable reference points to it:

- A reference is explicitly set to `null`.
- A reference goes out of scope (e.g., a local variable when a method returns).
- Circular references exist among objects with no external references.

```java
String str = new String("Hello");
str = null;  // The "Hello" object is now eligible for GC

static List<String> list = new ArrayList<>();
list.add("Keep me");  // Not eligible - referenced by a static variable
```

### Types of GC Events

| GC Type | Scope | Frequency | Pause Duration |
|---------|-------|-----------|----------------|
| Minor GC | Young Generation only | High (every few seconds) | Fast (milliseconds) |
| Major GC | Old Generation only | Low (minutes to hours) | Slower (tens of ms to seconds) |
| Full GC | Entire Heap + Metaspace | Rare (under memory pressure) | Slowest (seconds) |

### GC Roots

GC roots are the starting points for reachability analysis. Any object reachable from a GC root is considered live and will not be collected.

Types of GC roots:

- Local variables in active method call stacks.
- Static variables (stored in the Method Area).
- JNI (Java Native Interface) references.
- Active thread objects.
- Objects held by the system class loader.

### Mark and Sweep Algorithm

**Mark Phase:**

1. Start traversal from all GC roots.
2. Recursively follow object references.
3. Mark all reachable objects as live.

**Sweep Phase:**

1. Scan the entire heap.
2. Identify all unmarked (dead) objects.
3. Deallocate memory used by dead objects.
4. Reset marks for the next GC cycle.

**Optional Compact Phase:**

- Move all live objects contiguously.
- Eliminate heap fragmentation.
- Update all references to reflect new object locations.

### Generational Garbage Collection

Based on the empirical observation that most objects die young (the weak generational hypothesis).

**Strategy:**

- Minor GC collects the Young Generation frequently and quickly.
- Major GC collects the Old Generation infrequently but at higher cost.
- Objects that survive multiple GC cycles are promoted to the Old Generation.

**Write Barriers** are used to track references from the Old Generation into the Young Generation, ensuring Minor GC can accurately identify all live young objects.

### Concurrent vs. Parallel GC

**Parallel GC:**

- Multiple GC threads work simultaneously.
- All application threads are paused (Stop-the-World).
- Reduces total GC time but does not reduce pause duration for the application.

**Concurrent GC:**

- GC threads run alongside application threads with minimal Stop-the-World pauses.
- Requires more CPU resources.
- Better for latency-sensitive applications.

### GC Ergonomics

The JVM can auto-tune GC parameters to meet configured goals:

- **Pause Time Goal:** `-XX:MaxGCPauseMillis=200`
- **Throughput Goal:** `-XX:GCTimeRatio=99` (target 1% of time in GC)
- **Memory Footprint:** Minimize heap size

The JVM adjusts generation ratios, GC trigger points, and algorithm parameters dynamically based on observed application behavior.

Monitoring ergonomics: `-XX:+PrintGCApplicationStoppedTime`

## 3. GC Algorithms

### Serial GC

- Single-threaded GC with Stop-the-World pauses.
- Minimal memory overhead and simple implementation.
- Not suitable for multi-core production systems.

**Suitable for:** Small applications (heap under 100 MB), single-core machines, embedded systems, development and testing.

```bash
-XX:+UseSerialGC
```

### Parallel GC (Throughput Collector)

- Multi-threaded GC using multiple CPU cores.
- Still Stop-the-World, but GC work is parallelized.
- Suitable for large heaps and batch processing jobs.

**Variants:**

- **Parallel GC** - Multi-threaded Minor GC, single-threaded Major GC.
- **Parallel Old GC** - Multi-threaded for both Minor and Major GC. Was the default before Java 9.

```bash
-XX:+UseParallelGC
-XX:+UseParallelOldGC
```

**Suitable for:** Throughput-oriented workloads where latency is not the primary concern.

### CMS (Concurrent Mark-Sweep)

Deprecated as of Java 9. Included here for understanding legacy systems.

**Phases:**

1. **Initial Mark (STW):** Mark objects directly reachable from GC roots.
2. **Concurrent Mark:** Traverse and mark all reachable objects concurrently with the application.
3. **Concurrent Preclean:** Handle objects modified during the concurrent mark phase.
4. **Remark (STW):** Finalize marking of all live objects.
5. **Concurrent Sweep:** Free memory of dead objects concurrently.
6. **Concurrent Reset:** Prepare internal structures for the next cycle.

**Problems with CMS:**

- No compaction phase leads to heap fragmentation.
- Concurrent Mode Failure when CMS cannot keep up with the allocation rate.
- Background GC threads consume significant CPU.
- Promotion Failure when fragmentation prevents Young Generation objects from being promoted.

**Mitigation:** `-XX:CMSInitiatingOccupancyFraction=70`

Do not use CMS in new projects. Migrate to G1GC.

### G1 GC (Garbage First)

Default GC since Java 9. Designed for large heaps with predictable pause times.

**Key design differences from traditional collectors:**

- Heap is divided into equal-sized regions (1 MB to 32 MB) rather than fixed contiguous Young/Old areas.
- Regions are dynamically assigned as Eden, Survivor, Old, or Humongous.
- Humongous regions hold objects larger than 50% of a region size.
- Collects regions with the most garbage first (hence the name).

**Evacuation process:**

1. Select regions based on garbage ratio relative to the pause time goal.
2. Scan roots and remembered sets to find live references.
3. Copy live objects out of selected regions into new regions.
4. Update all references to point to the new object locations.

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200     # Target max pause time
-XX:G1HeapRegionSize=16m     # Explicit region size (optional)
```

**Suitable for:** Most modern server applications. Balances throughput and latency for heaps from a few GB upward.

### ZGC

Available from Java 11. Designed for ultra-low latency at any heap size.

**Key features:**

- Pause times consistently under 10 ms, regardless of heap size.
- Heap support up to 16 TB.
- Uses colored pointers to encode object state directly in the reference.
- Uses load barriers to check object state on every heap access.
- Most GC work is performed concurrently with application threads.

**Trade-offs:**

- Higher memory overhead (approximately 2-16%).
- CPU overhead from load barriers.
- Requires Java 11 or later.

```bash
-XX:+UseZGC
```

**Suitable for:** Applications with very large heaps or strict latency requirements where pauses above 10 ms are unacceptable.

### Shenandoah GC

Available in OpenJDK builds.

**Key features:**

- Concurrent evacuation - moves live objects without Stop-the-World pauses.
- Pause times typically under 10 ms.
- Different internal implementation from ZGC but similar latency goals.

```bash
-XX:+UseShenandoahGC
```

**Suitable for:** Real-time or near-real-time systems with large heaps where consistent low pause times matter more than raw throughput.

## 4. GC Algorithm Comparison

| Collector | Threads | Pause Time | Throughput | Best For |
|-----------|---------|------------|------------|----------|
| Serial GC | Single | High | Low | Small apps, embedded systems |
| Parallel GC | Multiple | Medium | High | Batch processing, throughput-focused |
| CMS | Concurrent | Low | Medium | Low-latency apps (deprecated Java 9+) |
| G1 GC | Concurrent | Low (tunable) | High | Modern default, balanced performance |
| ZGC / Shenandoah | Concurrent | Under 10 ms | High | Massive heaps, ultra-low latency |

**Decision tree:**

1. Heap under 100 MB - use Serial GC.
2. Throughput is more important than latency - use Parallel GC.
3. Balanced needs, modern application - use G1 GC (recommended default).
4. Ultra-low latency or very large heap - use ZGC or Shenandoah.
5. Legacy low-latency requirement - CMS (avoid in new projects).

## 5. Memory Tuning Parameters

### Heap Sizing

```bash
-Xms2g                     # Initial heap size
-Xmx8g                     # Maximum heap size
-XX:NewRatio=3             # Old:Young generation ratio (3:1)
```

For production environments, setting `-Xms` equal to `-Xmx` avoids heap resizing overhead.

### Stack Sizing

```bash
-Xss1m                     # Stack size per thread
```

### Metaspace

```bash
-XX:MetaspaceSize=256m     # Initial metaspace size
-XX:MaxMetaspaceSize=512m  # Maximum metaspace size
```

### GC Tuning

```bash
-XX:+UseG1GC               # Select G1 collector
-XX:MaxGCPauseMillis=100   # Target maximum pause time
-XX:G1HeapRegionSize=16m   # G1 region size
```

### GC Logging

```bash
# Java 9 and later
-Xlog:gc*

# Pre-Java 9
-verbose:gc
-XX:+PrintGC
-XX:+PrintGCDetails
-XX:+PrintGCTimeStamps
```

### Heap Dump on OOM

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dumps/
```

### Heap Sizing Guidelines

Factors to consider:

- Leave sufficient memory for the OS and other processes.
- Account for expected object allocation rates.
- Larger heaps reduce GC frequency but increase individual pause duration.
- As a starting point, allocate 25-50% of available RAM to the JVM heap.

Example for a 16 GB system:

```
OS + other processes: ~4 GB
Available for JVM: ~12 GB
Recommended heap: -Xms6g -Xmx8g
```

## 6. Memory Optimization Techniques

### Avoiding Memory Leaks

Common leak patterns:

```java
// Static collections that grow without bound
private static List<Object> staticList = new ArrayList<>();

// Event listeners that are never removed
button.addActionListener(listener);  // Must call removeActionListener in cleanup

// ThreadLocal values that are never cleared
private static ThreadLocal<Object> threadLocal = new ThreadLocal<>();
// Always call threadLocal.remove() in a finally block

// Non-static inner classes holding an implicit reference to the outer instance
public class OuterClass {
    class InnerClass {
        // Holds a reference to OuterClass, preventing its GC
    }
}
```

Correct approach using a bounded cache:

```java
private static final int MAX_SIZE = 1000;
private static final Map<String, String> cache =
    new LinkedHashMap<String, String>(16, 0.75f, true) {
        protected boolean removeEldestEntry(Map.Entry<String, String> eldest) {
            return size() > MAX_SIZE;
        }
    };
```

### Reducing Object Allocation

Avoid creating unnecessary temporary objects:

```java
// Poor - creates a new String object on every iteration
public String concatenateStrings(List<String> strings) {
    String result = "";
    for (String s : strings) {
        result += s;
    }
    return result;
}

// Better - StringBuilder reuses its internal buffer
public String concatenateStrings(List<String> strings) {
    StringBuilder sb = new StringBuilder(strings.size() * 16);
    for (String s : strings) {
        sb.append(s);
    }
    return sb.toString();
}

// Best for simple cases (Java 8+)
public String concatenateStrings(List<String> strings) {
    return String.join("", strings);
}
```

### Resource Management

Always use try-with-resources for I/O and other closeable resources:

```java
try (FileInputStream fis = new FileInputStream("file.txt");
     BufferedReader reader = new BufferedReader(new InputStreamReader(fis))) {
    return reader.lines().collect(Collectors.toList());
}
// Resources are automatically closed even if an exception occurs
```

### ThreadLocal Cleanup

```java
private static ThreadLocal<Object> threadLocal = new ThreadLocal<>();

public void process() {
    threadLocal.set(new Object());
    try {
        // Use the ThreadLocal value
    } finally {
        threadLocal.remove();  // Prevents memory leaks in thread pool environments
    }
}
```

### Additional Practices

- Use object pooling for expensive-to-create objects (use cautiously - pooling can cause memory retention).
- Prefer primitive types over boxed types to avoid autoboxing overhead.
- Apply lazy initialization where appropriate.
- Size collections explicitly when the expected capacity is known.
- Prefer immutable objects; they tend to be shorter-lived and are more GC-friendly.

## 7. Monitoring and Tooling

### Runtime Memory Information

```java
Runtime runtime = Runtime.getRuntime();
long maxMemory   = runtime.maxMemory();    // Corresponds to -Xmx
long totalMemory = runtime.totalMemory();  // Currently allocated from the OS
long freeMemory  = runtime.freeMemory();   // Available within allocated heap

long usedMemory  = totalMemory - freeMemory;
System.out.printf("Used: %d MB, Free: %d MB, Max: %d MB%n",
    usedMemory  / 1024 / 1024,
    freeMemory  / 1024 / 1024,
    maxMemory   / 1024 / 1024);
```

`System.gc()` is a hint to the JVM and is not guaranteed to trigger a collection.

### Command-line Tools

```bash
jstat -gc -t <pid> 250ms       # GC statistics sampled every 250 ms
jmap -histo <pid>               # Histogram of object types and counts
jmap -dump:file=heap.hprof <pid>  # Generate a heap dump file
jcmd <pid> VM.classloaders      # Inspect classloader hierarchy
```

### Analysis Tools

| Tool | Purpose |
|------|---------|
| Eclipse MAT | Heap dump analysis - identify retained objects and leak suspects |
| JVisualVM | Real-time monitoring with heap dump and profiling support |
| JConsole | JMX-based monitoring of memory, threads, and MBeans |
| Java Flight Recorder | Low-overhead continuous profiling built into the JDK |
| GCViewer | GC log analysis and visualization |
| JProfiler | Commercial profiler with deep call tree and memory analysis |
| AppDynamics / New Relic | APM tools for production monitoring |

### Key GC Log Metrics to Track

- GC frequency - how often collections are triggered.
- GC pause duration - time the application is stopped per GC event.
- Throughput - percentage of time the application runs vs. time spent in GC.
- Heap utilization pattern - is Old Generation growing steadily over time?

**Warning signs in GC logs:**

- Frequent Full GC events.
- Pause times exceeding 100 ms in a latency-sensitive application.
- Continuously increasing Old Generation usage.
- OutOfMemoryError entries.

## 8. Common Errors and Troubleshooting

### OutOfMemoryError Variants

| Error | Root Cause | Resolution |
|-------|-----------|------------|
| `Java heap space` | Heap exhausted | Increase `-Xmx`; investigate memory leaks |
| `Metaspace` | Excessive class loading | Increase `-XX:MaxMetaspaceSize`; check for classloader leaks |
| `GC overhead limit exceeded` | Over 98% of time in GC with under 2% memory freed | Increase heap size; optimize allocation patterns |
| `Direct buffer memory` | NIO direct buffers exhausted | Increase `-XX:MaxDirectMemorySize` |

### StackOverflowError

**Common causes:**

- Infinite or very deep recursion.
- Very large local variable allocations.
- Stack size configured too small.

**Debugging steps:**

1. Examine the stack trace for repeated method call patterns indicating infinite recursion.
2. Temporarily increase the stack size with `-Xss2m` to determine if the issue is size or logic.
3. Fix the recursive logic or convert to an iterative implementation.

```java
// Missing base case causes StackOverflowError
public int factorial(int n) {
    return n * factorial(n - 1);
}

// Corrected with a base case
public int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
```

### Troubleshooting Workflow

1. Enable GC logging and reproduce the issue.
2. Generate a heap dump on OutOfMemoryError with `-XX:+HeapDumpOnOutOfMemoryError`.
3. Open the heap dump in Eclipse MAT and run the Leak Suspects report.
4. Identify the dominator tree - the objects retaining the most memory.
5. Trace back to the root cause in application code.
6. Apply fixes and verify with GC log analysis using GCViewer.
