## 1. JVM Memory Architecture Overview

JVM memory is divided into two main categories: heap memory and non-heap memory.

```
JVM Memory
├── Heap Memory (Objects)
│   ├── Young Generation
│   │   ├── Eden Space
│   │   ├── Survivor 0 (S0)
│   │   └── Survivor 1 (S1)
│   └── Old Generation (Tenured)
│
└── Non-Heap Memory
    ├── Stack Memory (per thread)
    ├── Method Area / Metaspace
    ├── PC Registers
    └── Native Method Stacks
```

### Memory Area Responsibilities

- **Heap** - Stores all objects and instance variables. Shared across all threads.
- **Stack** - Stores method calls, local variables, and object references. One stack per thread.
- **Method Area / Metaspace** - Stores class-level metadata, static variables, and method bytecode.
- **PC Registers** - Each thread has a program counter that tracks the current instruction being executed.
- **Native Method Stacks** - Used for JNI (Java Native Interface) calls to native code.

### Process Memory Layout

```
Process Memory Layout:
┌─────────────────┐
│   Heap Memory   │  Objects, arrays (-Xmx controls size)
├─────────────────┤
│   Metaspace     │  Class metadata (native memory)
├─────────────────┤
│ Thread Stacks   │  One per thread (-Xss controls size)
├─────────────────┤
│ Direct Memory   │  NIO, off-heap allocations
├─────────────────┤
│ Code Cache      │  Compiled native code (JIT)
└─────────────────┘
```

### JVM Memory Flags

- **Heap:** `-Xms` / `-Xmx` for minimum and maximum size.
- **Stack:** `-Xss` per thread (default approximately 1 MB).
- **Metaspace:** Grows dynamically in native memory; cap with `-XX:MaxMetaspaceSize`.
- **Direct Memory:** Limited by system memory; tune with `-XX:MaxDirectMemorySize`.
- **Code Cache:** Controlled with `-XX:ReservedCodeCacheSize`.

## 2. Heap Memory

### Young Generation

All new objects are allocated in the Young Generation, specifically in Eden Space. The Young Generation is designed for short-lived objects and is collected frequently via Minor GC.

**Eden Space**

- All new objects start here, except very large objects which may go directly to the Old Generation.
- When Eden fills, a Minor GC is triggered.

**Survivor Spaces (S0 and S1)**

- Hold objects that have survived at least one Minor GC.
- Only one survivor space is active at any given time.
- Objects alternate between S0 and S1 during successive Minor GC cycles.
- After surviving a configured number of GC cycles (default 15, controlled by `-XX:MaxTenuringThreshold`), objects are promoted to the Old Generation.

### Old Generation (Tenured)

- Holds long-lived objects promoted from the Young Generation.
- Larger than the Young Generation (default Young:Old ratio is approximately 1:2).
- Collected during Major GC or Full GC, which are less frequent but more expensive.

### Object Lifecycle

```
New Object → Eden Space → Minor GC → Survivor Space (S0/S1)
→ Multiple GC Cycles → Old Generation → Major GC → Collected
```

1. Object is allocated in Eden.
2. If reachable at the time of the first Minor GC, it is copied to a Survivor space.
3. In subsequent Minor GC cycles it moves between S0 and S1.
4. After surviving the tenuring threshold, it is promoted to the Old Generation.
5. It is eventually collected when no reachable reference points to it.

### What Triggers Promotion

- The object has survived the maximum number of Minor GC cycles.
- The Survivor space is full, causing premature promotion.
- The object is too large to fit in the Survivor space.
- JVM ergonomics decisions based on observed allocation patterns.

### What Happens When Eden Fills

1. All application threads are paused (Stop-the-World).
2. Reachable objects in Eden are copied to the active Survivor space.
3. Eden is cleared.
4. The active and inactive Survivor spaces are swapped.
5. Application threads resume.

### TLAB (Thread Local Allocation Buffers)

Each thread is assigned a small private region of Eden space called a TLAB. Object allocation inside a TLAB requires no synchronization, making it very fast.

When a TLAB fills up, the thread requests a new one from Eden. If Eden itself is exhausted, a Minor GC is triggered. Very large objects bypass TLABs and are allocated directly.

### Humongous Objects in G1GC

Objects larger than 50% of a G1 heap region size are classified as Humongous Objects. They are allocated directly in the Old Generation rather than going through the Young Generation lifecycle. Excessive Humongous Object allocation can hurt performance. Adjusting `-XX:G1HeapRegionSize` can reduce the number of objects classified as humongous.

### Heap Memory Example

```java
// All of these objects start in Eden Space
String name = new String("John");
List<String> list = new ArrayList<>();
Person person = new Person("Alice", 25);

// After surviving multiple GC cycles, long-lived objects
// are promoted to Old Generation
```

## 3. Stack Memory

### Characteristics

| Aspect | Details |
|--------|---------|
| Scope | Thread-specific - each thread has its own stack |
| Structure | LIFO (Last In, First Out) |
| Contents | Method calls, local variables, object references |
| Management | Automatic - cleaned up when a method returns |
| Speed | Very fast allocation and deallocation |

### What Stack Stores

- Method call frames.
- Local primitive variables (`int`, `double`, `boolean`, etc.).
- Object references (the reference pointer, not the object itself).
- Method parameters.
- Return addresses.
- Intermediate computation results from the operand stack.

Objects themselves are never stored on the stack. Only references to heap objects are stored on the stack.

### Stack Frame

Each method invocation pushes a new Stack Frame onto the thread's stack. The frame contains:

- **Local Variable Array** - Stores method parameters and all local variables.
- **Operand Stack** - Used for intermediate calculations during bytecode execution.
- **Frame Data** - Method metadata, return address, and exception table references.

When a method returns, its frame is immediately popped and all local variables are cleared. Object references held in that frame are lost, potentially making the referenced heap objects eligible for GC.

### Stack Memory Visualization

```
Stack Memory (Thread 1):
┌─────────────────────────┐  <- Top of Stack
│  processData()          │
│  - param: 42            │
│  - result: 84           │
├─────────────────────────┤
│  demonstrateMemory()    │
│  - localPrimitive: 42   │
│  - localReference: ref  │ ──> Heap Objects
│  - localList: ref       │ ──>
├─────────────────────────┤
│  main()                 │
│  - args: ref            │ ──>
└─────────────────────────┘
```

### Stack and Heap Interaction

```java
public void example() {
    int primitive = 42;                    // Stack: stores value directly
    String reference = "Hello";            // Stack: stores reference -> Heap: String Pool
    List<String> list = new ArrayList<>(); // Stack: stores reference -> Heap: ArrayList object
}
```

### Stack Size and Performance

The stack size per thread is configured with `-Xss`. A larger value allows deeper call stacks but consumes more memory per thread.

For applications with many threads, stack size has a significant aggregate impact. For example, 1000 threads each with a 1 MB stack consumes 1 GB of memory for stacks alone.

- `-Xss512k` - suitable for lightweight applications with shallow call stacks.
- `-Xss2m` - appropriate for applications with deep recursion or many nested calls.

### StackOverflowError

Thrown when a thread's stack exceeds its configured size limit. The most common cause is infinite or unintended deep recursion.

Resolution: verify that all recursive methods have correct base cases, convert recursion to iteration where the depth is unbounded, or increase stack size with `-Xss`.

## 4. Method Area and Metaspace

### What Method Area Stores

The Method Area is a logical JVM concept implemented as Metaspace in Java 8 and later. It stores:

- Class metadata: structure, method signatures, field definitions.
- Method bytecode.
- Runtime Constant Pool: symbolic references and class-level constants.
- Static variables and static method references.

String literals are stored in the String Pool, which resides in the Heap (not Metaspace).

### PermGen vs Metaspace

| Aspect | PermGen (Java 7 and earlier) | Metaspace (Java 8+) |
|--------|------------------------------|---------------------|
| Location | Inside JVM heap | Native (OS) memory |
| Size | Fixed (`-XX:MaxPermSize`) | Dynamic growth |
| GC | Part of heap GC | Separate cleanup |
| OOM Risk | Common (`OutOfMemoryError: PermGen space`) | Rare unless capped |
| String Pool | Stored in PermGen | Moved to Heap |

### Key Improvements in Metaspace

- **Dynamic sizing** - Metaspace grows automatically as needed, unlike PermGen's fixed size.
- **Native memory** - Uses OS memory rather than heap space, reducing pressure on heap GC.
- **Better GC behavior** - Reduces the frequency of `OutOfMemoryError` caused by class metadata growth.
- **Explicit cap available** - Use `-XX:MaxMetaspaceSize` to set an upper limit and prevent native memory exhaustion.

### Metaspace Monitoring

```bash
-XX:+TraceClassLoading     # Log each class as it is loaded
-XX:+TraceClassUnloading   # Log each class as it is unloaded
jcmd <pid> VM.classloaders # Inspect classloader hierarchy at runtime
```

### Class Unloading

A class can be unloaded from Metaspace only when all three conditions are met:

1. No instances of the class exist anywhere on the heap.
2. The ClassLoader that loaded the class is itself eligible for GC.
3. The `Class` object is not referenced from anywhere.

Class unloading occurs during Full GC. When a class is unloaded, its Metaspace memory is returned to the OS (unlike PermGen, which returned it to the fixed-size pool).

### What Causes OutOfMemoryError: Metaspace

- Excessive dynamic class generation (frameworks, proxies, byte manipulation libraries).
- ClassLoader leaks in application servers during hot redeployment.
- Heavy use of reflection or runtime code generation.
- Frequent application redeployment without proper class unloading.

Resolution: increase `-XX:MaxMetaspaceSize`, fix ClassLoader leaks, optimize dynamic class generation, and monitor classloading activity.

## 5. Static vs Instance Memory

```java
public class MemoryExample {

    // Stored in Metaspace / Method Area as part of class-level data
    private static int staticVar = 100;

    // Reference stored in Heap; "Hello" stored in String Pool (Heap)
    private static final String CONSTANT = "Hello";

    // Stored in Heap inside the object instance
    private int instanceVar = 50;

    public static void staticMethod() {
        // Method bytecode stored in Metaspace
        // Execution uses a Stack frame
        int localVar = 10;  // Stored in Stack during execution
    }
}
```

### Behavior of Static Variables

- A single copy is shared across all instances and all threads.
- Lifecycle is tied to the ClassLoader: loaded when the class is first loaded, eligible for GC only when the class is unloaded.
- Static variables that are mutated by multiple threads require synchronization.

```java
public class Counter {
    private static volatile int count = 0;  // volatile for visibility across threads

    public static synchronized void increment() {
        count++;  // synchronized for atomicity
    }
}
```

### Static Initialization Order

```java
public class Example {
    private static int counter = 0;           // Initialized at class loading
    private static final String CONST = "Hi"; // Compile-time constant

    static {
        counter = 100;  // Static initializer block runs during class initialization
    }
}
```

## 6. Memory Optimization Techniques

### Reducing Object Allocation

Avoid creating unnecessary temporary objects, particularly in hot code paths:

```java
// Creates a new String on every iteration
String result = "";
for (String item : items) {
    result += item;
}

// Correct - reuses a single buffer
StringBuilder sb = new StringBuilder(items.size() * 16);
for (String item : items) {
    sb.append(item);
}
return sb.toString();
```

### Pre-Sizing Collections

When the expected size of a collection is known, initialize it with that capacity to avoid repeated internal array resizing and copying:

```java
// Avoids rehashing by specifying initial capacity
Map<String, String> map = new HashMap<>(expectedSize, 0.75f);
List<String> list = new ArrayList<>(expectedSize);
```

### Measuring Memory Usage

```java
Runtime runtime = Runtime.getRuntime();
long before = runtime.totalMemory() - runtime.freeMemory();

// Code under measurement
List<String> data = createLargeDataSet();

long after = runtime.totalMemory() - runtime.freeMemory();
System.out.println("Memory used: " + (after - before) + " bytes");
```

### String Pool Behavior

```java
String s1 = "Hello";              // Stored in String Pool
String s2 = "Hello";              // Reuses the same pool object
String s3 = new String("Hello");  // New object allocated in Heap, outside the pool
String s4 = s3.intern();          // Returns the pool reference

System.out.println(s1 == s2);     // true  - same pool object
System.out.println(s1 == s3);     // false - different objects
System.out.println(s1 == s4);     // true  - both refer to pool object
System.out.println(s1.equals(s3));// true  - same character content
```

### Batch Processing for Large Datasets

```java
int batchSize = 1000;
for (int i = 0; i < data.size(); i += batchSize) {
    processBatch(data.subList(i, Math.min(i + batchSize, data.size())));
    // Processing in batches limits the number of live objects at any one time,
    // reducing Old Generation pressure
}
```

### Additional Optimization Practices

- Use primitive types and primitive-specialized collections to eliminate autoboxing overhead.
- Apply lazy initialization to defer object creation until it is actually needed.
- Use object pooling for expensive-to-create resources such as connections and parsers.
- Use `StringBuilder` for string concatenation in loops; use `String.join()` for simple cases.
- Remove event listeners explicitly when they are no longer needed.
- Call `threadLocal.remove()` in a `finally` block to prevent leaks in thread pool environments.

## 7. JVM Security Model

The JVM was designed with security in mind, particularly for executing untrusted code such as applets downloaded from the network.

### Bytecode Verification

Before a class is executed, the JVM verifier checks that the bytecode is well-formed and adheres to JVM specification rules. This prevents:

- Illegal memory access.
- Stack underflow or overflow at the bytecode level.
- Type-unsafe operations.
- Invalid control flow.

### Class Loader Restrictions

The class loading subsystem enforces a delegation model (parent-first) that prevents malicious classes from replacing core Java classes. A class loader at a given level can only see classes loaded by itself or its parent, not its children.

This prevents untrusted code from substituting a fake `java.lang.String` or `java.lang.SecurityManager`.

### Security Manager

`java.lang.SecurityManager` (deprecated from Java 17, removed in Java 24) enforced runtime access control for:

- File system reads and writes.
- Network socket connections.
- System property access.
- Reflective access to private members.
- Class loading from arbitrary sources.

Applications could define custom security policies using `java.policy` files and activate them with `-Djava.security.manager -Djava.security.policy=app.policy`.

### Sandboxing

Sandboxing restricts untrusted code to a limited set of operations. For example, a sandboxed applet could read its own resources but not access the host file system or make arbitrary network connections.

Modern Java alternatives to SecurityManager for sandboxing include container isolation (Docker), OS-level sandboxing, and module system encapsulation introduced in Java 9.

### Access Control

`java.security.AccessController.checkPermission()` was used alongside SecurityManager to perform permission checks before sensitive operations. The Java module system (`module-info.java`) now provides compile-time and load-time encapsulation as a more robust alternative.

## 8. Security Risks in JVM Applications

### Deserialization Attacks

Java object deserialization via `ObjectInputStream` can execute arbitrary code if the stream comes from an untrusted source and the classpath contains gadget chains (exploitable classes). This is one of the most critical vulnerability classes in the Java ecosystem.

Mitigation: avoid deserializing untrusted data, use `ObjectInputFilter` to whitelist allowed classes, prefer safer serialization formats (JSON, Protocol Buffers).

### Reflection Misuse

The Reflection API allows code to bypass access modifiers and inspect or modify private fields and methods. Malicious or poorly written code can use reflection to:

- Access private credentials or keys.
- Modify immutable fields.
- Invoke private methods that bypass validation logic.

Mitigation: use the Java module system to restrict which packages can be accessed reflectively, avoid deep reflection on untrusted data.

### Unrestricted Class Loading

Loading untrusted bytecode without verification can compromise system integrity. Dynamic class generation frameworks (such as cglib, Javassist, and ASM) can be vectors if they process untrusted input.

Mitigation: validate all sources of dynamically loaded classes, use a restrictive ClassLoader, apply SecurityManager or module encapsulation.

### JNI Vulnerabilities

Native code called via JNI operates outside the JVM's safety guarantees. It can:

- Access arbitrary memory addresses.
- Bypass type safety and access control.
- Crash the JVM process directly.

Mitigation: minimize JNI surface area, audit native code with the same rigor as the JVM code, keep native libraries up to date.

## 9. JVM Security Best Practices

### Code-Level Practices

- Avoid using `ObjectInputStream` on untrusted data. Use `ObjectInputFilter` if deserialization is unavoidable.
- Restrict reflection to controlled and well-understood cases.
- Declare classes and fields `final` where mutation is not required to prevent subclassing attacks.
- Validate and sanitize all user-supplied inputs before processing.
- Do not log sensitive data (passwords, tokens, private keys).

### JVM Configuration Practices

- Use the latest supported JDK to benefit from security patches.
- Cap Metaspace with `-XX:MaxMetaspaceSize` to prevent native memory exhaustion.
- Restrict the classpath to trusted directories only.
- Enable GC logging and heap dump on OOM to enable post-incident analysis without re-creating issues.

### Deployment Practices

- Run the JVM process under a non-root OS user with the minimum required privileges.
- Containerize JVM applications (Docker, Kubernetes) to provide OS-level process isolation.
- Sign JAR files with `jarsigner` and verify signatures before deployment.
- Enforce TLS for all network communication. Configure cipher suites to exclude weak algorithms.
- Use HTTPS with certificate pinning for sensitive service-to-service calls.

### ClassLoader and Metaspace Hygiene

- In application servers, ensure that web application ClassLoaders and all their referenced objects are fully dereferenced on undeploy to prevent ClassLoader leaks.
- Monitor class loading counts in long-running processes. Steadily increasing counts after stabilization indicate a leak.
- Avoid frameworks that generate unbounded numbers of anonymous classes at runtime.

## 10. Tools for Monitoring and Security

### Memory and Performance Monitoring

| Tool | Purpose |
|------|---------|
| JConsole | GUI-based JMX monitoring of heap, threads, and MBeans |
| JVisualVM | Visual profiler with heap dump capture and CPU profiling |
| Java Flight Recorder (JFR) | Low-overhead continuous profiling built into JDK 11+ |
| jstat | Command-line GC and heap statistics sampled at intervals |
| jmap | Generate heap histograms and heap dump files |
| Eclipse MAT | Heap dump analysis - leak suspects, dominator trees |
| GCViewer | GC log visualization and pause time analysis |

```bash
jstat -gc -t <pid> 250ms           # GC stats sampled every 250 ms
jmap -histo <pid>                   # Object type histogram
jmap -dump:file=heap.hprof <pid>   # Full heap dump
jcmd <pid> JFR.start               # Start a Flight Recorder session
```

### Security-Focused Tools

- **JAR signing (`jarsigner`)** - Signs JAR files and verifies integrity before deployment.
- **Security auditing tools** - Static analysis tools (SpotBugs with FindSecBugs plugin, Checkmarx) can identify deserialization risks, reflection misuse, and injection vulnerabilities.
- **Dependency scanning** - Tools such as OWASP Dependency-Check or Snyk scan classpath dependencies for known CVEs.
- **Java Flight Recorder** - Can be used to detect anomalous classloading patterns at runtime.

### Key JVM Flags for Diagnostics

```bash
-XX:+HeapDumpOnOutOfMemoryError          # Capture heap dump on OOM
-XX:HeapDumpPath=/path/to/dumps/         # Specify dump location
-XX:+TraceClassLoading                   # Log every class load
-XX:+TraceClassUnloading                 # Log every class unload
-Xlog:gc*                                # Detailed GC logging (Java 9+)
-XX:+PrintGCDetails -XX:+PrintGCTimeStamps  # GC logging (pre-Java 9)
```
