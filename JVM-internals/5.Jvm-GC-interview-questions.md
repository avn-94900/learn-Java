## 1. Garbage Collection Fundamentals

**Level: Beginner**

---

**Q: What is Garbage Collection and why is it needed?**

Garbage Collection is the JVM's automatic memory management system. It identifies objects that are no longer reachable from any live thread, reclaims the memory they occupy, and prevents memory leaks.

Java does not have explicit memory deallocation operators such as `free()` or `delete`. Without GC, developers would need to manage heap memory manually, which is error-prone and a common source of bugs in languages like C and C++. GC removes this burden while also preventing dangling pointer issues.

---

**Q: What makes an object eligible for Garbage Collection?**

An object becomes eligible for GC when no live, reachable reference points to it. Specific conditions include:

- A reference is explicitly set to `null`.
- A reference goes out of scope when a method returns and the local variable is cleared.
- Circular references exist among a group of objects with no external references pointing into the group.

```java
// Eligible for GC after this line
String str = new String("Hello");
str = null;

// Not eligible - reachable via a static variable
static List<String> list = new ArrayList<>();
list.add("Keep me");
```

---

**Q: What are the different types of Garbage Collection events?**

There are three categories of GC events in the JVM:

- **Minor GC** collects only the Young Generation. It occurs frequently (every few seconds in active applications) and completes in milliseconds. Most objects die here.
- **Major GC** collects the Old Generation. It is less frequent and takes longer because the Old Generation contains more data and longer-lived objects.
- **Full GC** collects the entire heap including Metaspace. It is the most disruptive, can take seconds, and typically occurs under memory pressure or when explicit compaction is required.

## 2. GC Process and Algorithms

**Level: Intermediate**

---

**Q: Explain the Mark and Sweep algorithm.**

Mark and Sweep is the foundational algorithm underlying most JVM GC implementations. It operates in two phases:

**Mark Phase:**

The JVM starts from all GC roots (local variables in active stack frames, static variables, JNI references, and active thread objects) and traverses the entire object graph recursively. Every reachable object is marked as live.

**Sweep Phase:**

The JVM scans the heap and deallocates memory for every object that was not marked as live. It then resets marks in preparation for the next cycle.

**Optional Compact Phase:**

Some collectors add a compaction step that moves all live objects to contiguous memory, eliminating fragmentation. This requires updating all references to moved objects.

---

**Q: What are GC roots and why are they important?**

GC roots are the anchor points of the object reachability graph. An object is considered live if and only if it is reachable by following a chain of references starting from a GC root. Any object not reachable from a GC root is garbage and eligible for collection.

Types of GC roots in the JVM:

- Local variables on the stack of any active thread.
- Static fields in loaded classes (stored in the Method Area).
- References held by the JNI (Java Native Interface).
- Active thread objects themselves.
- Objects held by the system or bootstrap class loader.

The correctness of GC depends entirely on accurate identification of GC roots.

---

**Q: How does Generational Garbage Collection work?**

Generational GC is based on the empirical observation that the vast majority of objects die shortly after allocation (the weak generational hypothesis).

The heap is divided into generations:

- The **Young Generation** holds newly created objects. It is collected frequently using Minor GC, which is fast because most objects in it are already dead.
- The **Old Generation** holds objects that have survived a configured number of Minor GC cycles. It is collected less frequently during Major GC.

Objects are promoted from Young to Old Generation based on their survival count (tenuring threshold). Write barriers track references from Old to Young so that Minor GC can correctly identify all live young objects without scanning the entire heap.

This design lets the JVM spend most of its GC effort on the smallest, most productive area of the heap.

## 3. Advanced GC Concepts

**Level: Advanced**

---

**Q: What is the difference between concurrent and parallel garbage collection?**

These terms are often confused but describe fundamentally different properties:

**Parallel GC** refers to using multiple GC threads simultaneously to complete a collection faster. The application threads are still paused (Stop-the-World) for the entire duration. Parallel GC improves GC throughput but does not improve application pause time.

**Concurrent GC** refers to GC threads running alongside application threads with only minimal, brief Stop-the-World pauses. The GC work is interleaved with application execution. Concurrent GC reduces perceived pause times at the cost of higher CPU utilization.

The trade-off is throughput versus latency. Parallel GC maximizes the amount of work done per unit of time. Concurrent GC minimizes the time the application is paused.

Examples: ParallelGC is a parallel collector. G1GC, CMS, and ZGC are concurrent collectors.

---

**Q: Explain the concept of GC ergonomics.**

GC ergonomics is the JVM's ability to automatically tune GC parameters at runtime to meet configured performance goals, without requiring the developer to manually adjust every parameter.

The JVM adapts:

- Young and Old Generation size ratios.
- GC trigger thresholds.
- Survivor space sizing and tenuring thresholds.

Goals can be configured:

```bash
-XX:MaxGCPauseMillis=200    # Target maximum pause time
-XX:GCTimeRatio=99          # Target 1% of time spent in GC
```

The JVM monitors actual GC behavior and incrementally adjusts parameters to move closer to the configured goals. This is particularly useful in G1GC, which uses ergonomics to select which heap regions to collect each cycle.

Ergonomics removes the need to manually tune many parameters but should be monitored in production to verify it is achieving the intended results.

## 4. GC Algorithms - Serial and Parallel

**Level: Beginner**

---

**Q: When would you use Serial GC?**

Serial GC should be used when simplicity and low memory overhead are more important than performance:

- Small applications with heaps under 100 MB.
- Single-core machines where multiple GC threads provide no benefit.
- Client-side applications with modest memory requirements.
- Embedded systems with constrained resources.
- Development and testing environments.

Serial GC is a poor choice for any multi-core server application serving concurrent users.

```bash
-XX:+UseSerialGC
```

---

**Q: What is the difference between Parallel GC and Parallel Old GC?**

Both use multiple threads for GC work, but they differ in which generation they parallelize:

- **Parallel GC** uses multiple threads for Minor GC (Young Generation collection) but falls back to a single thread for Major GC (Old Generation collection).
- **Parallel Old GC** uses multiple threads for both Minor GC and Major GC. It provides better performance for applications where Old Generation collection is a bottleneck.

Parallel Old GC was the default collector before Java 9. Both share the same throughput-oriented design with Stop-the-World pauses for all collections.

```bash
-XX:+UseParallelGC
-XX:+UseParallelOldGC
```

## 5. GC Algorithms - Concurrent Mark-Sweep

**Level: Intermediate**

---

**Q: How does CMS GC work and what are its phases?**

CMS (Concurrent Mark-Sweep) was designed to reduce Old Generation pause times by performing most GC work concurrently with the application. It has been deprecated since Java 9.

The six phases of CMS:

1. **Initial Mark (STW):** A brief Stop-the-World pause to mark objects directly reachable from GC roots.
2. **Concurrent Mark:** Traverses the entire object graph concurrently. Application threads continue running.
3. **Concurrent Preclean:** Handles objects whose references were modified during the Concurrent Mark phase.
4. **Remark (STW):** A second brief Stop-the-World pause to finalize marking and account for changes during the concurrent phases.
5. **Concurrent Sweep:** Frees memory of unreachable objects concurrently with the application.
6. **Concurrent Reset:** Resets internal CMS data structures to prepare for the next cycle.

The two STW pauses (Initial Mark and Remark) are short. The bulk of the work is done concurrently.

---

**Q: What problems can occur with CMS GC?**

CMS has several well-known issues that contributed to its deprecation:

- **Heap Fragmentation:** CMS does not compact the heap after sweeping. Over time, free memory becomes fragmented and large objects cannot be allocated even when sufficient total free memory exists.
- **Concurrent Mode Failure:** If the allocation rate exceeds the rate at which CMS can collect, it falls back to a single-threaded Full GC with a long STW pause to recover.
- **Promotion Failure:** Fragmentation can prevent Young Generation objects from being promoted to the Old Generation even when the Old Generation has free space in aggregate.
- **CPU Overhead:** Background CMS threads consume CPU continuously, reducing application throughput.

Mitigation for legacy systems: `-XX:CMSInitiatingOccupancyFraction=70` triggers CMS earlier to reduce the risk of Concurrent Mode Failure. The correct long-term solution is to migrate to G1GC.

## 6. GC Algorithms - G1 Garbage Collector

**Level: Intermediate**

---

**Q: How does G1GC differ from traditional generational collectors?**

Traditional collectors like Parallel GC use fixed contiguous memory regions for the Young and Old generations. Young objects always go in one fixed area, and promotion moves them to another.

G1GC takes a fundamentally different approach. The heap is divided into a configurable number of equal-sized regions (typically 1 MB to 32 MB each). These regions are not fixed to a particular generation. Instead, the JVM dynamically assigns regions as Eden, Survivor, Old, or Humongous (for objects larger than 50% of a region size).

Key benefits of this design:

- Pause times are predictable because G1 collects only a subset of regions per cycle, choosing those with the most garbage.
- Large heaps are handled more efficiently because collection work is incremental.
- Both throughput and latency goals can be balanced through the `-XX:MaxGCPauseMillis` target.

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
```

---

**Q: What is the G1GC evacuation process?**

Evacuation is the process by which G1GC moves live objects out of selected regions to reclaim their memory.

The process:

1. G1 selects a collection set of regions whose collection will fit within the configured pause time target, prioritizing regions with the highest ratio of garbage.
2. It scans GC roots and remembered sets to identify all references into the collection set.
3. Live objects in the collection set are copied (evacuated) into new, empty regions.
4. All references pointing to evacuated objects are updated to their new addresses.
5. The original regions are now entirely free and returned to the pool.

The pause time target (`-XX:MaxGCPauseMillis`) directly controls how many regions are collected per evacuation cycle. If the target is too aggressive, G1 will collect fewer regions per cycle, which can allow the Old Generation to grow until a Full GC is forced.

## 7. GC Algorithms - Low-Latency Collectors

**Level: Advanced**

---

**Q: What are the key features of ZGC?**

ZGC is designed to achieve sub-10-millisecond pause times regardless of heap size, including heaps up to 16 TB.

Key technical features:

- **Colored pointers:** Object state information (marked, remapped, finalizable) is encoded directly in the object reference bits. This allows the GC to determine an object's GC state without following the pointer.
- **Load barriers:** Every time a thread reads a heap reference, a small piece of code checks the object's state. If the object has been relocated, the barrier transparently updates the reference before returning it to the application.
- **Concurrent operations:** Almost all GC work - marking, relocation, and reference remapping - is done concurrently with application threads.

Trade-offs to be aware of:

- Memory overhead of approximately 2-16% for colored pointer metadata.
- CPU overhead from load barriers on every heap read.
- Requires Java 11 or later.

```bash
-XX:+UseZGC
```

---

**Q: When should you consider using Shenandoah GC?**

Shenandoah GC targets the same problem as ZGC: consistent low pause times for large heaps. The key differentiator is that Shenandoah performs concurrent evacuation - it moves live objects to new locations while application threads continue running, using read and write barriers to keep both the old and new copies consistent.

Consider Shenandoah when:

- The application has strict latency requirements and pause times above 10 ms cause user-visible problems.
- The heap is large (multiple GB) and traditional collectors produce unacceptable pause durations.
- You are running OpenJDK and want an alternative to ZGC with a different implementation and sometimes different behavior on specific workloads.
- Pause time consistency matters more than raw application throughput.

```bash
-XX:+UseShenandoahGC
```

## 8. JVM Parameters

**Level: Beginner**

---

**Q: What are the most important JVM memory tuning parameters?**

**Heap sizing:**

```bash
-Xms2g                     # Initial heap size
-Xmx8g                     # Maximum heap size
-XX:NewRatio=3             # Old:Young generation ratio
```

**Stack sizing:**

```bash
-Xss1m                     # Stack size per thread
```

**Metaspace:**

```bash
-XX:MetaspaceSize=256m     # Initial metaspace size
-XX:MaxMetaspaceSize=512m  # Maximum metaspace size
```

**GC selection and tuning:**

```bash
-XX:+UseG1GC               # Select G1 collector
-XX:MaxGCPauseMillis=200   # Target maximum pause time
```

In production, set `-Xms` equal to `-Xmx` to avoid heap resizing overhead and make memory behavior predictable.

---

**Q: How do you calculate appropriate heap size for an application?**

There is no single formula, but the following factors inform the decision:

- **Available system memory:** Reserve memory for the OS, other processes, JVM overhead, and native memory (Metaspace, direct buffers, thread stacks).
- **Live data size:** Profile the application to determine the size of the live object set at peak load. Old Generation should be at least 1.5 to 2 times the live data size.
- **GC overhead trade-off:** A larger heap reduces GC frequency but increases pause duration per collection.
- **Starting point:** Allocate 25-50% of available RAM to the JVM heap.

Example for a 16 GB server:

```
OS and other processes: approximately 4 GB
JVM heap: -Xms6g -Xmx8g
Remaining native memory: available for Metaspace, thread stacks, direct buffers
```

Monitor GC logs after deployment and adjust based on observed behavior.

## 9. Performance Monitoring

**Level: Intermediate**

---

**Q: What tools can you use to monitor Java memory usage?**

**JDK built-in tools:**

- **JConsole** - GUI tool for JMX-based monitoring of memory, threads, and MBeans.
- **JVisualVM** - Visual profiler with heap dump capture, CPU and memory profiling.
- **Java Flight Recorder (JFR)** - Low-overhead continuous profiling built into the JDK from Java 11. Use with `jcmd <pid> JFR.start`.
- **jstat** - Command-line tool for real-time GC statistics.

```bash
jstat -gc -t <pid> 250ms    # GC statistics sampled every 250 ms
jmap -histo <pid>            # Object histogram by type
jmap -dump:file=heap.hprof <pid>  # Heap dump
```

**Third-party and analysis tools:**

- **Eclipse MAT** - Heap dump analysis; identifies retained memory, leak suspects, and dominator trees.
- **GCViewer** - GC log visualization and analysis.
- **JProfiler** - Commercial profiler with detailed call tree and memory analysis.
- **AppDynamics / New Relic** - APM platforms for production monitoring.

---

**Q: How do you analyze GC logs for performance issues?**

The process for GC log analysis:

1. Enable detailed GC logging: `-Xlog:gc*` (Java 9+) or `-XX:+PrintGCDetails -XX:+PrintGCTimeStamps` (pre-Java 9).
2. Load the log file into GCViewer or a similar tool.
3. Review the following key metrics:
   - GC frequency and regularity.
   - Pause duration for Minor, Major, and Full GC events.
   - Application throughput (percentage of time not in GC).
   - Heap usage before and after each collection.
4. Identify the pattern of Old Generation growth over time.

Warning signs that require action:

- Full GC events occurring more than once per hour in a stable workload.
- Pause times consistently exceeding the SLA threshold.
- Old Generation that grows steadily and never shrinks - a strong indicator of a memory leak.
- GC overhead errors indicating the JVM is spending more time collecting than running the application.

## 10. Memory Optimization

**Level: Advanced**

---

**Q: What are the best practices for minimizing GC pressure?**

GC pressure refers to the rate at which the application creates short-lived objects, which increases Minor GC frequency.

**Reduce unnecessary object creation:**

```java
// Creates a new String on every concatenation
String result = "";
for (String item : items) {
    result += item;
}

// Correct - reuses a single buffer
StringBuilder sb = new StringBuilder(items.size() * 16);
for (String item : items) {
    sb.append(item);
}
return sb.toString();
```

**Additional techniques:**

- Use object pooling for expensive-to-create, frequently reused objects (connection pools, thread pools).
- Use primitive types and primitive collections instead of boxed types to eliminate autoboxing.
- Pre-size collections with an expected capacity.
- Apply lazy initialization to defer object creation until it is actually needed.
- Avoid allocating objects in hot loops.
- Prefer immutable objects where feasible - they tend to be shorter-lived.

---

**Q: How do you handle memory leaks in Java applications?**

**Common leak patterns:**

- Static collections that grow without a removal strategy.
- Event listeners registered but never removed.
- `ThreadLocal` values not cleared with `remove()` in thread pool environments.
- Non-static inner classes holding implicit references to outer class instances.
- ClassLoader leaks in application servers during redeployment.

**Detection approach:**

1. Enable GC logging and observe whether Old Generation usage grows continuously without stabilizing.
2. Generate heap dumps at intervals using `jmap -dump` or wait for `-XX:+HeapDumpOnOutOfMemoryError`.
3. Open the heap dump in Eclipse MAT.
4. Run the Leak Suspects report and examine the dominator tree.
5. Identify the root object retaining the most memory and trace it back to application code.

**Prevention patterns:**

```java
// Bounded cache with automatic eviction of eldest entry
private static final Map<String, Object> cache =
    new LinkedHashMap<String, Object>(16, 0.75f, true) {
        protected boolean removeEldestEntry(Map.Entry<String, Object> eldest) {
            return size() > 10000;
        }
    };

// ThreadLocal always cleaned up in finally
private static ThreadLocal<Object> threadLocal = new ThreadLocal<>();

public void process() {
    threadLocal.set(computeValue());
    try {
        // Use the value
    } finally {
        threadLocal.remove();
    }
}
```

## 11. Troubleshooting and Errors

**Level: Beginner**

---

**Q: What are the different types of OutOfMemoryError?**

| Error Type | Root Cause | Resolution |
|-----------|-----------|------------|
| `Java heap space` | The heap is exhausted and GC cannot free enough memory | Increase `-Xmx`; investigate memory leaks via heap dump |
| `Metaspace` | Too many classes are loaded into native memory | Increase `-XX:MaxMetaspaceSize`; check for classloader leaks |
| `GC overhead limit exceeded` | Over 98% of elapsed time is spent in GC while freeing under 2% of the heap | Increase heap size; optimize allocation patterns |
| `Direct buffer memory` | NIO direct buffer allocation has exceeded its limit | Increase `-XX:MaxDirectMemorySize`; audit NIO buffer usage |

Each variant points to a different area of memory and requires a different investigation path.

---

**Q: What causes StackOverflowError and how do you debug it?**

StackOverflowError is thrown when the JVM stack for a thread exceeds its configured size limit. Each method invocation pushes a new stack frame, and deep or infinite recursion causes the stack to overflow.

**Common causes:**

- A recursive method missing a base case, causing infinite recursion.
- Mutually recursive methods that call each other without a termination condition.
- Very deep call stacks in legitimate but deeply nested code.

**Debugging steps:**

1. Examine the exception stack trace for repeated method names, which indicate infinite or unintended recursion.
2. Temporarily increase the stack size with `-Xss2m` to determine whether the issue is depth or infinite recursion. If the error disappears, the recursion is deep but finite. If it persists, the recursion is infinite.
3. Add a base case or refactor the recursive logic to an iterative implementation using an explicit stack data structure.

```java
// Missing base case - will always throw StackOverflowError
public int factorial(int n) {
    return n * factorial(n - 1);
}

// Corrected
public int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
```
