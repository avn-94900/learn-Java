## 1. JVM Memory Architecture

**Level: Beginner**

---

**Q1.1: What are the main memory areas in JVM?**

The JVM divides process memory into five main areas:

- **Heap Memory** - Shared across all threads. Stores all objects, arrays, and instance variables.
- **Stack Memory** - One stack per thread. Stores method call frames, local variables, and object references.
- **Method Area / Metaspace** - Shared. Stores class metadata, static variables, and method bytecode.
- **PC Registers** - One per thread. Tracks the address of the current instruction being executed.
- **Native Method Stacks** - Supports JNI calls to native (non-Java) code.

---

**Q1.2: What is the difference between Heap and Stack memory?**

| Aspect | Heap | Stack |
|--------|------|-------|
| Scope | Shared across all threads | One per thread, not shared |
| Contents | Objects, arrays, instance variables | Method frames, local variables, references |
| Speed | Slower - managed by the GC | Very fast - LIFO allocation and deallocation |
| Size | Large and configurable (`-Xmx`) | Smaller, configured per thread (`-Xss`) |
| Lifetime | Until GC collects unreachable objects | Until the enclosing method returns |

The Stack holds references that point to objects stored in the Heap.

---

**Q1.3: Can you store objects in Stack memory?**

No. The Stack can only store:

- Primitive values (`int`, `double`, `boolean`, etc.) directly.
- Object references, which are pointers to heap addresses.
- Method call metadata such as return addresses.

Objects themselves are always allocated on the Heap. When you write `new SomeClass()`, the object lives in the Heap. The variable holding the reference may live on the Stack.

---

**Level: Intermediate**

---

**Q1.4: How do Stack and Heap interact with each other?**

The Stack holds references and the Heap holds the objects those references point to. When a method executes and declares an object variable, the reference is stored in the Stack frame. The object it points to is stored in the Heap.

```java
public void example() {
    int primitive = 42;                    // Value stored directly in Stack
    String reference = "Hello";            // Reference in Stack -> String Pool in Heap
    List<String> list = new ArrayList<>(); // Reference in Stack -> ArrayList object in Heap
}
```

When the method returns, the Stack frame is destroyed and all references in it are lost. The objects they pointed to remain in the Heap until the GC collects them.

---

**Q1.5: What happens to Stack and Heap when a method completes execution?**

- **Stack** - The method's Stack Frame is immediately popped (LIFO). All local variables and references stored in that frame are instantly cleared.
- **Heap** - Objects in the Heap are not immediately affected. They remain allocated.
- **GC eligibility** - If the destroyed Stack frame held the only reference to a Heap object, that object is now unreachable and becomes eligible for garbage collection. The GC will reclaim it at the next appropriate collection cycle.

---

**Level: Advanced**

---

**Q1.6: Explain the memory layout of a Java process in detail.**

A Java process occupies several distinct memory regions at the OS level:

```
┌─────────────────┐
│   Heap Memory   │  All Java objects and arrays. Size controlled by -Xms / -Xmx.
├─────────────────┤
│   Metaspace     │  Class metadata in native (OS) memory. Grows dynamically.
├─────────────────┤
│ Thread Stacks   │  One stack per thread. Size controlled by -Xss.
├─────────────────┤
│ Direct Memory   │  NIO ByteBuffers and off-heap allocations via Unsafe.
├─────────────────┤
│ Code Cache      │  JIT-compiled native code. Controlled by -XX:ReservedCodeCacheSize.
└─────────────────┘
```

The total memory consumption of a JVM process is the sum of all these regions, not just the heap. This is why a Java process often consumes significantly more OS memory than the configured `-Xmx` value.

---

**Q1.7: How does JVM decide memory allocation between different areas?**

Each area is sized by a combination of explicit configuration and JVM ergonomics:

- **Heap** - Sized by `-Xms` (initial) and `-Xmx` (maximum). JVM ergonomics auto-selects defaults based on available system memory if not set explicitly.
- **Stack** - Each thread gets the size specified by `-Xss`. The default varies by OS and JVM implementation, typically between 256 KB and 1 MB.
- **Metaspace** - Starts small and grows dynamically using native memory. Unbounded unless capped with `-XX:MaxMetaspaceSize`.
- **Direct Memory** - Limited by available system memory. Bounded with `-XX:MaxDirectMemorySize`.
- **Code Cache** - Pre-allocated at JVM startup. Resizable with `-XX:ReservedCodeCacheSize`.

## 2. Heap Memory

**Level: Beginner**

---

**Q2.1: What are the different generations in Heap memory?**

The heap is divided into two main generations:

**Young Generation** - Holds newly created objects. Divided into:
- Eden Space: where all new objects are first allocated.
- Survivor Space S0 and Survivor Space S1: hold objects that survived at least one Minor GC.

**Old Generation (Tenured)** - Holds long-lived objects that have been promoted from the Young Generation after surviving multiple GC cycles. Larger than the Young Generation; the default ratio is approximately 1:2.

---

**Q2.2: Where are new objects created in Heap?**

All new objects start in Eden Space within the Young Generation. The only exception is very large objects, which may bypass Eden and be allocated directly in the Old Generation (or in Humongous regions in G1GC) if they exceed a size threshold.

---

**Q2.3: What is the purpose of Survivor spaces S0 and S1?**

Survivor spaces act as a staging area between Eden and the Old Generation. Their purpose is to give objects a chance to die before being promoted to the more expensive Old Generation.

- Only one Survivor space is active at any given time; the other is empty.
- When a Minor GC runs, live objects from Eden are copied into the active Survivor space.
- In the next Minor GC, live objects from both Eden and the currently active Survivor space are copied to the other Survivor space. The previously active space becomes empty.
- This alternating copy process continues until an object reaches the tenuring threshold (default 15), at which point it is promoted to the Old Generation.

The parameter `-XX:MaxTenuringThreshold` controls the promotion threshold.

---

**Level: Intermediate**

---

**Q2.4: Explain the lifecycle of an object from creation to garbage collection.**

```
New Object -> Eden Space -> Minor GC -> Survivor Space (S0 or S1)
-> Multiple Minor GC Cycles -> Old Generation -> Major GC -> Collected
```

1. **Creation** - Object is allocated in Eden Space via TLAB or directly.
2. **First Minor GC** - If the object is reachable, it is copied to the active Survivor space. Its age counter is incremented to 1.
3. **Subsequent Minor GCs** - The object continues to alternate between S0 and S1. Its age increments with each survival.
4. **Promotion** - When the object's age reaches the tenuring threshold, it is copied to the Old Generation.
5. **Major or Full GC** - Eventually, when no live reference points to the object, it is collected during a Major or Full GC.

---

**Q2.5: When does an object get promoted from Young to Old Generation?**

An object is promoted when any of the following conditions is true:

- Its age (number of Minor GC cycles survived) reaches the maximum tenuring threshold. Default is 15, configurable with `-XX:MaxTenuringThreshold=15`.
- The Survivor space is too full to hold all surviving objects; excess objects are promoted immediately (premature promotion).
- The object is too large to fit in a Survivor space.
- JVM ergonomics determines that the dynamic tenuring threshold should be lowered based on Survivor space occupancy.

Premature promotion is a performance concern because it increases Old Generation pressure and can lead to more frequent Major GC events.

---

**Q2.6: What happens if Eden space fills up?**

A Minor GC is triggered. The sequence of events is:

1. All application threads are paused (Stop-the-World).
2. The GC marks all reachable objects in Eden and the active Survivor space.
3. Live objects are copied to the inactive Survivor space. Objects that exceed the tenuring threshold are promoted to the Old Generation.
4. Eden is completely cleared.
5. The roles of S0 and S1 are swapped (the previously inactive space becomes active).
6. Application threads resume.

Minor GC is typically fast (a few milliseconds) because most objects in Eden are already dead at collection time.

---

**Level: Advanced**

---

**Q2.7: How does heap memory allocation work internally?**

The JVM uses Thread Local Allocation Buffers (TLABs) to make object allocation extremely fast.

When a thread is created, it is assigned a small private region of Eden space as its TLAB. Object allocation within a TLAB requires no synchronization because no other thread writes to it. The allocation is simply a pointer bump: advance the pointer by the object size.

When a TLAB fills up, the thread requests a new one from Eden. If Eden cannot satisfy the request, a Minor GC is triggered to reclaim space.

Very large objects bypass TLABs and are allocated directly in Eden (or in Humongous regions in G1GC). This allocation path requires synchronization.

The fast path for object allocation is therefore: check TLAB space, bump pointer, return reference - all in a handful of instructions.

---

**Q2.8: What are Humongous Objects in the G1GC context?**

In G1GC, the heap is divided into equal-sized regions. A Humongous Object is any object that is larger than 50% of a single region size.

Humongous Objects are handled specially:

- They are allocated directly in a sequence of contiguous Old Generation regions rather than going through the Young Generation.
- They bypass TLAB allocation and require global heap lock.
- They are collected during the G1GC cleanup phase, not during regular Young GC cycles.
- Excessive Humongous Object allocation can fragment the heap and trigger more frequent Full GCs.

If Humongous allocations are a bottleneck, increasing `-XX:G1HeapRegionSize` (valid values: 1 MB to 32 MB, must be a power of 2) raises the threshold for what qualifies as humongous.

## 3. Stack Memory

**Level: Beginner**

---

**Q3.1: What is stored in Stack memory?**

Each thread's Stack holds:

- Method call frames, one per active method call.
- Local primitive variables (`int`, `long`, `double`, `boolean`, etc.) stored by value.
- Object references (the pointer to a heap object, not the object itself).
- Method parameters passed by the caller.
- Return addresses indicating where execution resumes after the method returns.
- The operand stack used for intermediate bytecode calculations.

---

**Q3.2: Is Stack memory shared between threads?**

No. Each thread has its own private Stack. This is what makes local variables and method arguments inherently thread-safe: no other thread can see or modify them. Only Heap memory (objects and static variables) is shared between threads and therefore requires synchronization when accessed concurrently.

---

**Q3.3: What is a Stack Frame?**

A Stack Frame is the unit of memory pushed onto a thread's Stack when a method is called and popped when it returns. Each frame contains:

- **Local Variable Array** - Holds all method parameters and local variables. Primitives are stored by value; objects are stored as references.
- **Operand Stack** - A temporary working area used by JVM bytecode instructions for arithmetic and method arguments.
- **Frame Data** - A reference to the method's runtime constant pool, the return address, and the exception handler table.

---

**Level: Intermediate**

---

**Q3.4: How does method invocation work with Stack?**

Each method call pushes a new Stack Frame on top of the calling method's frame. When the called method returns, its frame is popped and control returns to the caller's frame at the stored return address.

```java
public void methodA() {     // Frame A pushed; localVar x = 10 stored in Frame A
    int x = 10;
    methodB();              // Frame B pushed on top of Frame A
}                           // Frame A popped; x is gone

public void methodB() {     // Frame B; localVar y = 20 stored in Frame B
    int y = 20;
}                           // Frame B popped; y is gone
```

The LIFO structure means the most recently called method is always the first to complete, which matches the natural nesting of method calls.

---

**Q3.5: What happens to local variables when a method exits?**

The method's Stack Frame is immediately popped. This destroys all local variables and references stored in that frame instantly - no GC involvement.

If a reference in the frame was the only live reference to a heap object, that object is now unreachable and becomes eligible for garbage collection. The GC will reclaim its memory at the next appropriate cycle. The heap object itself is not immediately freed; only the reference is destroyed.

---

**Level: Advanced**

---

**Q3.6: What causes StackOverflowError and how do you fix it?**

`StackOverflowError` is thrown when a thread's Stack exceeds its configured size. The Stack grows with each method call and shrinks with each return. If calls accumulate faster than they return (deep or infinite recursion), the Stack eventually runs out of space.

Common causes:

- A recursive method with a missing or unreachable base case, causing infinite recursion.
- Mutually recursive methods that call each other without termination.
- Legitimately deep recursion on large inputs where the configured stack size is insufficient.

Resolution steps:

1. Examine the stack trace for repeated method names. Repetition indicates infinite or unintended recursion.
2. Add or correct the base case in the recursive method.
3. If the recursion is legitimately deep but finite, increase the stack size with `-Xss2m` and verify the issue is resolved.
4. Convert the recursion to an iterative implementation using an explicit stack data structure to eliminate the stack depth constraint entirely.

---

**Q3.7: How does Stack size affect application performance?**

Stack size has a direct impact on memory consumption in applications with many threads.

- A larger stack (`-Xss2m`) allows deeper call stacks but consumes more memory per thread.
- A smaller stack (`-Xss256k`) reduces memory per thread, allowing more concurrent threads on the same machine, but limits call depth.

The aggregate impact is significant: 1000 threads with a 1 MB stack each consume 1 GB of memory exclusively for stacks, before any heap allocation is considered.

For REST APIs and web servers using thread-per-request models, stack size is a meaningful tuning parameter. For applications using virtual threads (Project Loom, Java 21+), stack memory consumption per concurrent task is dramatically reduced.

Guidance: profile the actual maximum call depth of the application and set `-Xss` to the minimum that accommodates it with a safety margin.

## 4. Method Area and Metaspace

**Level: Beginner**

---

**Q4.1: What is Method Area and what does it store?**

Method Area is a logical JVM concept representing the storage for class-level data. In Java 8 and later it is implemented as Metaspace. It stores:

- Class metadata: class name, superclass, interfaces, access modifiers.
- Field and method signatures and access modifiers.
- Method bytecode.
- Runtime Constant Pool: symbolic references and compile-time constants.
- Static variable values (the variables themselves, referenced from the class data).

String literals are not stored in Metaspace. They are stored in the String Pool, which resides in the Heap.

---

**Q4.2: What is the difference between PermGen and Metaspace?**

| Aspect | PermGen (Java 7 and earlier) | Metaspace (Java 8+) |
|--------|------------------------------|---------------------|
| Location | Inside JVM heap | Native (OS) memory |
| Size | Fixed, set with `-XX:MaxPermSize` | Dynamic, grows as needed |
| Default limit | 64 MB or 128 MB | Bounded only by system memory unless capped |
| OOM risk | High in dynamic environments | Low unless `-XX:MaxMetaspaceSize` is set too small |
| String Pool | Stored in PermGen | Moved to Heap |
| GC | Collected as part of heap GC | Separate cleanup during Full GC |

The move to Metaspace eliminated the fixed-size constraint that caused frequent `OutOfMemoryError: PermGen space` errors in applications using frameworks with dynamic class generation.

---

**Level: Intermediate**

---

**Q4.3: Where are static variables stored and when are they initialized?**

Static variables are stored in the Method Area (Metaspace) as part of the class data. They are initialized when the class is first loaded by the ClassLoader.

Initialization order within a class:

1. Static fields are assigned their default values (`0`, `null`, `false`, etc.).
2. Static field initializers and static blocks are executed in the order they appear in the source file.

```java
public class Example {
    private static int counter = 0;           // Initialized to 0, then set to 0 (explicit)
    private static final String CONST = "Hi"; // Compile-time constant; inlined by compiler

    static {
        counter = 100;  // Executed during class initialization; overrides the field initializer
    }
}
```

---

**Q4.4: How do static variables behave with respect to memory and threads?**

A static variable has exactly one copy in memory, shared by all instances of the class and all threads in the JVM. This has two important implications:

- **Memory** - Static variables are not duplicated per object instance. They exist for the lifetime of the ClassLoader that loaded the class.
- **Thread safety** - Because static variables are shared, concurrent reads and writes from multiple threads can cause data races. Access must be protected with synchronization or by using atomic types.

```java
public class Counter {
    private static volatile int count = 0; // volatile ensures visibility across threads

    public static synchronized void increment() {
        count++; // synchronized ensures atomicity of read-modify-write
    }
}
```

`volatile` alone is insufficient for compound operations like `count++`. `synchronized` or `AtomicInteger` is required for correctness.

---

**Level: Advanced**

---

**Q4.5: What causes OutOfMemoryError: Metaspace?**

`OutOfMemoryError: Metaspace` is thrown when the JVM cannot allocate more native memory for class metadata and `-XX:MaxMetaspaceSize` has been reached (or system memory is exhausted if no cap is set).

Common causes:

- Excessive dynamic class generation by frameworks such as cglib, Javassist, or Spring's proxy infrastructure. Each dynamically generated class occupies Metaspace.
- ClassLoader leaks in application servers. When an application is redeployed, the old ClassLoader and all classes it loaded should become eligible for GC. If anything holds a reference to the old ClassLoader, the classes remain in Metaspace indefinitely.
- Libraries that generate a large number of anonymous or synthetic classes at runtime.

Resolution:

- Increase the cap: `-XX:MaxMetaspaceSize=512m`.
- Monitor class loading with `-XX:+TraceClassLoading` to identify which classes are accumulating.
- Use Eclipse MAT to analyze a heap dump and identify ClassLoader leak suspects.
- Fix the root cause in application or framework configuration to reduce dynamic class generation.

---

**Q4.6: How does class unloading work in Metaspace?**

A class can be unloaded from Metaspace only when all three of the following conditions are simultaneously true:

1. No instances of the class exist anywhere in the heap.
2. The ClassLoader that loaded the class is itself unreachable and eligible for GC.
3. The `Class` object representing the class is not referenced from anywhere.

Class unloading is triggered during Full GC. If the conditions are met, the class is unloaded and its Metaspace memory is returned directly to the OS (a significant improvement over PermGen, where memory was returned to the fixed-size pool but not the OS).

Bootstrap and system ClassLoaders are never unloaded, so classes loaded by them (all core JDK classes) are never unloaded during the lifetime of the process.

Monitoring: `-XX:+TraceClassUnloading`

## 5. Common Errors and Scenarios

**Level: Beginner**

---

**Q5.1: Explain the difference between Heap and Stack memory with a code example.**

```java
public class MemoryDemo {
    private static int staticVar = 100;   // Stored in Method Area (Metaspace)
    private int instanceVar = 200;        // Stored in Heap, inside the object instance

    public void methodDemo() {
        int localVar = 300;               // Stored in Stack (current frame)
        String localString = "Hello";     // Reference in Stack; object in String Pool (Heap)
        Object localObject = new Object();// Reference in Stack; object in Heap

        // When methodDemo() returns, localVar, localString, and localObject references
        // are removed from the Stack. The objects in the Heap become eligible for GC
        // if no other references point to them.
    }
}
```

Stack is thread-local, fast, and automatically cleaned up at method return. Heap is shared, GC-managed, and persists until no reachable reference exists.

---

**Q5.2: What happens during Garbage Collection?**

```java
public class GCDemo {
    public static void main(String[] args) {
        List<String> list1 = new ArrayList<>(); // Reachable; will survive GC
        List<String> list2 = new ArrayList<>(); // Will become eligible for GC

        list1.add("Keep me");
        list2.add("Temporary");

        list2 = null; // list2 reference is dropped; ArrayList object is now unreachable
    }
}
```

When a Minor GC is triggered (Eden fills):

1. **Mark phase** - JVM traverses object references from GC roots and marks all reachable objects as live.
2. **Copy phase** - Live objects (such as `list1`'s ArrayList) are copied to a Survivor space or promoted.
3. **Clear phase** - Eden and the inactive Survivor space are cleared. The unreachable `list2` ArrayList is reclaimed.
4. **Promotion** - Objects that have survived the tenuring threshold are moved to the Old Generation.

Major GC applies the same logic to the Old Generation. Full GC applies it to the entire heap including Metaspace.

---

**Level: Intermediate**

---

**Q5.3: When would you get StackOverflowError vs OutOfMemoryError?**

```java
// StackOverflowError - caused by infinite or very deep recursion
public class StackOverflowDemo {
    public void recursiveMethod() {
        recursiveMethod(); // No base case; stack frames accumulate until the stack is exhausted
    }
}

// OutOfMemoryError: Java heap space - caused by exhausting heap memory
public class OOMDemo {
    public static void main(String[] args) {
        List<byte[]> memoryHog = new ArrayList<>();
        while (true) {
            memoryHog.add(new byte[1024 * 1024]); // 1 MB per iteration; heap eventually exhausted
        }
    }
}
```

`StackOverflowError` is a Stack problem caused by too many nested method calls. The heap may be mostly empty when it occurs.

`OutOfMemoryError` is a Heap (or Metaspace, or Direct Memory) problem caused by allocating more objects than the GC can reclaim.

These are distinct failure modes with distinct root causes and distinct resolution strategies.

---

**Q5.4: Explain String Pool behavior with reference equality examples.**

```java
public static void main(String[] args) {
    String s1 = "Hello";              // Allocated in String Pool
    String s2 = "Hello";              // Reuses the same String Pool object
    String s3 = new String("Hello");  // New object allocated in Heap; not in Pool
    String s4 = s3.intern();          // intern() returns the Pool reference

    System.out.println(s1 == s2);     // true  - s1 and s2 point to the same pool object
    System.out.println(s1 == s3);     // false - s3 is a different heap object
    System.out.println(s1 == s4);     // true  - intern() returned the pool reference
    System.out.println(s1.equals(s3));// true  - equals() compares character content, not identity
}
```

The `==` operator compares reference identity (whether two variables point to the exact same object). The `equals()` method on `String` compares the character content. String literals with identical content share a single pool object; `new String()` always creates a new heap object regardless of the pool.

## 6. JVM Security

**Level: Beginner**

---

**Q6.1: Why is the JVM considered secure?**

The JVM provides several layers of protection that make it more secure than native code execution:

- **Bytecode Verification** - Before executing a class, the JVM verifier checks that its bytecode is well-formed, type-safe, and adheres to JVM specification rules. This prevents malformed code from causing illegal memory accesses or stack corruption.
- **Class Loader Architecture** - The parent-delegation model prevents untrusted code from replacing core Java classes. A custom ClassLoader cannot override `java.lang.String` because the bootstrap ClassLoader always takes priority.
- **No Pointer Arithmetic** - Java programs cannot compute arbitrary memory addresses or dereference raw pointers, eliminating entire categories of buffer overflow attacks.
- **Security Manager (historical)** - Provided runtime access control for sensitive operations. Deprecated in Java 17 and removed in Java 24, with the module system providing encapsulation-based alternatives.
- **Type Safety** - The type system enforced at both compile time and runtime prevents illegal casts and type confusion attacks.

---

**Q6.2: What is the role of the Security Manager?**

The `java.lang.SecurityManager` (deprecated from Java 17, removed in Java 24) was the JVM's runtime access control mechanism. Before any sensitive operation, the JVM called the Security Manager to check whether the calling code had the required permission.

Operations controlled by Security Manager included:

- Reading and writing files.
- Opening network connections.
- Reading system properties.
- Loading native libraries.
- Reflective access to private members.
- Spawning new threads or processes.

Applications configured a `java.policy` file listing allowed permissions and activated the Security Manager with `-Djava.security.manager`.

Modern alternatives include the Java module system for encapsulation, OS-level containerization (Docker, Kubernetes) for process isolation, and framework-level access control.

---

**Level: Intermediate**

---

**Q6.3: What are common JVM security risks?**

**Deserialization attacks** - Java's `ObjectInputStream` can instantiate and invoke methods on arbitrary classes when deserializing an untrusted stream. Attackers exploit gadget chains (sequences of existing classes) to achieve remote code execution. This is one of the most serious vulnerability classes in Java applications.

**Reflection misuse** - The Reflection API can bypass access modifiers, allowing code to read private fields (credentials, keys), modify immutable data, or invoke private methods that skip validation logic. Untrusted code or unsanitized inputs that influence reflective operations are particularly dangerous.

**Unrestricted class loading** - Dynamically loading untrusted bytecode without verification allows arbitrary code execution. Frameworks that process external inputs to determine which classes to load must validate and restrict those inputs.

**JNI vulnerabilities** - Code called via JNI operates outside JVM safety guarantees. Native code can access arbitrary memory, bypass type safety, and crash the JVM process. Bugs in JNI code (buffer overflows, use-after-free) directly compromise the JVM.

---

**Level: Advanced**

---

**Q6.4: How do you secure a JVM application in production?**

Security in production requires measures across code, configuration, and deployment:

**Code-level measures:**

- Never deserialize data from untrusted sources using `ObjectInputStream`. Use `ObjectInputFilter` if Java deserialization is unavoidable. Prefer safer serialization formats such as JSON with schema validation or Protocol Buffers.
- Restrict reflection to known, controlled cases. Use the module system to prevent external code from accessing internal packages via `--add-opens`.
- Declare classes and fields `final` where mutation is not intended.
- Validate and sanitize all external inputs before use.

**JVM configuration:**

- Run on the latest patched JDK release.
- Set `-XX:MaxMetaspaceSize` to prevent native memory exhaustion from classloader leaks.
- Enable `-XX:+HeapDumpOnOutOfMemoryError` with a restricted output path to capture post-incident state.

**Deployment:**

- Run the JVM process as a non-root OS user with minimum required privileges.
- Use container isolation (Docker, Kubernetes security contexts) to restrict what the JVM process can access on the host.
- Sign JAR files with `jarsigner` and verify signatures before deployment to detect tampering.
- Enforce TLS for all service-to-service communication and configure cipher suites to exclude known-weak algorithms.
- Scan all classpath dependencies for known CVEs using tools such as OWASP Dependency-Check or Snyk as part of the CI/CD pipeline.
- Restrict the classpath to trusted, verified directories only.

## 1. JVM Memory Architecture

### 1.1 Basic Concepts 🟢

**Q1.1:** What are the main memory areas in JVM?



- **Heap Memory**: Stores all objects and arrays (shared)
- **Stack Memory**: Method calls, local variables (per thread)
- **Method Area/Metaspace**: Class metadata, static variables
- **PC Registers**: Program counter for each thread
- **Native Method Stacks**: For JNI calls
</details>

**Q1.2:** What is the difference between Heap and Stack memory?



| Aspect | Heap | Stack |
|--------|------|-------|
| **Scope** | Shared across all threads | Per thread |
| **Stores** | Objects, arrays, instance variables | Method calls, local variables, references |
| **Speed** | Slower (managed by GC) | Faster (LIFO structure) |
| **Size** | Larger, configurable | Smaller, per thread |
| **Management** | Automatic (GC) | Automatic (method scope) |
</details>

**Q1.3:** Can you store objects in Stack memory?



**No.** Stack can only store:
- Primitive variables (int, double, etc.)
- Object references (pointers to heap objects)
- Method call information

Objects themselves are always created in Heap memory.
</details>

### 1.2 Memory Relationships 🟡

**Q1.4:** How do Stack and Heap interact with each other?



```java
public void example() {
    int primitive = 42;              // Stack: stores value directly
    String reference = "Hello";      // Stack: stores reference → Heap: stores object
    List<String> list = new ArrayList<>(); // Stack: reference → Heap: ArrayList object
}
```
Stack holds references that point to objects in Heap.
</details>

**Q1.5:** What happens to Stack and Heap when a method completes execution?



- **Stack**: Method's stack frame is immediately removed (LIFO)
- **Heap**: Objects remain until garbage collected
- **References**: Lost when stack frame is removed, making objects eligible for GC
</details>

```
│ Code Cache      │ ← Compiled native code (JIT)
```

## 2. Heap Memory

### 2.1 Heap Structure 🟢

**Q2.1:** What are the different generations in Heap memory?



1. **Young Generation**:
   - Eden Space (new objects)
   - Survivor 0 (S0) 
   - Survivor 1 (S1)

2. **Old Generation** (Tenured):
   - Long-lived objects promoted from Young Gen

**Default Ratio**: Young:Old ≈ 1:2
</details>

**Q2.2:** Where are new objects created in Heap?



**Eden Space** (part of Young Generation). All new objects start here, except for very large objects which may go directly to Old Generation.
</details>

**Q2.3:** What is the purpose of Survivor spaces (S0 and S1)?



- Hold objects that survived at least one Minor GC
- **Only one is active** at a time
- Objects move between S0 ↔ S1 during GC cycles
- After surviving 8-15 cycles, objects are promoted to Old Generation
</details>

### 2.2 Object Lifecycle 🟡

**Q2.4:** Explain the lifecycle of an object from creation to garbage collection.



```
Object Lifecycle:
New Object → Eden Space → Minor GC → Survivor Space (S0/S1) 
→ Multiple GC Cycles → Old Generation → Major GC → Collected
```

1. **Creation**: Object allocated in Eden
2. **First GC**: If reachable, moves to Survivor space
3. **Subsequent GCs**: Moves between S0 ↔ S1
4. **Promotion**: After 8-15 GC cycles, moves to Old Generation
5. **Collection**: Eventually garbage collected when unreachable
</details>

**Q2.5:** When does an object get promoted from Young to Old Generation?



**Conditions for promotion:**
1. Object survives multiple Minor GC cycles (default: 15)
2. Survivor space is full (premature promotion)
3. Object is too large for Survivor space
4. JVM optimization decisions

**Parameter**: `-XX:MaxTenuringThreshold=15`
</details>

**Q2.6:** What happens if Eden space fills up?



**Minor GC is triggered:**
1. All application threads are paused (Stop-the-World)
2. Reachable objects in Eden are copied to Survivor space
3. Eden space is cleared
4. Active Survivor space is swapped (S0 ↔ S1)
5. Application threads resume
</details>

### 2.3 Advanced Heap Concepts 🔴

**Q2.7:** How does heap memory allocation work internally?



**TLAB (Thread Local Allocation Buffers)**:
- Each thread gets a small portion of Eden space
- Reduces synchronization overhead for object allocation
- When TLAB fills up, thread requests new TLAB
- Very large objects bypass TLAB and allocate directly

**Allocation Process**:
1. Check TLAB space
2. If available, allocate in TLAB (fast path)
3. If TLAB full, get new TLAB
4. If Eden full, trigger Minor GC
</details>

**Q2.8:** What are Humongous Objects in G1GC context?



**Humongous Objects**: Objects larger than 50% of G1 heap region size
- **Allocation**: Directly allocated in Old Generation
- **Collection**: Special handling during GC cycles  
- **Impact**: Can cause performance issues if too many
- **Tuning**: Adjust `-XX:G1HeapRegionSize` to optimize
</details>

## 3. Stack Memory

### 3.1 Stack Basics 🟢

**Q3.1:** What is stored in Stack memory?



**Stack stores:**
- Method call frames
- Local primitive variables (int, double, boolean, etc.)
- Object references (not the objects themselves)
- Method parameters
- Return addresses
- Intermediate computation results
</details>

**Q3.2:** Is Stack memory shared between threads?



**No.** Each thread has its own Stack memory. This ensures thread safety for local variables and method calls.
</details>

**Q3.3:** What is a Stack Frame?



A **Stack Frame** is created for each method call and contains:
- **Local Variable Array**: Method parameters and local variables
- **Operand Stack**: For intermediate calculations
- **Frame Data**: Method metadata, return address, exception handling info
</details>

### 3.2 Stack Operations 🟡

**Q3.4:** How does method invocation work with Stack?



```java
public void methodA() {          // Frame A pushed
    int x = 10;                  // Local variable in Frame A
    methodB();                   // Frame B pushed on top
}                               // Frame A popped when methodA ends

public void methodB() {          // Frame B
    int y = 20;                 // Local variable in Frame B  
}                               // Frame B popped when methodB ends
```

**LIFO Order**: Last method called is first to complete.
</details>

**Q3.5:** What happens to local variables when a method exits?



**Automatic cleanup:**
- Stack frame is immediately removed
- All local variables are instantly cleared
- Object references are lost (objects become eligible for GC)
- No manual cleanup required
</details>

### 3.3 Stack Performance & Issues 🔴

**Q3.6:** What causes StackOverflowError and how to fix it?



**Causes:**
- Infinite or very deep recursion
- Too many nested method calls
- Insufficient stack size for application needs

**Solutions:**
- Fix recursive logic with base conditions
- Convert recursion to iterative approach
- Increase stack size: `-Xss2m`
- Optimize method call depth
</details>

**Q3.7:** How does Stack size affect application performance?



**Impact:**
- **Larger Stack**: More memory per thread, can handle deeper calls
- **Smaller Stack**: Less memory usage, but limited call depth
- **Trade-off**: Memory vs. functionality

**Calculation**: For 1000 threads with 1MB stack each = 1GB memory just for stacks

**Tuning**: `-Xss512k` for lightweight applications, `-Xss2m` for deep recursion
</details>

### 4.1 Method Area Basics 🟢

**Q4.1:** What is Method Area and what does it store?

- Class metadata (structure, methods, fields)

- Static variables and methods
- Method bytecode
- String literals (in Runtime Constant Pool)

**Q4.2:** What is the difference between PermGen and Metaspace?

| **Location** | Inside Heap | Native Memory |
| **Size** | Fixed (-XX:MaxPermSize) | Dynamic growth |
| **GC** | Part of Heap GC | Separate cleanup |

| **String Pool** | In PermGen | Moved to Heap |

### 4.2 Static Variables & Methods 🟡

**Q4.3:** Where are static variables stored and when are they initialized?



**Storage**: Method Area (Metaspace)
**Initialization**: When class is first loaded by ClassLoader


**Q4.4:** How do static variables behave with respect to memory and threads?



**Characteristics:**
- **Single copy**: Shared across all instances and threads
- **Memory location**: Method Area (not per-object)
- **Lifecycle**: Loaded with class, GC'd when class is unloaded
- **Thread safety**: Need synchronization if modified by multiple threads

3. Class object itself is not referenced

**Process:**
- Metaspace cleanup happens during Full GC
- Classes are unloaded if conditions are met
- Memory is returned to the system (unlike PermGen)

**Monitoring**: `-XX:+TraceClassUnloading`

## 5. Garbage Collection

### 5.1 GC Fundamentals 🟢

**Q5.1:** What is Garbage Collection and why is it needed?



**Garbage Collection** is automatic memory management that:
- **Identifies** unreachable objects
- **Reclaims** memory used by these objects
- **Prevents** memory leaks
- **Optimizes** memory usage

**Need**: Java doesn't have explicit memory deallocation (no free/delete)
</details>

**Q5.2:** What makes an object eligible for Garbage Collection?



An object is eligible for GC when:
1. **No references** point to it from any reachable object
2. **Reference is set to null**: `obj = null;`
3. **Reference goes out of scope**: method ends, local variable cleared
4. **Circular references** with no external references

```java
// Eligible for GC
String str = new String("Hello");
str = null;  // Original "Hello" object eligible for GC

// Not eligible 
static List<String> list = new ArrayList<>();
list.add("Keep me");  // Referenced by static variable
```
</details>

**Q5.3:** What are the different types of Garbage Collection?



1. **Minor GC**: 
   - Scope: Young Generation only
   - Frequency: High (every few seconds)
   - Duration: Fast (milliseconds)

2. **Major GC**: 
   - Scope: Old Generation only  
   - Frequency: Low (minutes/hours)
   - Duration: Slower (tens of milliseconds to seconds)

3. **Full GC**:
   - Scope: Entire Heap + Metaspace
   - Frequency: Rare (under memory pressure)
   - Duration: Slowest (seconds)
</details>

### 5.2 GC Process & Algorithms 🟡

**Q5.4:** Explain the Mark and Sweep algorithm.



**Mark Phase:**
1. Start from GC roots (static variables, local variables, JNI references)
2. Traverse object references recursively  
3. Mark all reachable objects as "live"

**Sweep Phase:**
1. Scan through entire heap
2. Identify unmarked (dead) objects
3. Deallocate memory used by dead objects
4. Reset marks for next GC cycle

**Optional Compact Phase:**
- Move live objects together
- Eliminate fragmentation
- Update references to moved objects
</details>

**Q5.5:** What are GC roots and why are they important?



**GC Roots** are starting points for reachability analysis:

**Types of GC Roots:**
- Local variables in active method calls (Stack)
- Static variables (Method Area)
- JNI (Java Native Interface) references
- Active thread objects
- Objects in system class loader

**Importance:** Objects reachable from GC roots are considered "live" and won't be garbage collected.
</details>

**Q5.6:** How does Generational Garbage Collection work?



**Generational Hypothesis:** Most objects die young

**Strategy:**
1. **Young Generation GC**: Frequent, fast collection of short-lived objects
2. **Old Generation GC**: Infrequent, slower collection of long-lived objects
3. **Benefits**: Optimizes for common allocation patterns

**Write Barriers**: Track references from Old → Young generation for accurate GC
</details>

### 5.3 Advanced GC Concepts 🔴

**Q5.7:** What is the difference between concurrent and parallel garbage collection?



**Parallel GC:**
- Multiple GC threads work simultaneously
- All application threads are paused (Stop-the-World)
- Reduces GC time but doesn't reduce pause time
- Example: ParallelGC

**Concurrent GC:**
- GC threads run alongside application threads
- Minimal Stop-the-World pauses
- May use more CPU resources
- Example: G1GC, CMS, ZGC

**Trade-off:** Throughput vs. Latency
</details>

**Q5.8:** Explain the concept of GC ergonomics.



**GC Ergonomics**: JVM automatically tunes GC parameters based on:

**Goals:**
- **Pause Time Goal**: `-XX:MaxGCPauseMillis=200`
- **Throughput Goal**: `-XX:GCTimeRatio=99` (1% time in GC)
- **Memory Footprint**: Minimize heap size

**Auto-tuning:**
- Adjusts heap generations ratios
- Modifies GC algorithm parameters  
- Balances competing goals
- Adapts to application behavior

**Monitoring**: `-XX:+PrintGCApplicationStoppedTime`
</details>

## 6. GC Algorithms

### 6.1 Serial & Parallel GC 🟢

**Q6.1:** When would you use Serial GC?



**Use Cases:**
- Small applications (heap < 100MB)
- Single-core machines
- Client applications
- Embedded systems
- Development/testing environments

**Characteristics:**
- Single-threaded GC
- Simple implementation
- Low memory overhead
- Not suitable for multi-core production systems

**Flag**: `-XX:+UseSerialGC`
</details>

**Q6.2:** What is the difference between Parallel GC and Parallel Old GC?



**Parallel GC** (ParallelGC):
- Multi-threaded Minor GC (Young Generation)
- Single-threaded Major GC (Old Generation)
- Good for throughput-oriented applications

**Parallel Old GC**:
- Multi-threaded both Minor and Major GC
- Better performance for Old Generation collection
- Default before Java 9

**Flags**: 
- `-XX:+UseParallelGC`
- `-XX:+UseParallelOldGC`
</details>

### 6.2 Concurrent Mark Sweep (CMS) 🟡

**Q6.3:** How does CMS GC work and what are its phases?



**CMS Phases:**
1. **Initial Mark** (STW): Mark objects directly reachable from GC roots
2. **Concurrent Mark**: Mark all reachable objects (concurrent with app)
3. **Concurrent Preclean**: Handle objects modified during marking
4. **Remark** (STW): Finalize marking of all live objects
5. **Concurrent Sweep**: Free memory of dead objects (concurrent)
6. **Concurrent Reset**: Prepare for next cycle

**Benefits**: Low pause times
**Drawbacks**: CPU overhead, fragmentation, deprecated in Java 9+
</details>

**Q6.4:** What problems can occur with CMS GC?



**Issues:**
1. **Fragmentation**: No compaction phase leads to memory fragmentation
2. **Concurrent Mode Failure**: CMS can't keep up with allocation rate
3. **CPU Overhead**: Background threads consume CPU
4. **Promotion Failure**: Young Gen objects can't be promoted due to fragmentation

**Solutions:**
- Increase heap size
- Tune CMS trigger points: `-XX:CMSInitiatingOccupancyFraction=70`
- Switch to G1GC (modern alternative)
</details>

### 6.3 G1 Garbage Collector 🟡

**Q6.5:** How does G1GC differ from traditional generational collectors?



**Traditional GC**: Fixed Young/Old generation regions
**G1GC**: Heap divided into equal-sized regions (1MB-32MB)

**G1 Regions:**
- **Eden Regions**: New object allocation
- **Survivor Regions**: Objects that survived GC
- **Old Regions**: Long-lived objects
- **Humongous Regions**: Large objects (>50% region size)

**Benefits:**
- Predictable pause times
- Better handling of large heaps
- Incremental collection
- Automatic tuning
</details>

**Q6.6:** What is the G1GC evacuation process?



**Evacuation**: Moving live objects from selected regions

**Process:**
1. **Pause Prediction**: Select regions based on garbage ratio and pause goal
2. **Root Scanning**: Find references to selected regions
3. **Update Remembered Sets**: Track inter-region references
4. **Object Copy**: Evacuate live objects to new regions
5. **Reference Update**: Update all references to moved objects

**Goal**: Achieve target pause time (`-XX:MaxGCPauseMillis=200`)
</details>

### 6.4 Low-Latency Collectors 🔴

**Q6.7:** What are the key features of ZGC?



**ZGC Features:**
- **Ultra-low pause times**: <10ms regardless of heap size
- **Concurrent operations**: Most work done concurrently
- **Colored pointers**: Encode object state in pointer
- **Load barriers**: Check object state during access
- **Massive heap support**: Up to 16TB

**Trade-offs:**
- Higher memory overhead (2-16%)
- CPU overhead for load barriers
- Requires Java 11+

**Flag**: `-XX:+UseZGC`
</details>

**Q6.8:** When should you consider using Shenandoah GC?



**Use Cases:**
- Applications requiring consistent low latency
- Large heaps (multi-GB) with latency requirements
- Real-time or near-real-time systems
- When pause time is more critical than throughput

**Characteristics:**
- Concurrent evacuation
- Low pause times (typically <10ms)
- Available in OpenJDK
- Similar goals to ZGC with different implementation

**Flag**: `-XX:+UseShenandoahGC`
</details>

### 7.3 Memory Optimization 🔴

**Q7.5:** What are the best practices for minimizing GC pressure?



**Object Allocation Optimization:**
```java
// ❌ BAD - Creates many temporary objects
public String buildString(List<String> items) {
    String result = "";
    for (String item : items) {
        result += item;  // New String object each iteration
    }
    return result;
}

// ✅ GOOD - Reuses StringBuilder
public String buildString(List<String> items) {
    StringBuilder sb = new StringBuilder(items.size() * 16);
    for (String item : items) {
        sb.append(item);
    }
    return sb.toString();
}
```

**Other Practices:**
- Object pooling for expensive objects
- Primitive collections (avoid boxing)
- Lazy initialization
- Proper collection sizing
- Avoid memory leaks (listeners, static references)

**Q7.6:** How do you handle memory leaks in Java applications?



**Common Leak Patterns:**
1. **Static Collections**: Unbounded growth
2. **Listeners**: Not removing event listeners
3. **ThreadLocal**: Not calling `remove()`
4. **Inner Classes**: Holding outer class references
5. **ClassLoader Leaks**: In application servers

**Detection:**
- Heap dump analysis with Eclipse MAT
- Monitor heap growth over time
- Profile with tools like JProfiler

**Prevention:**
```java
// ✅ Proper cleanup
public class LeakPrevention {
    private static final Map<String, Object> cache = 
        new ConcurrentHashMap<>();
    
    // Bounded cache
    public void addToCache(String key, Object value) {
        if (cache.size() > 10000) {
            cache.clear(); // Simple strategy
        }
        cache.put(key, value);
    }
    
    // ThreadLocal cleanup
    private static ThreadLocal<Object> threadLocal = new ThreadLocal<>();
    
    public void cleanup() {
        try {
            // Use ThreadLocal
        } finally {
            threadLocal.remove(); // Always clean up
        }
    }
}
```

## 8. Troubleshooting & Errors

### 8.1 OutOfMemoryError Variants 🟢

**Q8.1:** What are the different types of OutOfMemoryError?



1. **`OutOfMemoryError: Java heap space`**
   - Cause: Heap memory exhausted
   - Solution: Increase `-Xmx`, fix memory leaks

2. **`OutOfMemoryError: Metaspace`**  
   - Cause: Too many classes loaded
   - Solution: Increase `-XX:MaxMetaspaceSize`

3. **`OutOfMemoryError: GC overhead limit exceeded`**
   - Cause: >98% time in GC, <2% memory freed
   - Solution: Increase heap, optimize code

4. **`OutOfMemoryError: Direct buffer memory`**
   - Cause: NIO direct buffers exhausted
   - Solution: Increase `-XX:MaxDirectMemorySize`
