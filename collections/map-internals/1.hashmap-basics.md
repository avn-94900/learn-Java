## Internal Structure and Working

HashMap stores key-value pairs in an array of nodes called **buckets**.

### How a put() works

1. Java calls `hashCode()` on the key.
2. The hash is processed to find the bucket index: `index = hash(key) & (n-1)` where `n` is the capacity.
3. The key-value pair is stored as a `Node` at that index.

### Node structure evolution

| Version | Collision Structure | Worst-case Lookup |
|---------|--------------------|--------------------|
| Before Java 8 | Linked list per bucket | O(n) |
| Java 8+ | Linked list, converts to Red-Black Tree at 8 nodes | O(log n) |

- Converts to tree when nodes in a bucket >= **TREEIFY_THRESHOLD (8)**
- Converts back to list when nodes fall below **UNTREEIFY_THRESHOLD (6)**

### Null support

- Allows **one null key** and **multiple null values**.

```java
HashMap<String, Integer> userScores = new HashMap<>();
userScores.put("Alice", 95);
userScores.put("Bob", 82);

int score = userScores.get("Alice"); // 95
boolean exists = userScores.containsKey("David"); // false

for (Map.Entry<String, Integer> entry : userScores.entrySet()) {
    System.out.println(entry.getKey() + ": " + entry.getValue());
}
```

## Collision Handling

A **collision** happens when two keys hash to the same bucket.

### Resolution process

1. Compute the key's `hashCode()`.
2. Find the bucket index.
3. If the bucket is empty, insert a new node.
4. If the bucket is occupied, compare keys using `equals()`.
   - Key matches: replace the value.
   - Key is new: add to the chain (list or tree).

### Internal node classes (simplified)

```java
// Linked list node
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;
}

// Tree node (used after treeification)
static final class TreeNode<K,V> extends LinkedHashMap.Entry<K,V> {
    TreeNode<K,V> parent, left, right, prev;
    boolean red;
}
```

> A well-implemented `equals()` and `hashCode()` in your key class reduces collisions.

## Duplicate Keys

When you `put()` a key that already exists:

- The **new value replaces the old value**.
- `put()` **returns the previous value**.
- The **size of the map stays the same**.

```java
HashMap<String, Integer> map = new HashMap<>();
map.put("apple", 10);
map.put("banana", 20);

Integer previous = map.put("apple", 30);
System.out.println(previous); // 10
System.out.println(map);      // {apple=30, banana=20}
System.out.println(map.size()); // 2
```

### Related methods

| Method | Behavior |
|--------|----------|
| `put(k, v)` | Always inserts or replaces |
| `putIfAbsent(k, v)` | Inserts only if key is absent or mapped to null |
| `replace(k, v)` | Replaces only if key already exists |

```java
map.putIfAbsent("apple", 40); // No change, apple already exists
map.putIfAbsent("cherry", 50); // Adds cherry=50
```

## Load Factor and Capacity

These two parameters control **performance** and **memory usage**.

### Initial Capacity

- **Default**: 16 (always a power of 2)
- Number of buckets when the map is created.
- Higher capacity = less resizing, more memory used upfront.

### Load Factor

- **Default**: 0.75
- Formula: `Load Factor = entries / buckets`
- Resize triggers when: `entries > capacity × load factor`

| Load Factor | Effect |
|-------------|--------|
| Higher (e.g. 0.9) | Less memory, more collisions, slower lookups |
| Lower (e.g. 0.5) | More memory, fewer collisions, faster lookups |

### Resizing (Rehashing)

- Creates a new array with **double the capacity**.
- Rehashes all existing entries.
- Cost: **O(n)** — avoid by setting initial capacity upfront.
- Capacity always grows as a power of 2: `16 → 32 → 64 → 128`

### Constructors

```java
HashMap<String, Integer> map1 = new HashMap<>();           // capacity=16, load=0.75
HashMap<String, Integer> map2 = new HashMap<>(32);         // capacity=32, load=0.75
HashMap<String, Integer> map3 = new HashMap<>(32, 0.6f);   // capacity=32, load=0.6
```

### Best practice: pre-size for known data

```java
int expectedEntries = 100;
float loadFactor = 0.75f;
int initialCapacity = (int)(expectedEntries / loadFactor);

HashMap<String, Integer> map = new HashMap<>(initialCapacity, loadFactor);
```

## Designing a Good Key

### Core Requirements

- **Immutable (preferred)** — key fields should not change after insertion.
  If fields used in `equals()` or `hashCode()` change, lookup will fail.

- **Consistent `hashCode()`**
  - Must return the **same value while the object is in the map**.
  - Equal objects **must** have the same hash code.

- **Correct `equals()`**
  - Defines logical equality between keys.
  - If `equals()` is overridden, **`hashCode()` must also be overridden**.

- **Good hash distribution**
  - Hash values should spread evenly across buckets.
  - Avoid constant or poorly distributed hash codes.

### Recommended Key Types

| Type | Notes |
|------|-------|
| `String` | Immutable, optimized hash implementation |
| Wrapper classes (`Integer`, `Long`) | Safe and efficient |
| `Enum` | Ideal — unique constants |
| `Record` | Best modern choice — auto generates `equals()` and `hashCode()` |
| Custom `final` class | Must override `equals()` and `hashCode()` |

### Custom Key Example (Correct Design)

```java
import java.time.LocalDate;
import java.util.Objects;

public final class PersonKey {

    private final String firstName;
    private final String lastName;
    private final LocalDate dateOfBirth;

    public PersonKey(String firstName,
                     String lastName,
                     LocalDate dateOfBirth) {

        this.firstName = firstName;
        this.lastName = lastName;
        this.dateOfBirth = dateOfBirth;
    }

    @Override
    public boolean equals(Object o) {

        if (this == o) return true;

        if (o == null || getClass() != o.getClass())
            return false;

        PersonKey pk = (PersonKey) o;

        return Objects.equals(firstName, pk.firstName)
            && Objects.equals(lastName, pk.lastName)
            && Objects.equals(dateOfBirth, pk.dateOfBirth);
    }

    @Override
    public int hashCode() {

        return Objects.hash(
                firstName,
                lastName,
                dateOfBirth
        );
    }
}
```

✔ Immutable  
✔ Correct equals  
✔ Correct hashCode  
✔ Safe as HashMap key

### Mutable Key Problem (Correct Demonstration)

```java
import java.util.HashMap;
import java.util.Objects;

// BAD: mutable key
class MutableKey {

    private String value; // mutable field

    public MutableKey(String value) {
        this.value = value;
    }

    public void setValue(String value) {
        this.value = value;
    }

    @Override
    public int hashCode() {
        return Objects.hashCode(value);
    }

    @Override
    public boolean equals(Object obj) {

        if (this == obj) return true;

        if (!(obj instanceof MutableKey))
            return false;

        MutableKey other = (MutableKey) obj;

        return Objects.equals(this.value, other.value);
    }
}

public class Demo {

    public static void main(String[] args) {

        HashMap<MutableKey, String> map = new HashMap<>();

        MutableKey key = new MutableKey("initial");

        map.put(key, "value");

        // Change key after insertion
        key.setValue("changed");

        System.out.println(map.get(key));
        // prints null

        System.out.println(map.size());
        // prints 1 (entry still exists internally)
    }
}
```

### Key Design Checklist

| Rule | Why It Matters |
|------|----------------|
| Make fields `final` | Prevents hash from changing |
| Override both `hashCode()` and `equals()` | Required for correct lookup |
| Keep key immutable | Prevents lookup failure |
| Include all identity fields | Avoids incorrect equality |
| Use `Objects.hash()` for multiple fields | Clean and correct |
| Ensure good hash distribution | Avoids performance issues |
| Never modify key after insertion | Prevents hidden bugs |

### Extra Best Practice (Modern Java)

If using modern Java, use **records** when possible.

```java
public record PersonKey(
        String firstName,
        String lastName,
        LocalDate dateOfBirth
) {}
```

Automatically provides:

✔ equals  
✔ hashCode  
✔ immutability

Very reliable HashMap key.

### Final One-Line Memory Rule

> **If a field used in `hashCode()` or `equals()` changes after insertion, HashMap lookup will fail even though the entry still exists.**

## HashMap Variants

### LinkedHashMap

- Extends `HashMap`.
- Maintains **insertion order** using a doubly-linked list connecting all entries.
- Each node has `before` and `after` pointers in addition to the standard bucket chain.
- Iteration order matches insertion order.
- Slightly more memory than `HashMap` due to extra pointers.
- Use when **order of insertion matters**.

### WeakHashMap

- Keys are held as **weak references**.
- When a key has no strong references elsewhere, the GC can collect it.
- The entry is automatically removed after GC collects the key.
- Use for **caches and metadata** where entries should not prevent GC.

```java
WeakHashMap<Object, String> cache = new WeakHashMap<>();
Object key = new Object();
cache.put(key, "data");

key = null; // no more strong references
System.gc();
// entry may be removed from map after GC
```

### EnumMap

- Keys must be from a **single enum type**.
- Internally uses an array indexed by enum ordinal — no hashing needed.
- Faster and more memory-efficient than `HashMap` for enum keys.
- Does **not allow null keys**.
- Use when your key set is a fixed enum.

### IdentityHashMap

- Uses **reference equality (`==`)** instead of `equals()` for key comparison.
- Uses `System.identityHashCode()` instead of `hashCode()`.
- Two keys are the same only if they are the **exact same object in memory**.
- Use for object graph traversal, serialization, or proxy tracking.

### Variant Comparison

| Map | Key Comparison | Order | Null Keys | Best For |
|-----|---------------|-------|-----------|----------|
| `HashMap` | `equals()` | None | 1 allowed | General use |
| `LinkedHashMap` | `equals()` | Insertion order | 1 allowed | Ordered iteration |
| `WeakHashMap` | `equals()` | None | 1 allowed | Caches |
| `EnumMap` | `==` (enum) | Enum order | Not allowed | Enum keys |
| `IdentityHashMap` | `==` | None | Allowed | Object identity |

## ConcurrentHashMap vs Hashtable

### Why ConcurrentHashMap is faster

| Feature | Hashtable | ConcurrentHashMap |
|---------|-----------|-------------------|
| Locking | Locks the entire map | Locks only the affected bucket |
| Concurrency | Single thread at a time | Multiple threads simultaneously |
| Null keys/values | Not allowed | Not allowed |
| Performance | Poor under contention | High under contention |

### How ConcurrentHashMap works internally

- Uses an array of **Node** buckets, same as HashMap.
- Uses `synchronized` on individual **bucket heads** (not the whole map).
- Uses **CAS (Compare-And-Swap)** operations for lock-free reads and inserts on empty buckets.
- Converts buckets to **Red-Black Trees** at threshold 8, same as HashMap.
- `size()` uses a distributed counter for efficiency.

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("key", 1);
map.computeIfAbsent("other", k -> 42);
```

## HashMap vs HashSet

| Feature | HashMap | HashSet |
|---------|---------|---------|
| Stores | Key-value pairs | Unique values only |
| Backed by | Direct implementation | Internally uses a HashMap |
| Null support | 1 null key, many null values | 1 null element |
| Access | By key | No direct access |
| Use case | Key-based lookup | Uniqueness checks |

HashSet stores its elements as keys in an internal HashMap with a dummy value.

## Time Complexity

| Operation | Average | Worst Case |
|-----------|---------|------------|
| `get(key)` | O(1) | O(log n) with trees, O(n) pre-Java 8 |
| `put(key, value)` | O(1) | O(log n) with trees |
| `remove(key)` | O(1) | O(log n) with trees |
| `containsKey(key)` | O(1) | O(log n) with trees |
| `containsValue(value)` | O(n) | O(n) |
| Iteration | O(n) | O(n) |
| Resize (rehash) | O(n) | O(n) |

> Worst case happens when many keys hash to the same bucket.

## Synchronizing HashMap

HashMap is **not thread-safe**. Use one of these approaches:

### Option 1: Collections.synchronizedMap

```java
Map<String, Integer> syncMap = Collections.synchronizedMap(new HashMap<>());

// Must synchronize manually during iteration
synchronized (syncMap) {
    for (Map.Entry<String, Integer> entry : syncMap.entrySet()) {
        System.out.println(entry.getKey() + ": " + entry.getValue());
    }
}
```

### Option 2: ConcurrentHashMap (preferred)

```java
Map<String, Integer> concurrentMap = new ConcurrentHashMap<>();
concurrentMap.put("key", 1);
// Safe for concurrent reads and writes without manual synchronization
```

### Option 3: Hashtable (legacy — avoid)

```java
Map<String, Integer> hashtable = new Hashtable<>();
// Synchronized but poor performance — use ConcurrentHashMap instead
```

### Comparison

| Approach | Thread-safe | Performance | Notes |
|----------|-------------|-------------|-------|
| `HashMap` | No | Best | Single-threaded use only |
| `synchronizedMap` | Yes | Moderate | Locks entire map |
| `ConcurrentHashMap` | Yes | Best | Bucket-level locking |
| `Hashtable` | Yes | Worst | Legacy, avoid |

## Common Coding Problems

### 1. Frequency count

```java
String[] words = {"apple", "banana", "apple", "cherry", "banana", "apple"};
Map<String, Integer> freq = new HashMap<>();

for (String word : words) {
    freq.put(word, freq.getOrDefault(word, 0) + 1);
}
// {apple=3, banana=2, cherry=1}
```

### 2. Two Sum

```java
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> seen = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int complement = target - nums[i];
        if (seen.containsKey(complement)) {
            return new int[]{seen.get(complement), i};
        }
        seen.put(nums[i], i);
    }
    return new int[]{};
}
```

### 3. Group anagrams

```java
public List<List<String>> groupAnagrams(String[] strs) {
    Map<String, List<String>> map = new HashMap<>();
    for (String s : strs) {
        char[] chars = s.toCharArray();
        Arrays.sort(chars);
        String key = new String(chars);
        map.computeIfAbsent(key, k -> new ArrayList<>()).add(s);
    }
    return new ArrayList<>(map.values());
}
```

### 4. First non-repeating character

```java
public char firstNonRepeating(String s) {
    Map<Character, Integer> freq = new LinkedHashMap<>();
    for (char c : s.toCharArray()) {
        freq.put(c, freq.getOrDefault(c, 0) + 1);
    }
    for (Map.Entry<Character, Integer> entry : freq.entrySet()) {
        if (entry.getValue() == 1) return entry.getKey();
    }
    return '_';
}
```

### 5. Subarray sum equals K

```java
public int subarraySum(int[] nums, int k) {
    Map<Integer, Integer> prefixCount = new HashMap<>();
    prefixCount.put(0, 1);
    int count = 0, sum = 0;
    for (int num : nums) {
        sum += num;
        count += prefixCount.getOrDefault(sum - k, 0);
        prefixCount.put(sum, prefixCount.getOrDefault(sum, 0) + 1);
    }
    return count;
}
```
