## Internal Structure and Working

`ConcurrentHashMap` provides a thread-safe map with high concurrency by using **fine-grained locking at the bucket level** rather than locking the entire map.

### Java 8+ internal structure

- Uses the same **array of Node buckets** as `HashMap`.
- Each bucket is an independent unit of synchronisation.
- For empty buckets, insertion uses **CAS (Compare-And-Swap)** — no lock acquired.
- For non-empty buckets, only the **head node of that bucket** is `synchronized`.
- Buckets convert to **Red-Black Trees** at `TREEIFY_THRESHOLD (8)`, same as `HashMap`.

### Node types

```java
// Standard linked list node
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;
    final K key;
    volatile V value;   // volatile for visibility across threads
    volatile Node<K,V> next;
}

// Tree node (after treeification)
static final class TreeNode<K,V> extends Node<K,V> {
    TreeNode<K,V> parent, left, right, prev;
    boolean red;
}

// ForwardingNode — placed in a bucket during resize
// to redirect reads to the new table
static final class ForwardingNode<K,V> extends Node<K,V> { ... }
```

### How a put() works

1. Compute `hashCode()` and find the bucket index.
2. If the bucket is **empty**: use CAS to insert — no lock needed.
3. If the bucket head has hash `MOVED` (`-1`): a resize is in progress — help finish it.
4. Otherwise: **`synchronized` on the bucket head**, then insert into the list or tree.

### How a get() works

- Entirely **lock-free** — reads `volatile` fields and traverses the bucket chain without acquiring any lock.

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("alpha", 1);
map.put("beta",  2);

System.out.println(map.get("alpha")); // 1
```

## Concurrency Mechanism

### CAS (Compare-And-Swap)

Used for inserting into **empty buckets** without any lock.

```
CAS(table[i], null, newNode)
  → if table[i] is still null, atomically set it to newNode
  → if another thread already inserted, retry
```

### Bucket-level synchronization

Used for inserting into **non-empty buckets**.

```java
synchronized (bucketHead) {
    // traverse or append to the bucket's list/tree
}
```

Only one thread can modify a given bucket at a time, but **threads operating on different buckets run completely in parallel**.

### Resize cooperation

During a resize, `ConcurrentHashMap` uses `ForwardingNode` markers. When a thread encounters a `ForwardingNode` during a write, it **helps transfer entries** to the new table rather than waiting. This spreads resize work across multiple threads.

```
Thread 1: put() on bucket 3 → encounters ForwardingNode
          → helps transfer buckets 10-15 to new table
          → then retries its own put() in the new table
```

## Null Support

- **Null keys are not allowed** — throws `NullPointerException`.
- **Null values are not allowed** — throws `NullPointerException`.

This is intentional. In a concurrent context, a `get()` returning `null` would be ambiguous — it could mean the key is absent, or that the key is mapped to `null`. Disallowing null eliminates the ambiguity.

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

map.put(null, 1);    // NullPointerException
map.put("key", null); // NullPointerException

// Safe pattern: check containsKey explicitly
if (map.containsKey("key")) {
    int value = map.get("key");
}
```

## Atomic Operations

`ConcurrentHashMap` provides atomic compound operations that eliminate the check-then-act race conditions common with plain `HashMap`.

### putIfAbsent

Atomically inserts only if the key is absent. Equivalent to a thread-safe `if (!map.containsKey(k)) map.put(k, v)`.

```java
map.putIfAbsent("counter", 0);
```

### computeIfAbsent

Atomically computes and inserts a value if the key is absent. The function is called at most once.

```java
// Thread-safe grouping
map.computeIfAbsent("group-A", k -> new CopyOnWriteArrayList<>()).add("item");
```

### compute

Atomically computes a new value from the current value (or `null` if absent). Returning `null` removes the entry.

```java
// Thread-safe frequency count
map.compute("word", (k, v) -> (v == null) ? 1 : v + 1);
```

### merge

Atomically merges a new value with the existing value using a provided function.

```java
// Thread-safe frequency count — cleaner than compute
map.merge("word", 1, Integer::sum);
```

### replace

Atomically replaces a value, optionally only if the current value matches.

```java
map.replace("key", 42);              // unconditional replace
map.replace("key", 42, 99);          // replace only if current value is 42
```

### Comparison of atomic methods

| Method | Inserts if absent | Uses current value | Removes on null return |
|--------|------------------|-------------------|------------------------|
| `putIfAbsent(k, v)` | Yes | No | No |
| `computeIfAbsent(k, fn)` | Yes | No | No |
| `computeIfPresent(k, fn)` | No | Yes | Yes |
| `compute(k, fn)` | Yes | Yes | Yes |
| `merge(k, v, fn)` | Yes | Yes | Yes |

## ConcurrentHashMap vs HashMap vs Hashtable

| Feature | HashMap | Hashtable | ConcurrentHashMap |
|---------|---------|-----------|-------------------|
| Thread-safe | No | Yes | Yes |
| Locking strategy | None | Entire map | Per bucket (CAS + synchronized) |
| Null keys | 1 allowed | Not allowed | Not allowed |
| Null values | Allowed | Not allowed | Not allowed |
| Performance (single thread) | Best | Moderate | Slightly below HashMap |
| Performance (multi thread) | Unsafe | Poor | Best |
| Iteration | Fail-fast | Fail-fast | Weakly consistent |
| Atomic compound ops | No | No | Yes |
| Introduced | Java 2 | Java 1 | Java 5 |
| Recommended | Single-threaded | Never (legacy) | Multi-threaded |

## Weak Consistency and Iteration

Iterators over `ConcurrentHashMap` are **weakly consistent** — they reflect the state of the map at some point during iteration and do not throw `ConcurrentModificationException`.

- An iterator may or may not see changes made after it was created.
- Entries inserted during iteration may or may not be returned.
- No entry is returned more than once.

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("a", 1);
map.put("b", 2);
map.put("c", 3);

// Safe to iterate and modify concurrently
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    if (entry.getValue() < 2) {
        map.remove(entry.getKey()); // no ConcurrentModificationException
    }
}
```

> Weak consistency is a deliberate trade-off. Strict consistency would require locking the entire map during iteration, eliminating all concurrency benefits.

## size() and Aggregate Operations

### size() is approximate

`size()` uses a distributed counter spread across multiple `CounterCell` objects to avoid contention. The returned value may not reflect concurrent puts and removes at the exact moment of the call.

```java
int approxSize = map.size(); // accurate enough for most uses
long exactSize = map.mappingCount(); // preferred for large maps — returns long
```

> Use `mappingCount()` instead of `size()` for large maps since `size()` returns an `int` and can overflow.

### Bulk / parallel operations (Java 8+)

`ConcurrentHashMap` provides parallel bulk operations that process entries in segments using the common fork-join pool.

```java
ConcurrentHashMap<String, Integer> scores = new ConcurrentHashMap<>();
scores.put("Alice", 90);
scores.put("Bob",   75);
scores.put("Carol", 88);

// forEach — parallel processing
scores.forEach(1, (k, v) ->
        System.out.println(k + ": " + v));

// search — returns first non-null result
String topScorer = scores.search(1, (k, v) ->
        v >= 90 ? k : null);

// reduce — aggregate all values
int total = scores.reduceValues(1, Integer::sum);
```

The first argument (`parallelismThreshold`) controls when parallelism kicks in. Pass `1` to always use parallel execution; pass `Long.MAX_VALUE` to force sequential.

## When to Use ConcurrentHashMap

### Good use cases

- **Shared caches** — multiple threads reading and writing concurrently.
- **Frequency counters** — use `merge` or `compute` for thread-safe counting.
- **Concurrent grouping** — use `computeIfAbsent` to build groups safely.
- **Event dispatchers** — concurrent listener registries.

```java
// Thread-safe word frequency counter
ConcurrentHashMap<String, Integer> freq = new ConcurrentHashMap<>();

void countWord(String word) {
    freq.merge(word, 1, Integer::sum);
}
```

### Bad use cases

- **Single-threaded code** — use `HashMap` for better raw performance.
- **Compound operations needing external atomicity** — even `ConcurrentHashMap`'s atomic methods cannot guarantee correctness across multiple map calls without external synchronisation.

```java
// UNSAFE — two separate atomic ops, not one
if (!map.containsKey("key")) {
    map.put("key", computeValue()); // another thread may have inserted between
}

// SAFE — use computeIfAbsent
map.computeIfAbsent("key", k -> computeValue());
```

## Pitfalls and Gotchas

### Pitfall 1: Compound check-then-act is not atomic

```java
// WRONG — race condition between get and put
if (map.get("key") == null) {
    map.put("key", value);
}

// CORRECT
map.putIfAbsent("key", value);
// or
map.computeIfAbsent("key", k -> value);
```

### Pitfall 2: size() can be stale

Do not rely on `size()` for correctness decisions in concurrent code. Use atomic operations instead.

```java
// WRONG
if (map.size() == 0) {
    map.put("first", 1); // another thread may have added between size() and put()
}
```

### Pitfall 3: forEach is not atomic as a whole

Each individual `forEach` callback is applied atomically per entry, but the entire `forEach` loop is not atomic. Entries may be added or removed mid-loop.

### Pitfall 4: Nested compute calls can deadlock

Calling `compute`, `merge`, or `computeIfAbsent` on the **same key** inside the callback of another such call on that key will deadlock.

```java
// DEADLOCK — nested compute on same key
map.compute("key", (k, v) -> {
    map.compute("key", (k2, v2) -> v2 + 1); // waits for outer lock — deadlock
    return v + 1;
});
```

## Time Complexity

| Operation | Average | Worst Case |
|-----------|---------|------------|
| `get(key)` | O(1) | O(log n) with trees |
| `put(key, value)` | O(1) | O(log n) with trees |
| `remove(key)` | O(1) | O(log n) with trees |
| `containsKey(key)` | O(1) | O(log n) with trees |
| `containsValue(value)` | O(n) | O(n) |
| Iteration | O(n) | O(n) |
| Resize (rehash) | O(n) | O(n) — distributed across threads |

> `get()` is always **lock-free** and runs without any synchronisation overhead.

## Common Coding Problems

### 1. Thread-safe frequency counter

```java
ConcurrentHashMap<String, Integer> freq = new ConcurrentHashMap<>();

public void record(String word) {
    freq.merge(word, 1, Integer::sum);
}

public int count(String word) {
    return freq.getOrDefault(word, 0);
}
```

### 2. Thread-safe grouping (multi-map)

```java
ConcurrentHashMap<String, List<String>> groups = new ConcurrentHashMap<>();

public void addToGroup(String group, String item) {
    groups.computeIfAbsent(group, k -> new CopyOnWriteArrayList<>()).add(item);
}
```

### 3. Concurrent cache with lazy initialisation

```java
ConcurrentHashMap<Integer, String> cache = new ConcurrentHashMap<>();

public String getOrLoad(int id) {
    return cache.computeIfAbsent(id, this::loadFromDatabase);
}

private String loadFromDatabase(int id) {
    // expensive operation — called at most once per key
    return "data-" + id;
}
```

### 4. Thread-safe LRU using ConcurrentHashMap + LinkedList

```java
public class ConcurrentLRU<K, V> {

    private final int capacity;
    private final ConcurrentHashMap<K, V> map;
    private final Deque<K> order;

    public ConcurrentLRU(int capacity) {
        this.capacity = capacity;
        this.map   = new ConcurrentHashMap<>();
        this.order = new ArrayDeque<>();
    }

    public synchronized V get(K key) {
        if (!map.containsKey(key)) return null;
        order.remove(key);
        order.addLast(key);
        return map.get(key);
    }

    public synchronized void put(K key, V value) {
        if (map.containsKey(key)) order.remove(key);
        else if (map.size() >= capacity) map.remove(order.pollFirst());
        map.put(key, value);
        order.addLast(key);
    }
}
```

### 5. Parallel word count using bulk operations

```java
ConcurrentHashMap<String, Integer> wordCount = new ConcurrentHashMap<>();

public void processLines(List<String> lines) {
    lines.parallelStream()
         .flatMap(line -> Arrays.stream(line.split("\\s+")))
         .forEach(word -> wordCount.merge(word, 1, Integer::sum));
}

public int totalWords() {
    return wordCount.reduceValues(1, Integer::sum);
}
```
