

## Java 17 Features (LTS Release) — Interview Notes

Java 17 is a Long-Term Support (LTS) release focused more on stabilization and standardization of modern Java features rather than introducing many new concepts.

---

### 1. Sealed Classes (JEP 409)

Sealed classes allow a class or interface to restrict which other classes can extend or implement it.

#### Key Points

* Controls inheritance hierarchy
* Improves domain modeling and safety
* Useful when the set of subclasses is known

#### Example

```java
sealed class Shape permits Circle, Square {}

final class Circle extends Shape {}

final class Square extends Shape {}
```

#### Use Case

* Domain modeling where only specific types are allowed
* Preventing uncontrolled inheritance

---

### 2. Pattern Matching for `instanceof` (JEP 394)

Removes the need for explicit casting after `instanceof`.

#### Example

```java
if (obj instanceof String s) {
    System.out.println(s.toUpperCase());
}
```

#### Benefit

* Reduces boilerplate code
* Improves readability and safety

---

### 3. Records (Finalized in Java 16, Standard in Java 17)

Records provide a compact way to create immutable data carrier classes.

#### Example

```java
record User(String name, int age) {}
```

#### Features

* Automatically generates:

  * Constructor
  * Getters
  * `equals()`, `hashCode()`, `toString()`

#### Use Case

* DTOs (Data Transfer Objects)
* Simple data models

---

### 4. Switch Pattern Matching (Preview in Java 17)

Enhances switch statements with type-based pattern matching.

#### Example

```java
switch (obj) {
    case String s -> System.out.println(s);
    case Integer i -> System.out.println(i);
    default -> System.out.println("Unknown type");
}
```

#### Benefit

* Cleaner conditional logic
* Reduces `if-else` chains

---

### 5. Text Blocks (Standard)

Simplifies writing multi-line string literals.

#### Example

```java
String json = """
{
  "name": "Anil",
  "role": "developer"
}
""";
```

#### Benefits

* Improves readability
* Useful for JSON, SQL, HTML

---

### 6. Strong Encapsulation of JDK Internals

Internal JDK APIs are strongly encapsulated and not easily accessible.

#### Impact

* Improves security
* Encourages use of standard public APIs
* Reduces dependency on internal implementation details

---

### 7. Foreign Function & Memory API (Incubator)

Provides a safer and modern way to interact with native code and memory outside the JVM.

#### Use Cases

* Calling C libraries
* Direct memory access without JNI

#### Benefit

* Replaces complex JNI usage with simpler APIs

---

### 8. Enhanced Pseudo-Random Number Generators

Introduces new interfaces and implementations for better randomness control.

#### Improvements

* Multiple algorithms supported
* Better performance and flexibility

---

### 9. Deprecations and Removals

Java 17 removes or deprecates older legacy features.

#### Removed

* Applet API
* RMI Activation
* Experimental AOT and JIT compiler

#### Goal

* Clean up legacy and unused features
* Improve maintainability of the platform

---

## Key Interview Insight

Java 17 is not a feature-heavy release. It mainly:

* Finalizes features introduced in Java 14–16
* Improves language safety and readability
* Strengthens platform stability for enterprise applications

---

## Quick Summary

Java 17 focuses on:

* Better data modeling (Records, Sealed Classes)
* Cleaner syntax (Pattern Matching, Text Blocks)
* Safer platform design (Strong Encapsulation)
* Modern APIs (Foreign Memory, RNG improvements)

### Pattern Matching for `instanceof` (JEP 394)

This feature solves a very common Java problem: **boilerplate code and unsafe casting after `instanceof` checks**.

---

## 1. Problem in Old Java (Before Java 16 / 17)

#### Code before pattern matching

```java
Object obj = "Hello";

if (obj instanceof String) {
    String s = (String) obj;   // explicit cast needed
    System.out.println(s.toUpperCase());
}
```

---

### What is the problem here?

#### 1. Repeated casting

You must write:

```java
(String) obj
```

even though Java already knows it is a `String`.

---

#### 2. Risk of mistakes

If you forget the `instanceof` check:

```java
String s = (String) obj; // can throw ClassCastException
```

---

#### 3. Verbose and noisy code

You repeat:

* type check
* type cast
* variable usage

This reduces readability.

---

## 2. Solution in Java 17 (Pattern Matching for instanceof)

#### Code after improvement

```java
Object obj = "Hello";

if (obj instanceof String s) {
    System.out.println(s.toUpperCase());
}
```

---

### What changed?

#### 1. Automatic casting

Java automatically casts `obj` to `String`.

---

#### 2. New variable binding

`s` is created only if the check passes.

---

#### 3. Safer code

No risk of wrong casting inside the block.

---

## 3. Side-by-side comparison

| Feature        | Before Java 17 | Java 17   |
| -------------- | -------------- | --------- |
| Type check     | Required       | Required  |
| Casting        | Manual         | Automatic |
| Extra variable | Separate line  | Inline    |
| Readability    | Verbose        | Clean     |
| Safety         | Manual risk    | Safer     |

---

## 4. Real-world example (common interview use case)

#### Before

```java
void printLength(Object obj) {
    if (obj instanceof String) {
        String s = (String) obj;
        System.out.println(s.length());
    }
}
```

#### After

```java
void printLength(Object obj) {
    if (obj instanceof String s) {
        System.out.println(s.length());
    }
}
```

---

## 5. Key idea (important interview point)

Pattern matching for `instanceof` removes the need for:

> “Check type → cast type → use variable”

It becomes:

> “Check type + bind variable in one step”

---

## 6. Critical thinking (interview perspective)

#### What assumptions does this feature remove?

* You assume casting is always needed after type check → now false

#### Possible limitation?

* Works only within the scope of the `if` block (not outside)

Example:

```java
if (obj instanceof String s) {
    // s is valid here
}
// s is NOT accessible here
```

---

## Summary

Pattern Matching for `instanceof` solves:

* Boilerplate casting
* Risk of incorrect casting
* Readability issues

By combining:

> type check + variable binding in one step
