SQL query execution becomes much easier if you understand the logical order.

Many people think SQL runs from top to bottom, but internally it does not.

## Actual Logical Execution Order

```sql id="m78kda"
FROM
JOIN
WHERE
GROUP BY
HAVING
SELECT
ORDER BY
LIMIT
```

---

# Example 1 — JOIN + WHERE + GROUP BY + HAVING

Suppose we have:

## employees

| emp_id | name | dept_id | salary |
| ------ | ---- | ------- | ------ |
| 1      | Anil | 10      | 50000  |
| 2      | Ravi | 10      | 60000  |
| 3      | Sam  | 20      | 40000  |
| 4      | John | 20      | 70000  |
| 5      | Ram  | 30      | 30000  |

---

## departments

| dept_id | dept_name |
| ------- | --------- |
| 10      | HR        |
| 20      | IT        |
| 30      | Sales     |

---

# Query

```sql id="zthk5t"
SELECT d.dept_name,
       COUNT(e.emp_id) AS total_emp,
       AVG(e.salary) AS avg_salary
FROM employees e
JOIN departments d
    ON e.dept_id = d.dept_id
WHERE e.salary > 40000
GROUP BY d.dept_name
HAVING AVG(e.salary) > 55000;
```

---

# Step-by-Step Execution

---

# STEP 1 — FROM

```sql id="3qgh0e"
FROM employees e
```

Start with employees table.

| emp_id | name | dept_id | salary |
| ------ | ---- | ------- | ------ |
| 1      | Anil | 10      | 50000  |
| 2      | Ravi | 10      | 60000  |
| 3      | Sam  | 20      | 40000  |
| 4      | John | 20      | 70000  |
| 5      | Ram  | 30      | 30000  |

---

# STEP 2 — JOIN

```sql id="v0nl72"
JOIN departments d
ON e.dept_id = d.dept_id
```

Now combine matching rows.

| name | dept_name | salary |
| ---- | --------- | ------ |
| Anil | HR        | 50000  |
| Ravi | HR        | 60000  |
| Sam  | IT        | 40000  |
| John | IT        | 70000  |
| Ram  | Sales     | 30000  |

---

# STEP 3 — WHERE

```sql id="h9rj3m"
WHERE e.salary > 40000
```

Filter rows BEFORE grouping.

| name | dept_name | salary |
| ---- | --------- | ------ |
| Anil | HR        | 50000  |
| Ravi | HR        | 60000  |
| John | IT        | 70000  |

Notice:

* Sam removed (40000 not > 40000)
* Ram removed

---

# STEP 4 — GROUP BY

```sql id="5q4s2l"
GROUP BY d.dept_name
```

Create groups.

### HR Group

* 50000
* 60000

### IT Group

* 70000

---

# STEP 5 — HAVING

```sql id="bm4x6n"
HAVING AVG(e.salary) > 55000
```

HAVING filters GROUPS.

### HR

Average:
\frac{50000+60000}{2}=55000

Not greater than 55000 → Removed

---

### IT

Average:
\frac{70000}{1}=70000

Greater than 55000 → Kept

---

# STEP 6 — SELECT

```sql id="z5f7f5"
SELECT d.dept_name,
       COUNT(e.emp_id),
       AVG(e.salary)
```

Final output:

| dept_name | total_emp | avg_salary |
| --------- | --------- | ---------- |
| IT        | 1         | 70000      |

---

# Important Difference

## WHERE vs HAVING

| WHERE                                   | HAVING                  |
| --------------------------------------- | ----------------------- |
| Filters rows                            | Filters groups          |
| Executes before GROUP BY                | Executes after GROUP BY |
| Cannot use aggregate functions normally | Used with aggregates    |
| Faster usually                          | Slower usually          |

# Example 2 — LEFT JOIN + WHERE Problem

Many interviewers ask this.

---

## Query

```sql id="7xalca"
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d
ON e.dept_id = d.dept_id
WHERE d.dept_name = 'HR';
```

People think this is still LEFT JOIN behavior.

Wrong.

---

# Step-by-Step

## STEP 1 — LEFT JOIN

Result:

| name  | dept_name |
| ----- | --------- |
| Anil  | HR        |
| Ravi  | HR        |
| Sam   | IT        |
| John  | IT        |
| Ram   | Sales     |
| Kiran | NULL      |

---

## STEP 2 — WHERE

```sql id="sg1j84"
WHERE d.dept_name = 'HR'
```

Now NULL rows removed.

Final:

| name | dept_name |
| ---- | --------- |
| Anil | HR        |
| Ravi | HR        |

Now it behaves almost like INNER JOIN.

---

# Correct LEFT JOIN Filtering

If you want to preserve LEFT JOIN behavior:

```sql id="r3dbvm"
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d
ON e.dept_id = d.dept_id
AND d.dept_name = 'HR';
```

Now unmatched rows remain.

# Simple Mental Model

## JOIN

Combine tables.

---

## WHERE

Filter individual rows.

---

## GROUP BY

Create buckets/groups.

---

## HAVING

Filter buckets/groups.

# Interview Trick Question

Will this work?

```sql id="rt1s2q"
SELECT dept_id, AVG(salary)
FROM employees
WHERE AVG(salary) > 50000
GROUP BY dept_id;
```

No.

Because:

* `WHERE` executes BEFORE grouping.
* `AVG()` not available yet.

Correct:

```sql id="rqeh5i"
SELECT dept_id, AVG(salary)
FROM employees
GROUP BY dept_id
HAVING AVG(salary) > 50000;
```
