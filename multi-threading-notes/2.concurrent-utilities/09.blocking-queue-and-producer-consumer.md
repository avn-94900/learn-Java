## 1. What is BlockingQueue

`BlockingQueue` is a thread-safe interface in `java.util.concurrent` designed specifically for producer-consumer scenarios.

### 1.1 Key Traits

- **Thread-safe** — multiple threads can access concurrently without manual synchronization.
- **Blocking operations** — automatically waits when the queue is full (`put`) or empty (`take`).
- Can be **bounded** (fixed capacity) or **unbounded** (grows as needed).

### 1.2 Interface Hierarchy

```
java.util.Collection<E>
    └── java.util.Queue<E>
            └── java.util.concurrent.BlockingQueue<E>
```

### 1.3 Core Methods

| Category | Method | Behavior |
|---|---|---|
| Blocking | `put(E e)` | Waits if full |
| Blocking | `take()` | Waits if empty |
| Non-blocking | `offer(E e)` | Returns `false` if full |
| Non-blocking | `poll()` | Returns `null` if empty |
| Timed | `offer(E e, long t, TimeUnit u)` | Waits up to timeout |
| Timed | `poll(long t, TimeUnit u)` | Waits up to timeout |
| Utility | `remainingCapacity()` | Space left in queue |
| Utility | `drainTo(Collection c)` | Bulk remove all elements to a collection |

## 2. Implementations

### 2.1 ArrayBlockingQueue

Bounded, array-backed, FIFO. Memory usage is fixed and predictable.

```java
BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(10);

// fair = true means threads acquire in arrival order
BlockingQueue<Integer> fairQueue = new ArrayBlockingQueue<>(100, true);
```

**Use when:** capacity is known upfront and predictable memory usage matters.

---

### 2.2 LinkedBlockingQueue

Optionally bounded, linked-node, FIFO. Higher throughput than `ArrayBlockingQueue`.

```java
BlockingQueue<String> unbounded = new LinkedBlockingQueue<>();
BlockingQueue<String> bounded   = new LinkedBlockingQueue<>(100);
```

**Use when:** high throughput or dynamic sizing is needed.

> **Warning:** An unbounded queue with a fast producer can grow until `OutOfMemoryError`. Always prefer bounded queues in production.

---

### 2.3 PriorityBlockingQueue

Unbounded, priority-ordered. Elements processed by natural order (`Comparable`) or a custom `Comparator`.

```java
BlockingQueue<Task> queue = new PriorityBlockingQueue<>(
    10,
    (t1, t2) -> Integer.compare(t1.getPriority(), t2.getPriority())
);
```

**Use when:** elements need priority-based processing instead of FIFO.

---

### 2.4 DelayQueue

Unbounded. Elements become available only after their delay expires. Elements must implement `Delayed`.

```java
BlockingQueue<DelayedTask> queue = new DelayQueue<>();
```

```java
class DelayedTask implements Delayed
{
    private final long expire;
    private final String task;

    public DelayedTask(String task, long delayMs)
    {
        this.task   = task;
        this.expire = System.currentTimeMillis() + delayMs;
    }

    @Override
    public long getDelay(TimeUnit unit)
    {
        // negative or zero means element is ready to be taken
        return unit.convert(expire - System.currentTimeMillis(), TimeUnit.MILLISECONDS);
    }

    @Override
    public int compareTo(Delayed o)
    {
        return Long.compare(this.expire, ((DelayedTask) o).expire);
    }
}
```

**Use when:** scheduled or delayed task processing is needed.

---

### 2.5 SynchronousQueue

Zero capacity. Every `put()` blocks until another thread calls `take()`. Direct thread-to-thread handoff with no buffer.

```java
BlockingQueue<String> queue = new SynchronousQueue<>();
```

**Use when:** direct transfer between threads with no intermediate storage is needed.

---

### 2.6 LinkedTransferQueue

Unbounded with extra `transfer()` methods. A producer can wait until a consumer actually picks up the element.

```java
TransferQueue<String> queue = new LinkedTransferQueue<>();

queue.transfer(item);    // blocks until a consumer takes it
queue.tryTransfer(item); // returns false immediately if no consumer is waiting
```

**Use when:** you need both buffered queuing and direct-transfer semantics.

---

### 2.7 Performance Summary

| Implementation | Throughput | Memory | Notes |
|---|---|---|---|
| `LinkedBlockingQueue` | Highest | Dynamic | Best general-purpose choice |
| `ArrayBlockingQueue` | Moderate | Fixed | Predictable memory usage |
| `PriorityBlockingQueue` | Lower | Dynamic | Sorting adds overhead |
| `SynchronousQueue` | Highest (direct) | Minimal | No buffering — zero storage |

## 3. Producer-Consumer Patterns

### 3.1 Approach 1 — wait() / notify()

The classic approach. Threads coordinate via a shared lock. Included here for learning and understanding legacy code. Do not use in new projects.

```java
class SharedQueue
{
    private int data;
    private boolean available = false;

    public synchronized void produce(int value) throws InterruptedException
    {
        while (available)
        {
            wait(); // wait until consumer has taken the current item
        }
        data = value;
        System.out.println("Produced: " + data);
        available = true;
        notify(); // wake up the consumer
    }

    public synchronized int consume() throws InterruptedException
    {
        while (!available)
        {
            wait(); // wait until producer has placed an item
        }
        System.out.println("Consumed: " + data);
        available = false;
        notify(); // wake up the producer
        return data;
    }
}

class Producer extends Thread
{
    private SharedQueue queue;

    Producer(SharedQueue queue)
    {
        this.queue = queue;
    }

    public void run()
    {
        for (int i = 1; i <= 5; i++)
        {
            try
            {
                queue.produce(i);
                Thread.sleep(500);
            }
            catch (InterruptedException e)
            {
                Thread.currentThread().interrupt();
            }
        }
    }
}

class Consumer extends Thread
{
    private SharedQueue queue;

    Consumer(SharedQueue queue)
    {
        this.queue = queue;
    }

    public void run()
    {
        for (int i = 1; i <= 5; i++)
        {
            try
            {
                queue.consume();
                Thread.sleep(500);
            }
            catch (InterruptedException e)
            {
                Thread.currentThread().interrupt();
            }
        }
    }
}

public class ProducerConsumerDemo
{
    public static void main(String[] args)
    {
        SharedQueue queue = new SharedQueue();
        new Producer(queue).start();
        new Consumer(queue).start();
    }
}
```

**Output:**
```
Produced: 1
Consumed: 1
Produced: 2
Consumed: 2
...
```

### 3.2 Approach 1b — Lambda Style

Same `SharedQueue` as above. Producer and Consumer classes replaced with lambdas. Behavior is identical.

```java
public class LambdaProducerConsumer
{
    public static void main(String[] args)
    {
        SharedQueue queue = new SharedQueue();

        Runnable producer = () ->
        {
            for (int i = 1; i <= 5; i++)
            {
                try
                {
                    queue.produce(i);
                    Thread.sleep(500);
                }
                catch (InterruptedException e)
                {
                    Thread.currentThread().interrupt();
                }
            }
        };

        Runnable consumer = () ->
        {
            for (int i = 1; i <= 5; i++)
            {
                try
                {
                    queue.consume();
                    Thread.sleep(500);
                }
                catch (InterruptedException e)
                {
                    Thread.currentThread().interrupt();
                }
            }
        };

        new Thread(producer, "Producer-Thread").start();
        new Thread(consumer, "Consumer-Thread").start();
    }
}
```

### 3.3 Approach 2 — BlockingQueue with Lambda

**Modern and recommended.** No manual `wait()`, `notify()`, or `synchronized` needed. `BlockingQueue` handles all coordination internally.

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;

public class Main
{
    public static void main(String[] args)
    {
        BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(1);

        Runnable producer = () ->
        {
            for (int i = 1; i <= 5; i++)
            {
                try
                {
                    queue.put(i);                         // blocks if queue is full
                    System.out.println("Produced: " + i);
                    Thread.sleep(500);
                }
                catch (InterruptedException e)
                {
                    Thread.currentThread().interrupt();
                }
            }
        };

        Runnable consumer = () ->
        {
            for (int i = 1; i <= 5; i++)
            {
                try
                {
                    int v = queue.take();                 // blocks if queue is empty
                    System.out.println("Consumed: " + v);
                    Thread.sleep(2000);
                }
                catch (InterruptedException e)
                {
                    Thread.currentThread().interrupt();
                }
            }
        };

        new Thread(producer, "Producer").start();
        new Thread(consumer, "Consumer").start();
    }
}
```

**Output:**
```
Produced: 1
Consumed: 1
Produced: 2
Consumed: 2
...
```

### 3.4 Approach 3 — Multiple Producers and Consumers

`LinkedBlockingQueue` safely handles concurrent access from multiple producers and consumers.

```java
import java.util.concurrent.*;

public class MultiProducerConsumer
{
    private static final BlockingQueue<String> queue = new LinkedBlockingQueue<>(50);

    static class Producer implements Runnable
    {
        private final String name;

        Producer(String name)
        {
            this.name = name;
        }

        @Override
        public void run()
        {
            try
            {
                for (int i = 1; i <= 10; i++)
                {
                    String item = name + "-Item-" + i;
                    queue.put(item);
                    System.out.println(name + " produced: " + item);
                    Thread.sleep(200);
                }
            }
            catch (InterruptedException e)
            {
                Thread.currentThread().interrupt();
            }
        }
    }

    static class Consumer implements Runnable
    {
        private final String name;

        Consumer(String name)
        {
            this.name = name;
        }

        @Override
        public void run()
        {
            try
            {
                while (true)
                {
                    String item = queue.take();           // blocks until an item is available
                    System.out.println(name + " consumed: " + item);
                    Thread.sleep(300);
                }
            }
            catch (InterruptedException e)
            {
                Thread.currentThread().interrupt();
            }
        }
    }

    public static void main(String[] args)
    {
        for (int i = 1; i <= 3; i++) new Thread(new Producer("Producer-" + i)).start();
        for (int i = 1; i <= 2; i++) new Thread(new Consumer("Consumer-" + i)).start();
    }
}
```

### 3.5 Anti-Pattern — No Coordination

Using a flag without any synchronization. Data can be missed or consumed out of order.

```java
class WrongQueue
{
    private int data;
    private boolean produced = false;

    public void produce(int value)
    {
        if (!produced)
        {
            data = value;
            System.out.println("Produced (Wrong): " + value);
            produced = true;
            // no notification — consumer has to poll or guess timing
        }
    }

    public void consume()
    {
        if (produced)
        {
            System.out.println("Consumed (Wrong): " + data);
            produced = false;
            // no notification — producer has to poll or guess timing
        }
    }
}
```

**Problem:** No synchronization means items can be skipped or consumed out of order.

```
Produced (Wrong): 1
Produced (Wrong): 2
Consumed (Wrong): 2   ← item 1 was lost
```

### 3.6 Approach Comparison

| Feature | `wait()` / `notify()` | `BlockingQueue` |
|---|---|---|
| Communication | Manual, coordinated by developer | Built-in |
| Synchronization | Requires `synchronized` block | Handled internally |
| Thread-safe | Yes | Yes |
| Boilerplate | High | Low |
| Risk of bugs | High (spurious wakeups, missed signals) | Low |
| Recommended for | Learning and legacy code | Production code |

## 4. Best Practices

### 4.1 Handle InterruptedException Correctly

Never swallow `InterruptedException`. Always restore the interrupt status so the caller can react.

```java
try
{
    queue.put(item);
}
catch (InterruptedException e)
{
    Thread.currentThread().interrupt(); // restore interrupt status
}
```

### 4.2 Graceful Shutdown

Use a `volatile` flag and timed `poll()` so consumers do not block forever during shutdown.

```java
private static volatile boolean shutdown = false;

// consumer loop — poll with timeout instead of blocking forever
while (!shutdown)
{
    String item = queue.poll(1, TimeUnit.SECONDS); // returns null if nothing arrives in 1s
    if (item != null)
    {
        process(item);
    }
}

// drain any remaining items after shutdown signal
while (!queue.isEmpty())
{
    process(queue.poll());
}
```

### 4.3 Monitor Queue Health

Use these to detect backpressure or starvation during operation.

```java
queue.size();              // current number of elements
queue.remainingCapacity(); // how many more can be added before blocking
queue.isEmpty();           // true if no elements present
```

## 5. Common Pitfalls

| Pitfall | Problem | Fix |
|---|---|---|
| Unbounded queue with fast producer | Queue grows until `OutOfMemoryError` | Always use bounded queues in production |
| Inserting `null` | All implementations throw `NullPointerException` | Never insert `null` — use a sentinel object if needed |
| Circular `put` dependencies | Two threads each blocked waiting on the other's queue | Avoid circular queue dependencies between threads |
| Blocking forever in consumers | Consumer stuck if no shutdown mechanism exists | Use `poll(timeout)` with a shutdown flag |
